# Explicit Dependency Testability Report

Date: 2026-05-05

## Executive Summary

The backend is now mostly aligned with the **Explicit Dependency Boundary**
standard:

> Code under unit test must not obtain behavior-changing inputs through hidden
> reads. Any input that can make the result non-reentrant must be passed
> explicitly by the caller, captured in a typed snapshot, or accessed through an
> injected port/provider that tests can replace.

The important shift is not "remove IO everywhere". The shift is:

- Core/domain logic receives explicit values, snapshots, or injected providers.
- Imperative shell code owns real IO such as env, time, ids, files, db, network,
  process state, and service discovery.
- Unit tests can reproduce behavior from explicit inputs instead of depending on
  ambient process state.

## Current Test Matrix

The canonical backend test entrypoint is:

```bash
scripts/run_all_tests.sh
```

It deliberately avoids bare root `pytest` over every submodule. Each submodule
has its own package root and dependency edges, so the runner supplies explicit
`PYTHONPATH` per module:

| Module | Command shape |

## Boundary Status By Service

### Agent Runtime

Status: mostly compliant.

What is now explicit:

- Queue/session orchestration takes clock and id providers.
- Workers accept runtime dependencies instead of internally constructing
  behavior-changing services.
- Tests cover deterministic queue/session behavior and worker dependency
  injection.

Remaining discipline:

- Keep DB and HTTP clients at repository/adapter boundaries.
- Do not reintroduce "current session", "current scope", or "current agent" as
  hidden globals inside state machines.

### Business

Status: mostly compliant.

What is now explicit:

- Dispatch subscriber IM aggregation is configured by an injected
  `IMAggregationConfig`.
- The env reader lives at `main_subscriber.py`, not inside aggregation logic.
- Subscriber worker id is provided from the entrypoint instead of generated in
  the subscriber domain class.
- Business action helpers use explicit dependency bundles where the action is
  unit-testable.

Remaining discipline:

- Entangled/entity store access is allowed in repositories and action adapters,
  but should not be hidden inside reducers or prompt/context builders.
- Product context such as profile, chat history, device binding, and subagent
  state must stay product data, not "Agent autonomous memory" side channels.

### Cortex

Status: mostly compliant.

What is now explicit:

- Scope state transition logging requires explicit timestamps when the transition
  log is written.
- `StackFrame` no longer has a `created_at` default generated by `time.time`.
- Workspace and scope-state paths are more explicit at construction/entrypoint
  boundaries.

Remaining discipline:

- Cortex may store work trace and payload refs. It should not silently summarize
  or infer durable memory from chat replies.
- Large payload content should remain behind Blob/payload refs unless an explicit
  observation tool reads or interprets it.

### Common

Status: compliant for LLM assembly contracts.

What is now explicit:

- LLM assembly inputs are represented as explicit contracts.
- Cross-module contract tests verify the schema boundary used by Runtime and
  Common.

Remaining discipline:

- `common` should stay schema/config/tool-contract authority. It should not grow
  hidden runtime ownership of business state.

### Blob Service

Status: mostly compliant.

What is now explicit:

- Storage backend, S3/disk choices, multipart behavior, and limit settings are
  explicit providers/config instead of hidden module defaults in core paths.
- Tests cover backend and blob-service behavior.

Remaining discipline:

- Blob owns bytes and metadata only. Business meaning belongs to caller services.

### LLM Factory

Status: mostly compliant.

What is now explicit:

- Provider conversion and retry behavior are more explicit and covered by route
  tests.

Remaining discipline:

- Provider discovery/config can read real environment at startup or route
  boundaries, but conversion/retry logic should keep taking explicit provider
  snapshots.

## What This Improves

- Unit tests are more deterministic.
- State-machine bugs are easier to reproduce.
- Prompt/context assembly can be tested from captured snapshots.
- Hidden production differences such as env, time, current working directory, or
  ambient service state are less likely to affect core decisions.
- Maintenance cost drops because there are fewer invisible branches.

## What This Does Not Mean

This does **not** mean every file, DB row, or network response must be loaded into
memory and passed through every layer. Real IO is normal at boundaries:

- CLI and service entrypoints may read env/config/argv.
- Repositories may read/write DB rows.
- HTTP clients may call other services.
- Blob adapters may read/write disk or S3.

The rule is that IO should be visible at the boundary, then converted into
explicit input for the logic being tested.

## Review Checklist For New Code

Before merging code that affects backend logic, ask:

1. Does the function's output depend on current time, random id, env, cwd,
   database, network, global registry, current user, current session, or current
   scope?
2. If yes, is that dependency passed as an explicit argument, typed snapshot, or
   injected provider?
3. Is real IO isolated in an entrypoint, repository, adapter, or client?
4. Is there a unit test that controls the dependency and proves deterministic
   behavior?
5. Is there any fallback path that preserves old hidden behavior?

If the answer to 2 or 4 is no, do not merge it as core logic.

## Recommended Next Hardening

1. Add targeted guardrails for the hottest core packages to reject hidden reads in
   prompt builders, state machines, and schedulers.
2. Expand cross-service integration tests around Business -> Environment ->
   Runtime -> Cortex -> Agent Monitor.
3. Keep `scripts/run_all_tests.sh` as the canonical backend test matrix and update
   it whenever a new backend submodule becomes part of the hot path.
4. Treat hidden fallback removal as a default review requirement, not a special
   cleanup project.
