# 多播并流：Android Scrcpy 防死锁黑魔法

> 路径：相关的 Android 与 Scrcpy 接管处理。

## 1. 原生 Scrcpy 服务端的问题
大家都会用原生 adb 连那个广为人知且伟大的开源项目 `Scrcpy` 来把实体安卓机顶盒接出来拿在屏幕上看。
但这存在极大的瓶颈。如果你有 3 个研究人员通过网关用网页版连了同一部在异地的安卓真机；原生 Scrcpy 服务器对于连接的管理在直接多终端请求时很容易因处理不过来视频流的分叉打死而挂掉。

## 2. Rust 的中间人 Broadcaster (代理转播者)
这就回到了 `vmcontrol` 的黑科技分身：充当一台极其优秀的内网分流器（Broadcaster）代理。
- 只有那个 Tauri `/ vmcontrol` 作为唯一一个 Socket 连接插驻跟本机的 `scrcpy-server.jar` 维持会话拿取那一丝单向不断喷涌的原始视频流数据。
- 随后它作为发布者。将这个视频轨克隆到了本地的一个大内存共享列表（并受到 Rust 所有权的安全把控甚至运用了跨线程通道如 `Tokio::broadcast`）。
- 现在如果有三个不同的外部用户建立对这台机子的 WebRTC 会话：对不起它们不准直接去问手机拿资源，而是直接变成订阅挂机在这套频道后面领到了同样的一路源分支。

## 3. IDR 关键帧求救与黑屏假死解封
在这套设计中我们花了很多时间处理一个大坑：“如果有人后加入了一个正在运行看视频的人群，那这老兄上来只能看到五颜六色的贴图碎块和花屏、乃至死黑！”
- 这是经典的 H264 P/I/B 帧问题：只要不是凑巧遇到了完整包含整屏幕全幅像素的一个“大帧”(IDR 关键帧），后加入的客户端解码器根本不知道之前的人在看啥，无法绘制基础贴图。
- 我们在通道处理加了个“求救触发器”（`Force IDR Generator`）！新一旦接入的 Peer 通道进入连线。触发器会用另一根逆向线给手机 Scrcpy 控制端塞进：赶紧临时刷爆一个全屏刷新！让新进来的人获取大快慰以平稳定格画面。拯救了跨设备连协时的死锁。
