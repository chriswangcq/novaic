# DOM 的无尽炼狱：超大型列表的宿命

> 路径参考：各种涉及消息流水和工作台 `Execution Logs` 执行台打印的大长框容器面板配置

## 1. 死循环渲染的坍塌
NovAIC 这个系统，它的 AI 有多残暴，它的日志输出量就能翻几十倍来回报开发者。
如果在 Agent 开启一场复杂的网页调查执行，你可能会在不到 30 秒中收到成天上千个极细节的命令状态翻转跟步骤日志返回。
传统的 `map(() => </Log>)` 管线在这时会让你的整个控制台由于渲染着数万千个极其精美带有圆角或者层叠上下文边框的 `div` 爆缸挂掉甚至内存溢出 OOM。

## 2. 只有神才能看十万页：List Virtualization
引入基于类似视图窗口剔除法的长列表虚拟化滚动组件概念（如强关联使用支持高度不确定的 `react-virtuoso` 甚至自家针对性截断）：
- 系统只承认并绘制只处在那极其狭窄的展示视口（Viewport）加上下各缓存的一小块高度（Buffer Bounds）的那几百个 DOM。
- 一旦用户滑动卷轴走掉。底下的元素全部被无情卸载并在瞬间挂替成下面滑进来的新记录节点。这就使得不管那堆 Agent 对话有一百万亿句：这个展示画板在这个网页上始终只占据 15 个 DIV 组件！！

## 3. Auto Scroll & Sticky 下滑
与传统做网页不同。因为这里是实时动态数据。
所有的展示挂靠了智能粘附锚点算法。只有且仅当你的视角处在这个列表的贴底部分时，随着消息来袭，界面才会紧紧粘着跟着跳动（自动追踪进度）。一旦你好奇的拉回上面企图看先前的日志记录：所有的追踪器锚点将被打断处于停滞，任由后面的记录暴走也不会疯狂自动把你滚回底部乱抛：这种体验才被称为“能给人类用、而不是反人类”的智能产品极限定位。
