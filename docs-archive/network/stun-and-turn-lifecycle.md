# TURN 票据下发策略与生存周期

> 核心路径： `novaic-gateway/gateway/api/turn.py` 配合外部 coturn 服务器

## 1. COTURN (外部支持者)
在最广泛的使用当中，我们一般基于成熟稳定经受考验的配置版本 `coturn` 做直接底层服务架设支持。并在服务器主机挂靠在标准的 `3478` 端口去守备那些四面八方来打洞或是走投无路求转接的数据。
由于长期的连接存在未释放和意外崩溃等断信，在我们的生产规范（参考 `thin-client` 里的记录）我们有着必须部署定期巡航与强制释放清理“幽灵（僵尸）通信节点释放”这套强运维策略来为其实机平稳站台。

## 2. 票据流转 (Token Delivery)
我们没有让前端页面毫无防备地把密码硬写在 JavaScript 甚至是环境代码里来让坏人随便偷跑抓包拉爆咱们中继带！
整个获取流程被紧紧封死通过网关进行认证派发（Dynamic Turn Credentialing）：

1. **前提拦截（校验第一重）**：你作为一侧要在网页拉视频了。你的终端通过 `/api/turn/credentials` 调用网关！而且这是在带有着最强验证体系并已经登录在会话中（持有有效的 Bearer JWT 获取的 User_ID）的情况下发生。
2. **派生专属时间片密码卡**：Gateway 在后台用全局隐秘 `TURN_SECRET` 为该请求专属通过诸如 HMAC 哈希签发一次专属、具有极短时效期命名的配给权限 Username 与配套验证参数并向源返回返回。
3. **注入环境并交给 `webrtc_offer` 打向冰川**：前端拿到了这个临时的 `ice_servers` 对象。然后再借前文说的的网关 websocket 下手连同自己的设备信息抛给了遥远的内机控制端告诉：我在找你要建立 WebRtc 通讯连接。

通过这种“身份与票源的极高度切割”。网络集群将对整个全链资产实行最细的安全和防止网络外泄风控接管把位。
