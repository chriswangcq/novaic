# QUIC 连线中继与 CDN 发射塔复合职能

> 路径参考：`novaic-app` 部署流 以及 `novaic-quic-service`

## 1. 为什么是 QUIC？(传输层巨变)
为了绕开陈旧互联网的基础环境对于传统的长尾 UDP 包直接进行黑箱化甚至丢弃造成的巨型波动。也是为了避免那极其费尽连接资源的 TCP 三次握手：
基于多线程穿透与多包连发控制极其先进极其适应恶劣环境的快速握手 `QUIC` 重筑通讯基石是我们解决复杂远控的最安全选项，用它我们能为绝望环境完成中继托底 `Relay`。让那些打洞失败原本准备看黑屏和报错的终端重新顺畅衔接拿取转发。

## 2. “趁手的一鱼两吃” (前端释放部署体系)
既然这套东西这不仅处在一个对于全局公网响应度最佳边缘且专门处理各类高速连结网络要求。我们将这个服务做了巨大的配置升级让其顺带成为了：**“最新版热分发静态前端宿主机”**。
我们砍掉了 Gateway 或者各机器为了承接前端 React 构建物打包结果所耗费的资源（在最古早或者传统的思路那必须配给极其繁杂又容易跟服务耦合起冲突的转发）以及各种奇怪的拉镜。

1. 你只需跑完构建打出发版命令。你的版本打包发版直接打到这。
2. `Novaic-app` （也就是之前所谓的薄壳 Tauri 载体只有几百 Kb 空白体），连上启动直接高速寻到这个 CDN Relay 前沿边际节点拉取这个编译出来的纯纯的基于 React / Vite 逻辑控制和样式表文件。前端开发修 Bug 并点构建上云后，全世界瞬间更新这套轻盈逻辑——直接跳出极其痛苦纠结的 Tauri 和云层验证部署体系周期，完成了完美的 `OTA 前置按需组装发射下发点`！
