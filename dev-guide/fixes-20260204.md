## 修复1：Watchdog Claim逻辑 - 原子性操作

### 问题

Watchdog反复处理同一条消息，导致死循环：

```
[Watchdog] Found sending message: msg-123
[Watchdog] Creating saga for msg-123
[Watchdog] Found sending message: msg-123  # 又找到同一条！
[Watchdog] Creating saga for msg-123
...
```

### 根本原因

- 旧逻辑：`GET /internal/messages/find-sending` 只查询`sending`消息
- Watchdog拿到消息后，创建saga前消息状态没变
- 下一次循环又找到同一条消息

### 解决方案

**文件**: `gateway/api/internal.py`

#### 变更1：新建原子claim API

```python
@router.post("/internal/messages/claim-and-prepare")
def claim_and_prepare_message():
    """
    原子性地claim一条sending消息为sent并返回详情。
    
    幂等性：每次调用最多claim一条，claim过的不会再次返回。
    """
    db = get_db()
    
    with db.get_connection() as conn:
        # 原子性更新：找到最早的sending消息并标记为sent
        cursor = conn.execute("""
            UPDATE chat_messages 
            SET status = 'sent'
            WHERE id = (
                SELECT id FROM chat_messages 
                WHERE status = 'sending'
                ORDER BY timestamp ASC 
                LIMIT 1
            )
            RETURNING id, agent_id, content, timestamp
        """)
        
        row = cursor.fetchone()
        cursor.close()  # ⚠️ 必须在commit前close
        conn.commit()
        
        if not row:
            return {"message": None}
        
        # 查询agent和runtime信息
        ...
        
        return {
            "message": {
                "id": row["id"],
                "agent_id": agent_id,
                "content": row["content"],
                ...
            }
        }
```

**关键点**：
1. `UPDATE ... RETURNING` 保证原子性
2. `cursor.close()` 必须在`commit()`前调用
3. 幂等：已claim的消息不会再返回

**文件**: `task_queue/workers/watchdog.py`

#### 变更2：Watchdog调用新API

```python
def _find_sending_message(self):
    """查找并claim一条sending消息"""
    try:
        resp = self.client.post("/internal/messages/claim-and-prepare")
        data = resp.json()
        return data.get("message")
    except Exception as e:
        logger.error(f"Failed to claim message: {e}")
        return None

def run(self):
    """主循环"""
    while True:
        try:
            # 1. Claim一条消息
            message = self._find_sending_message()
            
            if not message:
                time.sleep(self.interval)
                continue
            
            # 2. 立即创建saga
            self._trigger_message_process_saga(message)
            
            # 3. 不sleep，继续处理下一条（如果有）
            # 只有没消息时才sleep
            
        except Exception as e:
            logger.error(f"Watchdog error: {e}")
            time.sleep(self.interval)
```

**改进**：
- 使用POST而非GET（语义更准确）
- Claim后立即创建saga
- 有消息时不sleep，提高吞吐量

LLM调用返回401错误：

但实际上DB中已经配置了api_keys和candidate_models。

### 根本原因

`_build_llm_config_for_agent`错误地从环境变量读取API key：

```python
# ❌ 旧代码（错误）
def _build_llm_config_for_agent(db, agent_id: str):
    model_id = get_agent_model(agent_id) or "gpt-4o"

        return {"success": False, "error": "API key not configured"}
    ...
```

### 解决方案

**文件**: `gateway/api/internal.py`

完全重写函数，从DB的`api_keys`和`candidate_models`表读取：

```python
def _build_llm_config_for_agent(db, agent_id: str) -> Dict[str, Any]:
    """
    构建agent的LLM配置（从DB读取）
    
    逻辑：
    1. 获取agent的model_id（未设置则用default_model）
    2. 从candidate_models JOIN api_keys获取配置
    3. 返回完整的LLM配置
    """
    
    # 1. Get agent's model_id
    cursor = db.execute(
        "SELECT model_id FROM agents WHERE id = ?",
        (agent_id,)
    )
    row = cursor.fetchone()
    
    if not row:
        return {
            "success": False,
            "error": f"Agent '{agent_id}' not found",
            "agent_id": agent_id,
        }
    
    model_id = row["model_id"]
    
    # 2. If no model selected, use default from config
    if not model_id:
        default_row = db.execute(
            "SELECT value FROM config WHERE key = 'default_model'"
        ).fetchone()
        model_id = default_row[0].strip('"') if default_row else "gpt-4o"
    
    # 3. 从DB查询model和api_key配置（JOIN）
    cursor = db.execute("""
        SELECT 
            m.name as model_name,
            m.provider,
            k.provider as key_provider,
            k.api_key,
            k.api_base
        FROM candidate_models m
        JOIN api_keys k ON m.api_key_id = k.id
        WHERE m.name = ? AND m.available = 1
        LIMIT 1
    """, (model_id,))
    
    row = cursor.fetchone()
    
    if not row:
        return {
            "success": False,
            "error": f"Model '{model_id}' not found in candidate_models",
            "agent_id": agent_id,
            "model_id": model_id,
        }
    
    if not row[3]:  # api_key
        return {
            "success": False,
            "error": f"API key not configured for model '{model_id}'",
            "agent_id": agent_id,
            "model_id": model_id,
        }
    
    # 使用model表的provider或api_key表的provider
    provider = row[1] if row[1] else row[2]
    api_base = row[4] if row[4] else "https://api.openai.com/v1"
    
    return {
        "success": True,
        "agent_id": agent_id,
        "model": row[0],      # model_name
        "provider": provider,
        "api_key": row[3],    # api_key（完整key）
        "api_base": api_base,
    }
```

**关键点**：
1. **不再依赖环境变量**
2. **JOIN查询**：`candidate_models` ← `api_keys`
3. **回退逻辑**：agent.model_id → config.default_model → "gpt-4o"
4. **完整key返回**：内部API可返回完整API key（仅LLM handler使用）

## 修复4：SQLite Cursor必须在Commit前Close

### 问题

大量错误：
```
sqlite3.OperationalError: cannot commit transaction - SQL statements in progress
```

频率：171次 → 修复后：14次（92%减少）

### 根本原因

SQLite要求：
- `fetchone()`/`fetchall()`后，cursor仍处于活动状态
- 此时调用`conn.commit()`会失败
- 必须先`cursor.close()`再`commit()`

### 错误模式

```python
# ❌ 错误
cursor = db.execute("SELECT ...")
row = cursor.fetchone()
db.commit()  # 错误：cursor还在活动状态
```

## 修复5：MCP Client Async/Sync不匹配

### 问题

`tool.execute`任务失败：
```json
{
  "success": false,
  "error": "Gateway error: 500",
  "task_failed": true
}
```

所有`chat_reply`和`runtime_rest`工具调用都失败。

### 根本原因

**文件**: `task_queue/business/mcp.py`

```python
class MCPGatewayClient:
    def call_tool(self, mcp_url: str, tool_name: str, arguments: Dict) -> Dict:
        """同步方法"""
        conn = self._get_connection(mcp_url)
        return conn.call_tool(name=tool_name, arguments=arguments)
        #      ^^^^^^^^^^^^^ 
        #      调用了async方法但没有await！
        #      返回coroutine对象而非结果
```

**文件**: `mcp_gateway/mcp_client.py`

```python
class MCPServerConnection:
    async def call_tool(self, name: str, arguments: Dict) -> Dict:
        """async方法"""
        ...
```

### 解决方案

在`MCPGatewayClient.call_tool()`中使用`asyncio.run()`包装：

```python
class MCPGatewayClient:
    """HTTP client for MCP aggregate gateways."""
    
    def call_tool(self, mcp_url: str, tool_name: str, arguments: Dict[str, Any]) -> Dict[str, Any]:
        """同步调用工具（包装async方法）"""
        import asyncio
        conn = self._get_connection(mcp_url)
        
        # 同步包装async方法
        try:
            loop = asyncio.get_running_loop()
        except RuntimeError:
            # ✅ 没有运行的loop，创建新的
            return asyncio.run(conn.call_tool(name=tool_name, arguments=arguments))
        else:
            # ✅ 有运行的loop，使用run_in_executor避免嵌套
            import concurrent.futures
            with concurrent.futures.ThreadPoolExecutor() as executor:
                future = executor.submit(
                    asyncio.run,
                    conn.call_tool(name=tool_name, arguments=arguments)
                )
                return future.result()
    
    def list_tools(self, mcp_url: str, use_cache: bool = True) -> Dict[str, Any]:
        """同步获取工具列表（包装async方法）"""
        import asyncio
        conn = self._get_connection(mcp_url)
        
        try:
            loop = asyncio.get_running_loop()
        except RuntimeError:
            return asyncio.run(conn.list_tools(use_cache=use_cache))
        else:
            import concurrent.futures
            with concurrent.futures.ThreadPoolExecutor() as executor:
                future = executor.submit(
                    asyncio.run,
                    conn.list_tools(use_cache=use_cache)
                )
                return future.result()
```

**关键点**：
1. 检测是否有running loop（避免嵌套`asyncio.run()`）
2. 无loop：直接`asyncio.run()`
3. 有loop：使用`ThreadPoolExecutor + asyncio.run()`在新线程中执行
