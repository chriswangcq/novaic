| 需求 | 操作 (在根目录或对应目录执行) |
|---|---|
| 本地环境准备 | Node>=18, Rustup (stable), macOS 需 xcode-select |
| 本地跑客户端 | `cd novaic-app && npm install && npm run tauri:dev` |

> 最后更新：2026-04-21 — **PR-40 chat_reply UNIQUE 根治（Entangled id-fallback 删除，fail-fast）**：用户反馈 `chat_reply` 工具调用连续返回 `API error (400): UNIQUE constraint failed: chat_messages.id`，agent 只能回第一条消息，后续全部静默失败。从 prod log → DB `SELECT COUNT(*) FROM chat_messages WHERE id=agent_id` 反查到铁证：唯一存在的那条 `AGENT_REPLY` 的 `id` 被写成了 `agent_id` 本身，导致第二条 INSERT 主键冲突。**根因**：`SqlEntityStore._sql_create` / `append` 里两层 id 兜底 —— (a) 当 caller 不传 `id` 时拿 `params[key_params[0]]` 当主键，(b) 否则 `uuid.uuid4().hex` 静默造一个。流式实体（`messages` / `subagents` / `agent-memory` — `id_field != key_params[0]`）撞 (a)：scope-key 当主键，第一条成功第二条冲突 —— `chat_reply` 就是这个症状。即便躲过 (a)，(b) 的 silent uuid mint 也把"调用方忘填 id"的 bug 永远隐藏起来：插入成功但调用方自己的 id-tracking 是错的，没人能发现。**两层兜底都违反"无静默失败"原则**。**修复方向**：删 fallback，改 fail-fast。`_sql_create` / `append` 两处完全相同的 fallback 块替换为 loud `ValueError("missing required '<id_field>' on entity='<name>' ... Entangled does not mint ids (PR-40 fail-fast)")`，`entity_store.py` 里 `import uuid` 也顺手清掉（再无引用）。**关键不变性检查**：`AGENT_TOOLS_DEF` / `AGENT_STATE_DEF` / `AGENT_BINDING_DEF` 这三个"依赖 scope-key 做主键"的单例实体**从未真正依赖过 fallback** —— 代码最开头的 `for kp in defn.key_params: row[kp] = params[kp]` 循环会在 fallback 之前就把 `row[id_field]` 填好，所以删 fallback 不影响它们。**唯一违规调用方**：`tool_handlers._exec_chat_reply` 同步改为显式 `_uuid.uuid4().hex[:12]`，跟 `message_actions._store_add_message`（USER 消息路径）对齐，两个写消息路径收敛到同一 shape。全仓审计其它 `create/append` 调用方（`message_actions` / `files/registry` / `model_actions`×3 / `api_key_actions` / `subagent*`×3 / `agent-memory` / `devices`）全部已显式填 id。**测试**：Entangled 新增 5 cases（流式漏 id 必 raise / 显式 id 正常 / 单例不受影响 / INTEGER 自增不受影响 / 错误消息 grep-friendly），agent-runtime 新增 1 case（`test_chat_reply_mints_its_own_id` 回归守卫，两次 chat_reply 的 id 都是 hex12、互不相等、都不等于 agent_id）。全量测试：Entangled **105/105** + agent-runtime **72/72** + business **74/74** 全绿。生产脏数据（那条 `id=agent_id` 的旧行）保留不动当考古痕迹 —— `agent_id` 过滤查询不受影响。**架构判定**：一个看似"智能自愈"的 fallback，长期制造的 bug 远多于省掉的几行样板代码；通用 CRUD 代理的契约是"caller 显式一切"（id / timestamp / 其它 NOT NULL），Entangled 只做 runtime-fill-default（PR-35）和 runtime-required-check（PR-35），不做"猜你想要什么"。`SUBAGENTS_DEF` / `AGENT_SKILLS_DEF` / `AGENT_MEMORY_DEF` / `API_KEY_MODELS_DEF` 这几个 `id_field != key_params[0]` 的实体，未来任何新调用方若漏填 id 会在 CI 第一次跑就爆 `ValueError`，再也不会重演 "chat_reply stuck after one reply" 这种慢性症状。
