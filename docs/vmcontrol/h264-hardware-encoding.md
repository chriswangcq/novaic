# 硬件编码：VideoToolbox 与消颤防慢放算法

> 路径参考：`vmcontrol/src/capture/` （含 macOS VideoToolbox 魔改内容）

## 1. 软硬天堑：压垮 CPU 的渲染色域
原版很多远程桌面工具（包括老版本的 NovAIC 控制层）直接进行全桌面抓屏拿到底层色域后，如果采用 FFmpeg 用 CPU 死板地进行压缩：不但风扇狂转导致发热导致降频，更是会引发布局撕裂。
我们后来全面接入使用 MacOS 原生的 **VideoToolbox 硬件压缩管线**。这是显卡层面原生的硬件调用机制。即使你开着极度复杂高能游戏，打出的流也丝毫不会让 CPU 满载。

## 2. 修正魔改：抗掉线慢放 (Anti-Slowmo)
在移植这类高阶库（例如借鉴了远控圈子如 RustDesk 等项目的某些底层原理）打入我们的业务时，如果遭遇极端的“网络波动”，标准的 H264 传导极其容易出现：接收端（即我们的端侧 Web 视频）为了追求所有关键帧完整解包，而在某几帧极度延迟到达时采取降速拖长缓冲。导致了所谓的“画面时延越来越慢甚至积累达到惊人的十秒级延迟”。

解决方案是修改底层控制编码器的传输阈值以及发包缓冲逻辑：
- 我们实施了更严苛的丢帧 (Frame Dropping) 协议。对于不需要绝对不丢数据而仅仅要求高实时性的远程控制画面，一旦网络反馈 RTP 回包超时，主动选择忽略发送冗余帧去促使下一次强制要求给一发绝对的关键帧（IDR帧）。
- 控制使用了 **CBR (Constant Bitrate)** 而不是随意波动的动态码率，确保不会给网络打爆造成无意义的网络阻塞（这在高密度的云指令集反馈时尤为重要，免得画面堵死信令）。
