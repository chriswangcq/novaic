# 云网安环境与 Nginx 部署管线 (Deployment & Web-Server Routing)

> 路径：涉及根部 `部署集群机房 Nginx 配置文件` 和 `deploy_scripts`。

不要以为那些底层的代码跑起来发在 `0.0.0.0` 就算万事大吉了；由于含有极具隐秘且危险甚至能够控制真机的底层透传端口，在公网上发包跑测那是被大裸奔。

---

## 1. 隐形大墙 Nginx 的反向代理切割
在我们真实的域名配置（即你熟悉的 `api.xxx.com` 等前端入口）。Nginx 被配置成了带有多重屏障重切规则的高阶守门大将。

**路径规整策略 (Path Re-Routing)**:
- 面向用户的接口（如前端登录、发消息 HTTP）：由路由匹配规则 `/api/` 切下送发到藏在后面的 `Gateway (19999)` 去承接。
- 危险区保护 `/internal/`：这是那些下级系统模块们互相之间说“黑话”乃至诸如有机器进行握手挂载的地方。我们在 Nginx 层严防死守、乃至切除了无带有服务端间验证密钥头的调用拒绝访问。
- `Factory 映射域` (/factory-api/)：如果有涉及到诸如工厂的大配置项及监控数据需要被特权调阅，这是被单独发往另外一组（在 `services.json` 全不记载因为其过于隐秘并且常是跨机）LLM proxy 端点的接口。

## 2. 身份防偷漏：Auth_request 鉴权挂载
这使得你不必在开发比如向 Gateway 提交数据或者查询自己身份的某个单独方法上，又写一遍非常恶心的 JWT Token 解析。
我们依靠在 Nginx 设置全局通拦截点 `auth_request` 导回 `Gateway (auth 模块里的 /internal/auth/validate)` 拦截器中端校验：
只有如果通过并获得了用户的正确解包信息后（注入到 Header 并带行 `X-User-ID: xxxxxx`），再重新将请求带下去给目标机器。整个系统底层甚至不再需要碰你的凭证字符串。

## 3. 部署管线 (`./deploy`) 与环境剥夺
由于存在这种种不同机器配置分发！
你看到了为什么我们会有 `deploy frontend`, `deploy gateway`, `deploy relay` 这种各自为战互相独立的启动。
因为我们根本不在乎是否把整个玩意打在一个惊天大机器里的全家桶里运行。各个独立的模块甚至能完全通过带起各种各样针对机器优化的特有云镜像（甚至针对视频特优的高宽带 Relay 服与纯算力的 Worker 机器切分）。通过强大的内大网协议栈在远近跨洲组网互换。构成超分集散部署体系。
