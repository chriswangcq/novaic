# 19996 终极撞车与 Services.json 动态路由

> 路径参考：`common/config/services.json` 及全环境获取

## 1. 那一年被抢占的端口：19996 事件
如果你作为开发者在本地（macOS）拉起了前端 `novaic-app` 后面的那个自带 Rust 终端底座（它硬性启动且强占用着 `19996` 提供本地云桥或 WebRTC 收发）。
然后你此时又突发奇想不想去测远端的云服务器了，你想在本机也跑起强大的那组 `novaic-cortex` 或大脑组件。
你会绝望且抓狂地发现：`Cortex` 按照我们远端云上的设定，这尊拥有海量存储检索的大脑也是常年挂死并在 **19996**。
一跑，闪退！报错 `Address Already in Use`。

## 2. 环境偏移覆盖（Override Fallback）
对于这种业务设计上本处于不同机器不可能撞车，却因为程序员的“想把宇宙塞进本地一台电脑做测试”的行为而导致的系统事故。引入了多层覆盖权解决：
1. **基础法则映射表 (`services.json`)**：记载了那不可撼动的各模块常规映射表。
2. **ENV 最高权霸占法**：我们在获取这套配置的工厂函数上，加入了向对应运行语言所属当前环境提取的特权拦截。
即如果开发者在 `.env` 中塞入：`CORTEX_PORT=20000`。
即使 `services.json` 获取的还是原始数据，装载器会自动并掉合并进行 Override（特异化覆盖）。于是此时你电脑里跑的那个 `Cortex` 退了一步，缩放让出了 `19996` 自己安安静静在两万口候命。

## 3. 请求发现联动
而在那些原计划调控的调用者程序（在同样一台机器并挂着相同的系统 ENV 的其他 Worker 里），当读取到了 `CORTEX_PORT` 有这封口令后，它们去向发起 `cortex-api` 的目的地也就自然跟随漂移向了 `20000`。
完成了完全无血和不碰伤源码的开发环境冲突避免。
