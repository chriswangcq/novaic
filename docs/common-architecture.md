# 核心基座与共享契约：Common 库总览

> 对应路径：`novaic-common` 跨语言公共仓库

如果说各种微服务是强健的手脚与内脏，那么 `novaic-common` 就是系统里的**大动脉与神经协议网**。
由于我们在前后单端横跨了 Python, Rust, TpyeScript 等互相甚至不互通基本类型的极多开发语言环境；为了避免出现“前端用驼峰去问后端，后端只能给下划线拼装报错”这种经典的异构项目大雷，我们将这些共通元素全数压入了公共包。

---

## 1. 源于定义，生出三端 (Schema & Type Source of Truth)

Common 下面并没有多复杂的“代码或循环结构”，它极其纯粹：
- 所有的数据模型（Models）, 以及通信握手的 Pydantic Schema 全数存放于此。
- 在构建或者发版之时，我们将一套原始的定义表转化（借助各种自动化 Generator 比如说 Pydantic-to-TypeScript）向各异构语言派生类型定义文件。
这确保了即使某个粗心大意的开发为了网关改了返回实体的 `field` 名字；Rust 的编译阶段和 React 的 npm 检修期会立刻双重爆红拦截住这套类型安全！阻止线上雪崩！

## 2. services.json 字典总表发现法

如果各种端通过硬编码诸如 `127.0.0.1:19999` 来跑。不仅一旦运维发生大规模端口迁移会牵一发而动全身；在本地开发者（没有各种域名的恶劣环境）想要一键调试线上云脑时会死活改不全环境配置。
在 `common/config/services.json`：
我们编写了庞大的字典本：
```json
{
  "cortex": { "port": 19996, "path": "/api" },
  "gateway": { "port": 19999, "path": "/"}
}
```
并且提供了工具类在各语言下发获取。如果检测到拥有 `https` 特征的大规模部署域，它就会顺接解析组合上域名外围；如果是 `localhost` 或检测不到特殊环境：它就是那最可靠好找的跨端口直连测试版！
参阅同级 `docs/common/` 字典探索它是怎么解决本地 `Cortex` 与云桥 `vmcontrol` 被魔鬼级的端口 `19996` 冲突撞车的发报机。
