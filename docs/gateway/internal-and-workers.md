# 内部调度与 Worker 并联体系

> 路径参考：`novaic-gateway/gateway/api/internal/` 组 及其它相关协同调用脚本

## 1. 防护栅栏：Internal 接口认证机制
所有分布在 `/api/internal/*` 的端点并不是随便开给外网访问的（它通常在 Nginx 配置阶段就会阻挡所有发自公网域名的直接路由或仅放权特定头）。
由于它要接受由局域网里的兄弟（比如 Queue Worker 和 Agent 环境下的 Sagas 及 LLM Factory 传递反馈），通过读取在启动环境设置里提供的校验头秘钥 `INTERNAL_TASKS_SECRET`，所有非公用的任务通过带有内部暗号通过身份门槛而无需复杂的 JWT 配置签发。

## 2. 内部行为范畴
在此暴露的服务大量集中在以下这几方面反向代操：
- **VM 回馈/Agent状态调整**：
  在远端（可能根本不是我们本地的云）虚拟机内的 Worker 或者 Runtime。当他们处理完毕一场对话、或者发生了一个 SubAgent 自己生成或者出错挂起的生命流转时。它是没法从外部写那些网关独有的操作库的（他只有权限写自己的状态机），它会发 Http 到这里，命令网关帮其挂在界面可以呈现的结果区域：如改写 `status` 等生命周期信号发送给使用者。
- **与 Factory 和 Cortex 各自安好**：
  网关本身现在不持有与大语言模型连接或与智能分析组件 DFS 回撤连接的核心。
  网关仅仅给各界暴露关于使用者的环境基础（即：配置）。各处如果要问使用者是否设置好了新模型或者它选了啥，全向通过提供其 API。

通过这道接口隔离层，所有的执行逻辑均被剥离成“微服务形式的孤岛”，如果将来某天 Worker 全被用另一种语言（如 Go/Rust）进行重写，对这个中央调度主控网关来说也毫不费力即可通过这组 `/internal` 完成解耦无痛相接。
