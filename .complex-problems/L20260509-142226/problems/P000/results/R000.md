   - Cortex-rendered tool messages also do not carry `_round_id`.
   - Therefore `include_display` defaults true for all tool results, so old screenshots are repeatedly expanded into LLM-visible image/base64 content across later rounds.

## Root Cause

The main slowness is not CPU, DB load, queue health, or device tool execution. It is prompt/context bloat caused by screenshot payloads being repeatedly re-expanded into LLM context. Multi-round tool workflows multiply this cost.

## Recommended Fix Direction

1. Pass `current_round_id=source.round_id` from `prepare_llm_call()` into `expand_messages_for_llm()`.
2. Preserve round metadata on Cortex-rendered tool messages, or derive it from stored step metadata, so historical tool results can be rendered as text previews instead of display/base64 content.
3. Add a hard budget guard: historical screenshots/images must become placeholders or payload refs, never inline base64.
4. Add tests for a multi-round screenshot flow: after round N, only current-round images may be display-expanded; prior screenshots must remain bounded text.
5. Add operational cleanup separately: log rotation/retention and a stale finalize-task repair job.
