# Tool optimization audit result

## Summary

The audit found several tool optimization candidates. The key issue is not limited to `hd_screenshot`: the current tool execution contract is too generic (`dict -> JSON content -> Cortex payload -> formatted _mcp_content`) and does not distinguish text, large text, binary, image, audio, files, artifacts, UI-only display, and model-visible perception.

The best next direction is to introduce a first-class `ArtifactResult` / `ToolResultProjection` contract and migrate high-risk tools to resource-first outputs. `payload_*` tools and user attachment handling are already close to the desired shape. `hd_screenshot`, dynamic device tools, shell large output, `display`, and legacy multimodal projection are the main areas needing optimization.

## Verification

- Runtime `_EXECUTORS` includes `shell`, `display`, `audio_qa`, `payload_*`, `skill_*`, environment tools, and dynamic HD device tools.
- Business exposes mounted dynamic tools via `/agents/{agent_id}/builtin-tools`, while Runtime only provides executor routing for dynamic device tools.
- Cortex already externalizes large raw payload JSON above a threshold, but `read_formatted` can still reconstruct model-visible `_mcp_content` from stored payloads.
- User attachments already use the right reference-first strategy: image/audio attachments become textual hints to call `display` or `audio_qa`, not inline base64.
- Payload inspection tools are bounded and redacted enough to be a good model for other tools.

- Dynamic Device Service response schemas were not fully inspected here, so device-specific output normalization should be a follow-up implementation ticket.

## Audit Findings

### P0: Shared Tool Result Projection Needs A Typed Contract

Current Runtime tool handling is intentionally unified:

```text
executor(args, deps) -> dict -> json.dumps -> context.append -> Cortex payload
```

This is good for execution uniformity, but insufficient for context semantics. A dict can contain:

- small text;
- huge stdout;
- `_mcp_content` with image base64;
- binary metadata;
- files;
- display artifacts;
- UI-only data;
- model-visible perception.

Those need different projection policies. The current path collapses them too early.

Recommended contract:

```json
{
  "text": "small bounded summary",
  "artifacts": [
    {
      "uri": "blob://...",
      "modality": "image|audio|text|binary",
      "mime_type": "...",
      "size_bytes": 123,
      "sha256": "...",
      "display_hint": "display(file_url=\"blob://...\")",
      "context_policy": "manifest_only|current_turn_display|text_slice"
    }
  ],
  "llm_projection": {
    "current_turn": "bounded_display_or_text",
    "history": "manifest_only"
  },
  "monitor_projection": {
    "kind": "thumbnail|text|status"
  }
}
```

This should become the common substrate. Tool code can stay small, but output shape must be explicit.

### P0: `hd_screenshot` And Dynamic Device Tools Need Resource Normalization

`execute_mounted_device_tool()` currently returns the Device response as-is. That is too trusting for dynamic tools. `hd_screenshot` is the known failure case: screenshot-like data can end up as display/base64 in stored tool payloads, then get replayed by context projection.

Needed:

- Runtime-side normalization for each HD tool output class.
- `hd_screenshot`: Blob artifact + optional thumbnail + manifest, not raw base64 durable payload.
- `hd_file_pull`: file/blob artifact, not inline content except tiny text previews.
- `hd_shell_exec`: same policy as shell output: bounded stdout/stderr preview plus payload/file refs.
- `hd_clipboard_get`: bounded text, secret-aware redaction.

Dynamic does not mean untyped. The dynamic mount determines availability; Runtime still needs output semantics.

### P1: Shell Tool Large Output Is Still A Context Risk

Cortex shell execution returns full `stdout`/`stderr` without the `tool_read` truncation path. Large shell results may be externalized by Cortex payload storage, but `read_formatted` and fallback JSON serialization can still pull the large text back toward model context.

Needed:

- Shell result projection should include bounded `stdout_preview` / `stderr_preview`.
- Full stdout/stderr should be a `payload_ref` or workspace file artifact.
- Prompt should say: "output truncated; use payload_read/search or shell file reads".
- Shell should optionally support `max_output_chars` or "redirect large output to file" guidance at schema/tool level.

This is lower than screenshot but still important because shell is the most flexible high-volume tool.

### P1: `display` Should Be Marked Ephemeral Perception

`display(file_url=...)` is conceptually correct: it is the agent intentionally dereferencing a Blob resource. But today `_exec_display` returns `_mcp_content` with image base64 for small images. That is fine for the immediate turn, but dangerous if stored history later replays it.

Needed:

- `display` result should carry an explicit `ephemeral_perception=true` or `history_projection=manifest_only`.
- Stored payload should retain `file_url`, mime, size, maybe thumbnail, but not durable base64.
- Context assembly should treat a historical `display` result as: "Agent looked at file X; inspect again with display if needed."

`display` should remain powerful, but its visual bytes should be current-turn perception, not memory.

### P1: Legacy Multimodal Utilities Are A Compatibility Trap

`utils/multimodal.py` and `utils/context.py::process_multimodal_messages()` still contain a generic path that detects image data in tool results, strips placeholders, and adds images as user messages. This was probably useful before step-ref/resource projection matured, but it is now an overlapping mechanism.

Risk:

- It creates a second image-injection path after Cortex `read_formatted`.
- It operates on generic `_mcp_content`, not typed artifacts.
- It can make prompt behavior hard to reason about.

Needed:

- Either delete this path after ArtifactResult migration or restrict it to current-turn explicit display grants only.
- Add tests proving historical tool result images cannot become synthetic user image messages.

### P1: `step_result_projection.py` Needs To Become Manifest-Aware

`parse_tool_result()` currently converts `_mcp_content` images into `display_files`, including data URLs. `format_for_llm()` may turn those data URLs back into model-visible image data when `include_display=True`.

Needed:

- Parse artifact manifests first.
- Treat data URLs as legacy input to be converted into Blob artifacts or manifest placeholders.
- Never preserve data URL as the durable representation.
- Split projections:
  - `format_for_history_llm()`: manifest only.
  - `format_for_current_turn_llm()`: bounded display allowed.
  - `format_for_monitor()`: UI thumbnail/render info.

### P2: `audio_qa` Is Mostly OK But Needs Size/Duration Guardrails

`audio_qa` downloads a Blob audio file and sends base64 audio to Factory for transcription/QA, then returns only text. That means it does not create repeated prompt bloat in the normal history path.

Still worth optimizing:

- explicit audio size/duration limit before base64 encoding;
- better metadata in result: file size, mime, transcript chars, truncation flag;
- optional transcript artifact if very long;
- no raw audio in durable history.

This is not urgent but should be aligned with the resource model.

### P2: Payload Tools Are The Good Pattern

`payload_read`, `payload_search`, `payload_summarize`, and `payload_qa` are already close to ideal:

- explicit `payload_ref`;
- bounded reads/search;
- max input/output clamps;
- redaction before interpretation;
- text-only `_mcp_content`.

They should become the pattern other high-volume tools imitate.

Potential improvements:

- support artifact refs, not only payload refs;
- make payload ref discovery more visible in context projections;
- add "suggested next reads" metadata after truncation/search.

### P2: User Attachments Are Also A Good Pattern

`user_content.py` already avoids raw attachment injection. It tells the agent to call `display` for images and `audio_qa` for audio.

This should be generalized:

- all environment/user/tool resources should appear as handles plus suggested tools;
- prompt text should not imply raw bytes are already seen unless `display` was called.

### P2: IM History/Search/Context Are Bounded And Low Risk

Environment tools clamp limits:

- `im_history`: limit 1-50;
- `im_search`: limit 1-50;
- `im_context`: before/after bounded.

They are not a major context-bloat risk unless message attachments are expanded downstream. Keep them as-is, but ensure they return attachment refs, not raw bytes.

### P3: Skill Lifecycle And Sleep Are Not Artifact Problems

`skill_begin`, `skill_end`, `sleep`, `subagent_spawn`, and IM send/reply are mostly text/control-plane tools. They do not need resource optimization beyond existing bounded summaries and lifecycle correctness.

### P3: Monitor Projection Needs Artifact Awareness

`activity_projection.py` marks `has_payload`, which is useful, but it does not yet distinguish:

- "has payload text";
- "has visual artifact";
- "has audio artifact";
- "current-turn perception";
- "UI-only thumbnail available".

Once `ArtifactManifest` exists, monitor projection should read it and render richer, without affecting LLM prompt assembly.

## Priority List

3. P1: Shell large-output projection.
4. P1: Make `display` explicitly ephemeral/current-turn perception.
5. P1: Remove or constrain legacy generic multimodal image injection.
6. P1: Make Cortex `step_result_projection` manifest-aware.
7. P2: Add audio size/duration/transcript artifact guardrails.
8. P2: Extend payload tools to artifact refs and better discovery.

## Suggested Implementation Tickets

### Ticket A: ArtifactResult Contract

Define shared schema and projection policy:

- `text`
- `artifacts[]`
- `llm_projection`
- `history_projection`
- `monitor_projection`
- compatibility parser for current `display_files`/`files_created`.

### Ticket B: Device Tool Output Normalization

Normalize `hd_screenshot`, `hd_file_pull`, `hd_file_push`, `hd_shell_exec`, and clipboard tools into typed results.

### Ticket C: Cortex Manifest Projection

Teach Cortex `read_formatted`, `read_preview`, and context stack merge to project artifacts by policy, not by raw `_mcp_content`.

### Ticket D: Shell Output Resourceization

Large stdout/stderr become payload/file artifacts with bounded previews and suggested `payload_read/search`.

### Ticket E: Display Ephemeral Perception

Make `display` current-turn visual dereference only; historical display results become manifest summaries.

### Ticket F: Legacy Multimodal Cleanup

Delete or gate `process_multimodal_messages` image extraction so only explicit current-turn display grants create image messages.

### Ticket G: Prompt Budget Regression Tests

Add tests proving:

- repeated screenshots do not grow historical prompt with base64;
- shell 1MB output yields bounded prompt;
- display current-turn works but history is manifest-only;
- payload tools remain bounded.

## Bottom Line

Yes, there are other tools/pathways to optimize. The real optimization target is a shared output contract across tools. Screenshots exposed the bug, but shell output, dynamic device outputs, `display`, and legacy multimodal projection are the places where the same class of bug can reappear.
