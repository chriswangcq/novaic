# Audit Tests for Stale Contracts

## Problem Definition

Tests can accidentally preserve old behavior: fallback paths, old stable paths, old tool names, legacy materialization APIs, or compatibility helpers. We need to audit tests for stale contracts and update/delete only concrete bad tests.

## Proposed Solution

Run targeted scans over tests in the key submodules, inspect high-signal hits, and classify them as current guard, historical/negative assertion, lower-layer generic test, or stale contract. Update or delete tests only if they protect old behavior.

## Acceptance Criteria

- Runs focused scans over tests for old path/tool/compatibility terms.
- Inspects high-risk hits in Cortex/runtime/common/business where contract drift matters.
- Rewrites or deletes tests that only protect stale behavior.
- Runs affected test files after any changes.
- Records retained hits with justification.

## Verification Plan

Use `rg` over test directories for materialize, scratch, sandbox backing paths, fallback/compat/legacy, Blob object authority, and base64/tool-output contracts. Run affected pytest commands if test files change.

## Risks

- Some tests intentionally assert rejection of old paths; those should be retained.
- Lower-layer LogicalFS tests may use arbitrary paths that look stale but are not product contracts.

## Assumptions

- This child is test-focused; docs and CI guard scripts are handled by sibling children.
