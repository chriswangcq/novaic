# Audit Docs Architecture Stale Claims

## Problem Definition

Architecture docs, runbooks, and roadmap pages may still contain stale active-looking claims around old paths, old tool surfaces, old Blob/LogicalFS layering, or migration residue.

## Proposed Solution

Run targeted docs scans, inspect high-signal hits, and patch only docs that are current-facing and misleading. Historical roadmap content should be retained only if it is clearly historical or bannered.

## Acceptance Criteria

- Runs focused docs scans for old paths, tool names, compatibility words, materialization, scratch, Blob workspace authority, and shell/display contract residue.
- Classifies high-signal hits.
- Updates or deletes misleading current-facing doc text when found.
- Records retained historical hits with justification.

## Verification Plan

Use `rg` scans over docs and selected root Markdown files, inspect context with `sed`, and run docs guard scripts if changed docs interact with existing lints.

## Risks

- Roadmap archaeology intentionally mentions old names; over-editing it would erase useful history.
- Some terms like `fallback` and `compatibility` are valid when explicitly warning against old paths.

## Assumptions

- Current-facing architecture/runbook docs should use the final contract; old migration tickets may remain if clearly marked historical.
