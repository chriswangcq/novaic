# Review Guard False Positives and Stale Assumptions

## Problem Definition

Architecture guards can become harmful if they encode stale assumptions or use broad string bans that reject valid lower-layer code/tests/docs. We need to review high-signal guards for this risk.

## Proposed Solution

Inspect the guards most likely to false-positive: Blob workspace boundary, generated artifacts, retired runtime vocabulary, lifecycle ownership, wake continuity, no legacy file hot paths, and Cortex boundary. Record whether each has precise scan roots, allowed exceptions, and contract-specific terms. Patch only concrete false-positive/stale-assumption issues.

## Acceptance Criteria

- Reviews high-signal guard scripts for scan roots, broad terms, exceptions, and lower-layer valid-use cases.
- Fixes concrete stale or false-positive-prone guard logic if found.
- Records retained broad terms with justification if no patch is needed.

## Verification Plan

Use direct source inspection plus a few dry-run guard commands where cheap. If a guard is changed, rerun it.

## Risks

- Changing guards without a concrete issue can weaken protection.

## Assumptions

- Generated artifact cleanup already fixed the known failing guard condition in P666.
