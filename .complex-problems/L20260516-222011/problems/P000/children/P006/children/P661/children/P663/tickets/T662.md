# Remediate Architecture Guard Gaps

## Problem Definition

The inventory shows many guards, but we need to compare them against the current contracts and fix concrete guard gaps or stale guard assumptions. This is where optimization may require code changes.

## Proposed Solution

Use the inventory to build a contract checklist, inspect existing guards for each contract, and patch narrowly scoped guard gaps. Focus on gaps that could allow old logic to re-enter: raw media/base64 in normal LLM context, direct Blob workspace authority outside allowed layers, stale `/rw/scratch` Cortex contracts, sandbox backing path reuse, generated artifacts, stale runtime compatibility vocabulary, and deployment/service topology drift.

## Acceptance Criteria

- Compares current contracts against the guard inventory.
- Identifies concrete missing/stale guard coverage or records no-gap decisions with evidence.
- Adds/updates guards for concrete gaps only.
- Avoids over-broad static bans that block lower-layer generic tests/docs.
- Records changed files and rationale.

## Verification Plan

Run static scans and inspect guard code. If guard code changes, run the changed guard plus adjacent guard tests. If no changes are made, record evidence and defer execution proof to the verification sibling.

## Risks

- Guard changes can be too broad and create false positives.
- No-change decisions can miss gaps unless mapped against explicit contracts.

## Assumptions

- The current desired contracts are the architecture from prior work: Blob as cheap byte/object server, LogicalFS as live `/ro` `/rw` authority, sandbox as execution service without business ownership, Cortex as state semantics/context service, shell visible output as bounded terminal text, display as explicit visual perception path, and Queue/Runtime state via FSM/outbox.
