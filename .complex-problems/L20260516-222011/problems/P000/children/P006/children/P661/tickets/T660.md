# Audit Architecture Guards for Current Contracts

## Problem Definition

Architecture and CI guard scripts must enforce the current contracts after the recent FSM, Cortex event stream, shell/display output, Blob, LogicalFS, and sandbox layering work. Documentation and tests are not enough if static guards still miss old paths or protect stale assumptions.

## Proposed Solution

Inventory existing root and module-level guard scripts, map them to the current contracts they protect, scan for missing or stale coverage, and add or adjust narrowly scoped guards only where the inventory reveals a concrete gap. Re-run changed guards and any directly relevant existing guards.

## Acceptance Criteria

- Inventory root CI/static guard scripts related to old paths, tool contracts, lifecycle, Blob/LogicalFS boundaries, generated artifacts, and deployment/runtime entrypoints.
- Map each corrected/current contract to at least one guard or explicitly justify why tests/docs are the right boundary instead.
- Identify stale guards that still encode retired assumptions, if any.
- Add/update guards only for concrete missing coverage.
- Run changed guards and record outputs.

## Verification Plan

Use `rg`, `find`, and direct script inspection over `scripts/ci`, `.github/workflows`, and relevant module test guard files. Run any modified guard scripts and a small set of adjacent existing guards. If no changes are needed, record the inventory and targeted guard execution evidence.

## Risks

- Over-broad static guards can block legitimate lower-layer generic tests or docs; keep guard scopes precise.
- Some contracts are better protected by runtime unit tests than static text scans; do not duplicate low-signal guards.

## Assumptions

- This ticket is guard-focused; implementation behavior and stale tests are handled by sibling problems.
- Current architecture favors explicit boundaries and physical cleanup over compatibility branches.
