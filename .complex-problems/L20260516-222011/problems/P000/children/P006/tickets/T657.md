# Audit Tests Docs and Architecture Guards

## Problem Definition

The project may still contain stale docs, tests, guard scripts, old tool names, old paths, and migration residue that mislead future agents or allow old contracts to return. We need audit and optimization across docs/tests/architecture guards.

## Proposed Solution

Split the audit into focused child problems: stale docs/architecture claims, test/guard coverage, and old contract/path/tool-name residue. Remove or update misleading material and add targeted guard tests when a real contract gap is found.

## Acceptance Criteria

- Searches docs/tests/scripts for stale architecture claims, old contracts, old tool names, and misleading migration residue.
- Reviews existing architecture guard tests/scripts for coverage gaps.
- Updates or deletes stale/misleading material that is active enough to mislead future work.
- Adds or adjusts tests/guards for corrected contracts.
- Records remaining historical material as intentionally archived or creates follow-up work.

## Verification Plan

Use focused `rg` scans, inspect high-signal hits, update docs/tests/scripts only when needed, and run affected guard/test commands. Prefer targeted guards over broad noisy cleanup.

## Risks

- Historical roadmap docs intentionally contain old terms; deleting them blindly can erase useful context.
- Guard scripts can become noisy if too broad; they should protect specific contracts.
- Some dirty files may belong to prior work and should not be rewritten without clear relevance.

## Assumptions

- Physical cleanup is preferred when stale material is misleading and not purely historical.
- Archived docs are allowed only when clearly marked as historical/current-baseline-safe.
