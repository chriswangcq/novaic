# Classify Provider Request Serialization And Multimodal Projection

## Problem Definition

P579 must audit provider request serialization and multimodal projection paths to verify display/image content is sent to the LLM provider as structurally valid image content, not as raw base64 text embedded in a normal text field.

## Proposed Solution

Run targeted scans over Runtime provider preprocessing, multimodal conversion, Factory client transport, provider request-body tests, and any OpenAI/Anthropic image formatting branches. Record outputs, focused line slices, and classify each hit as intended, risky, removable, or follow-up.

## Acceptance Criteria

- Exact scan commands and outputs are recorded.
- Provider request builders and multimodal conversion slices have line references.
- Current-turn display/image content is proven to reach the provider in a valid structured shape.
- Raw base64-as-text and legacy compatibility branches are classified.
- Any high-confidence risky residue is forwarded to P554 remediation.

## Verification Plan

Search runtime and factory/client code for `image_url`, `base64`, `data:`, `multimodal`, `messages`, `chat/completions`, and provider adapters. Read line-numbered slices and run or cite focused tests that capture provider request body shape.

## Risks

- Provider APIs differ; a valid OpenAI shape may not be valid for another provider.
- Some data URLs are valid provider image URLs, so not every base64 string is a bug.
- The recent UI screenshot may have shown request body only before provider-side transformation; classify by actual code path.

## Assumptions

- OpenAI-compatible Factory accepts structured multimodal message content arrays.
- Historical tool messages must remain text/manifest-only; current display perception may become image content.
