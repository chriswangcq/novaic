# Classify Final Residual Risk and Worktree State

## Problem Definition

The final verification must not overclaim. We need to record the current worktree state, distinguish audit-owned changes from visible pre-existing/unrelated dirty state where possible, and state what local verification cannot prove about external repos or deployment.

## Proposed Solution

Run `git status`, concise diff stats, and targeted file lists. Classify changed files by ownership of this audit versus broader/pre-existing work visible in the repository. Record residual risks and whether any generated/untracked artifacts need follow-up.

## Acceptance Criteria

- Records current `git status` and concise diff statistics.
- Lists audit-owned changed files and evidence artifacts.
- Notes visible unrelated/pre-existing dirty state when identifiable without reverting it.
- Records local-vs-deployment/external-repo residual risk.
- Creates follow-up if a residual risk is blocking for this optimization audit.

## Verification Plan

Use git read-only commands and ledger tmp evidence. Do not stage, commit, deploy, or revert anything in this ticket.

## Risks

- The worktree may contain unrelated dirty files from prior tasks; do not conflate them with this audit.
- Git status across submodules may be noisy.

## Assumptions

- This is reporting/classification only; submit/deploy is a separate user command.
