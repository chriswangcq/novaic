# Blob Artifact and Display Contract Audit Ticket

## Problem Definition

Media/large-output CLI tools must behave like terminal commands: stdout should contain bounded text plus artifact manifests, not raw bytes/base64. `display(blob://...)` is the explicit perception path, but its historical/public tool projection must not reinsert raw base64 into future context.

## Proposed Solution

Split the audit across artifact-producing CLIs, display history/current projection, provider-native image handoff, and final base64 residue/regression checks. Fix any bounded gap found and rerun focused tests.

## Acceptance Criteria

- Media-producing shell CLIs return `tool-output.v1` manifests with `blob://runtime-artifact/...` refs, not raw base64 stdout.
- Display tool public/history projection is sanitized and text/placeholder only.
- Current display perception can still reach the provider as native image content when needed.
- Regression tests prevent raw image/audio bytes or `data:image` text from leaking into ordinary shell/display history context.

## Verification Plan

Inspect artifact manifest helpers, display executor/projection code, multimodal/context assembly, and tests; run focused unit tests; add minimal guards for uncovered contract edges.

## Risks

- Shell artifact manifest can be correct while display history still leaks image data.
- Sanitizing display too aggressively could break current-turn image perception.
- Tests may pass for screenshots but miss file-pull or audio-like media paths.

## Assumptions

- Direct provider image payloads may use base64/data URLs internally only at the provider adapter boundary; that is distinct from tool text/history projection.
