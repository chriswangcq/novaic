# Audit and remediate extracted service entrypoint residue

## Problem Definition

The monorepo has 8 services with launch scripts/entrypoints. Exploration reveals:
1. **Queue Service duplicate entrypoint**: `novaic-agent-runtime/queue_service/main.py` (standalone FastAPI app) and `novaic-agent-runtime/main_novaic.py queue-service` (CLI subcommand). All launch scripts use the CLI path. The standalone `main.py` is residual.
2. **novaic-llm-factory orphaned entrypoint**: has `main.py` (port 9100) and `factory/app.py` but is absent from all three launch scripts (`start.sh`, `start-backends.sh`, `launch_split_only.sh`). Orphaned service.
3. **Blob Service and Sandboxd each have two entrypoint files**: `main_*.py` wrapper + internal `main.py`. Need to verify both are intentional (wrapper pattern vs. duplicate).
4. **novaic-app/scripts/start-backends.sh**: may reference stale ports or services.

## Proposed Solution

Split into children:
1. Audit queue_service/main.py residual — determine if it can be deleted or is needed for direct testing.
2. Audit llm-factory launch status — determine if it should be added to start.sh or documented as standalone.
3. Audit blob/sandboxd dual entrypoints — verify wrapper pattern is intentional.
4. Cross-verify launch scripts against current service topology.

## Acceptance Criteria

- Each service entrypoint has a clear disposition: active, residual, or documented.
- Stale/duplicate entrypoints are removed or documented with rationale.
- Launch scripts match current service topology.
- Focused tests/lints pass after any changes.

## Verification Plan

- Read each entrypoint and classify.
- Check git blame for context on duplicates.
- Verify start.sh service list against actual service topology.
- Run relevant tests after any deletions.

## Risks

- Deleting a "residual" entrypoint that is actually used for local development or testing.

## Assumptions

- The three launch scripts (start.sh, start-backends.sh, launch_split_only.sh) are the authoritative launch surfaces.
