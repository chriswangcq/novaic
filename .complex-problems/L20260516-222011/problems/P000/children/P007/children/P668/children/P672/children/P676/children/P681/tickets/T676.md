# Verify and Patch Deployment Script Guards

## Problem Definition

After local and packaged launcher remediation, guard coverage must prevent old deployment/start contracts from returning. We need inspect existing deployment/resource guards and patch low-risk gaps around active launcher wording and stale packaged port claims.

## Proposed Solution

Inspect deployment-related guard scripts and tests, especially `test_no_legacy_file_hot_paths.py`, `test_app_resource_hygiene.py`, `check_start_config_contract.py`, `lint_deploy_fresh_smoke.py`, and `lint_runtime_worker_supervision.py`. Add focused guard coverage if no existing guard catches old all-backends/fake-port wording in active launcher files. Run changed guards locally.

## Acceptance Criteria

- Existing deployment guard scripts/tests are inspected against P679/P680 findings.
- Guard coverage either already protects or is patched to protect the corrected launcher contract.
- The guard avoids broad false positives over historical docs/build output.
- Changed guard tests/scripts pass locally.

## Verification Plan

Read guard scripts/tests, patch the most targeted active-path guard if needed, then run focused pytest/CI scripts. Use targeted `rg` scans to confirm old terms are absent from active launcher files.

## Risks

- Too-broad guard terms may false-positive historical roadmap docs or target build output.
- Guarding generated copies can be brittle if build output is included; target only checked-in active source/resource paths.

## Assumptions

- Guarding active launcher files and checked-in resource copies is enough; target/debug build outputs are generated and not authoritative.
