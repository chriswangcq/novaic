# Documentation media-byte residue classification result

## Summary

Documentation mostly aligns with the current Blob/artifact shell/display contract. The current contract docs explicitly say media-producing CLIs must return `tool-output.v1` artifact manifests and must not print raw bytes/base64 into shell text. One stale/misleading doc was found: `docs/mcp-vmuse/mcp-protocol-mapping.md` still describes VMuse MCP tooling as directly introduced into Runtime/LLM, which conflicts with the newer shell/Device proxy architecture and should be updated or marked historical.

## Done

- Scanned docs for `base64`, `data:image`, screenshot, `_mcp_content`, `tool-output.v1`, `runtime-artifact`, `blob://`, and `display(...)`.
- Classified current contract docs:
  - `docs/runtime/tool-chain-dispatch.md`: current and correct; states `devicectl hd screenshot` returns `tool-output.v1` artifact manifest with `blob://runtime-artifact/...`, and `display(blob://...)` is the explicit image perception path.
  - `docs/cortex/sandbox-shell.md`: current and correct; states `devicectl ...` can return `tool-output.v1` artifact manifest and should not print image/audio/base64 bytes to stdout.
  - `docs/blob-service-architecture.md`, `docs/reference/blob-service.md`, `docs/reference/blob-audio.md`, and `docs/roadmap/blob-large-file-multipart-audio.md`: current/historical mix but explicitly say base64 upload paths were removed or historical.
- Classified lower-level protocol docs:
  - `docs/gateway/cloudbridge-vm.md`: current lower-level CloudBridge typed command doc; `vm_screenshot` mention is protocol-level, not LLM shell context.
- Classified stale/misleading docs:
  - `docs/mcp-vmuse/mcp-protocol-mapping.md`: stale relative to current architecture. It says Runtime can directly introduce VMuse MCP tool schemas and send image/success payloads back through that path. Current architecture has final exposed tools narrowed to shell/display/skill/sleep, with VMuse reached through Device/VMuse proxy under shell capability wrappers.
- Classified illustrative/historical docs:
  - `docs/frontend/tauri-ipc-commands.md` mentions `base64_image` as an IPC anti-pattern example, not an active contract.
  - `docs/roadmap/tickets/*` are historical ticket records; many base64 mentions are explicitly about removal or previous states.
