# Blob artifact manifest and history replay discovery ticket

## Problem Definition

We need to verify how `tool-output.v1` artifact manifests and Blob URIs are persisted and replayed through Runtime/Cortex history, especially whether old display/image outputs are replayed as manifest-only text instead of raw base64 or provider image content.

## Proposed Solution

Inspect Runtime tool result parsing/projection, Cortex step result projection, context event projection, and tests that cover historical image replay. Record exact file pointers for manifest parsing, durable payload storage, display files, LLM-visible history projection, and any guardrails preventing historical media re-injection.

## Acceptance Criteria

- Artifact manifest parsing/storage path is identified with evidence.
- History replay behavior for screenshot/display artifacts is identified with evidence.
- Tests proving no historical tool image re-injection are identified or gaps recorded.
- Any active history replay path that serializes large media/base64 into LLM text is listed as a remediation candidate.

## Verification Plan

Run targeted searches and file inspections for `tool-output.v1`, `display_files`, `llm_content`, `history`, `public_content`, `durable_payload`, `raw`, `base64`, `data:image`, and `blob://runtime-artifact` in Runtime/Cortex projection modules and tests. Run focused tests if dependencies are available.

## Risks

- Runtime and Cortex each have projection code, so evidence must not stop at one package.
- Tests may encode expected behavior but not prove current deployment wiring.
- Old archived contexts can contain old base64; this ticket is about active replay behavior, not historical data cleanup.

## Assumptions

- Discovery only; remediation is deferred.
- Blob remains cheap binary storage, not live LogicalFS state authority.
