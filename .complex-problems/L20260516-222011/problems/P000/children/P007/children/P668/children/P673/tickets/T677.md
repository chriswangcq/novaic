# Inventory Worker and Service Entrypoints

## Problem Definition

The backend architecture has evolved through FSM, queue-service workers, sandbox/logicalfs/blob extraction, and launcher cleanup. We need a source-evidence-based topology map of worker and service entrypoints so user-facing explanations and deployment scripts do not rely on stale memory or old process names.

## Proposed Solution

Scan source/config/package metadata for executable entrypoints and launch commands across queue service, runtime workers, Cortex, sandbox, LogicalFS, Blob, health, scheduler, saga/outbox/task workers, and related service wrappers. Split the work into bounded subproblems for entrypoint discovery, role classification, stale/duplicate entrypoint remediation, and verification/docs alignment if the inventory is too broad for one safe pass.

## Acceptance Criteria

- Entrypoint files and launch commands are discovered from repository evidence rather than memory.
- Current worker/service roles are summarized with file paths and command/config pointers.
- Stale, duplicate, or misleading entrypoint code is either safely removed/updated or recorded as an explicit residual risk.
- Any changed entrypoint scripts/modules pass relevant syntax/import/static checks.
- The final result is small enough to support later backend-architecture explanations without carrying large raw scan output in context.

## Verification Plan

Use `rg`, `find`, package metadata inspection, and focused config reads to build the entrypoint map. If edits are made, run targeted Python import/compile checks, shell syntax checks, package-specific tests, and focused stale-term scans. Record scan artifacts in `.complex-problems/L20260516-222011/tmp/` and cite them in results/checks.

## Risks

- Entrypoint terminology may differ across Python modules, shell scripts, docker/deploy scripts, and app resources; broad scans can create noisy artifacts.
- Some duplicate-looking entrypoints may be intentionally different for local, packaged, and cloud contexts.
- Removing stale entrypoints without proving they are inactive could break deployment or packaged app behavior.

## Assumptions

- A split is likely appropriate because discovery, classification, remediation, and verification are separable and the user explicitly wants careful multi-depth ledger work.
- Live deployment is not required for this entrypoint inventory unless a later check creates a deployment-specific follow-up.
