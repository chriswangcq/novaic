# Residual production media-byte classification ticket

## Problem Definition

The first production media-byte scan classified the main Runtime/Cortex shell and display paths, but it left residual non-test production hits too broadly grouped. We need to finish classification at module/path granularity so remediation candidates are exact rather than inferred.

## Proposed Solution

Re-run targeted searches over the residual production packages and inspect each remaining surface that can emit or transform media/base64 content. Build a path-by-path classification for Device VmControl, VMuse MCP tools, VMuse HTTP endpoints, file binary APIs, and any small missed imports. Decide whether each path is active shell/runtime, current-round display/provider transport, standalone MCP/plugin, internal encoding, legacy compatibility, safe-to-retire, or remediation candidate.

## Acceptance Criteria

- Every residual production media/base64 hit is classified with a file pointer and rationale.
- `novaic-device/device/vmcontrol_routes.py` screenshot route has a clear status.
- `novaic-mcp-vmuse` media and file byte surfaces are classified by module/function, not only by package.
- Exact remediation candidates are listed for the parent cleanup phase.

## Verification Plan

Use `rg` targeted at `novaic-device/device` and `novaic-mcp-vmuse/src` excluding tests/docs, inspect code slices with `sed`, and confirm whether routes are mounted or referenced from active services. No code changes should be made in this classification ticket.

## Risks

- Standalone MCP/plugin code may be intentionally different from shell contract; classification must not conflate valid MCP image content with shell text leakage.
- A route may be dormant but still mounted; status must distinguish active product usage from mere code presence.

## Assumptions

- This ticket only classifies and identifies remediation candidates; actual remediation belongs to the parent cleanup phase or a later follow-up.
