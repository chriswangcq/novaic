# VMuse desktop screenshot MCP surface classification result

## Summary

VMuse desktop screenshot and mouse aim media-byte paths are standalone MCP/VMuse tool transport, not the current Cortex shell/devicectl terminal-output path. They intentionally use base64 internally (`tools/desktop.py`) and convert to FastMCP `ImageContent` in `main.py`. Because VMuse appears packaged/deployed as a VM-side service and proxied by Device, this is not dead code, but it is a separate protocol surface that should stay isolated from shell history/context projection.

## Done

- Inspected `novaic-mcp-vmuse/src/novaic_mcp_vmuse/main.py` desktop `screenshot()` wrapper:
  - Calls `DesktopTools.screenshot(...)`.
  - Decodes `result["screenshot"]` with `base64.b64decode`.
  - Wraps image bytes using `Image(data=image_bytes, format="png").to_image_content()` and returns `[image_content, info_text]`.
- Inspected `main.py` `mouse(action="aim")` wrapper:
  - Decodes the aim screenshot with `base64.b64decode`.
  - Wraps it as FastMCP image content and returns an image plus textual hint.
- Inspected `novaic-mcp-vmuse/src/novaic_mcp_vmuse/tools/desktop.py` screenshot construction:
  - Plain fullscreen path returns `"screenshot": base64.b64encode(screenshot_bytes).decode('utf-8')`.
  - Grid/region/zoom path returns the same base64 `screenshot` field.
  - Aim action calls `DesktopTools.screenshot(...)` and forwards the base64 screenshot in the aim result.
- Searched VMuse usage and found Device/App infrastructure around VMuse deployment/proxying, but no evidence that this FastMCP image payload is the active Cortex shell output path.

## Known Gaps

- No code was changed. If product policy is “all real-time file/media services must go through LogicalFS/Blob artifact manifests,” VMuse desktop screenshot transport would need a separate migration plan. That is broader than this classification ticket.
- The current classification depends on repository references; runtime deployments could still expose VMuse directly through Device proxy.

## Artifacts

- Classification: standalone MCP/VMuse image transport; not an active Cortex shell/history contract violation; not dead code; keep isolated or plan a separate VMuse media contract migration.
