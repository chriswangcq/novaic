# VMuse HTTP and file binary API classification succeeded

## Summary

`R719` satisfies the classification problem. It inspected screenshot endpoints, binary file APIs, and the active proxy chain, then classified VMuse HTTP/file APIs as active lower-level raw-byte protocol surfaces that are normalized by Cortex shell wrappers before LLM-facing history/context.

## Evidence

- `http_server.py` registers and implements `/api/desktop/screenshot`, `/api/browser/screenshot`, `/api/file/pull`, and `/api/file/push`.
- `tools/files.py` encodes binary reads as base64 and decodes binary writes from base64.
- `agent_vm_proxy.py`, Tauri `vmuse.rs`, and `pc_client.py` show this lower-level VMuse tool proxy is active/product-adjacent.
- `shell_capabilities.py` maps HD shell commands to proxy operations and wraps screenshot/file-pull outputs into Blob artifact manifests before shell output.

## Criteria Map

- HTTP screenshot endpoints classified: satisfied by `R719`.
- Binary file pull/push paths classified: satisfied by `R719`.
- Active shell/runtime violations separated from standalone/lower-level protocol: satisfied; lower-level active raw-byte protocol, not LLM-facing shell/history leak.
- Cleanup/remediation candidates listed: satisfied; future Blob/LogicalFS migration candidate if architecture expands below shell.

## Execution Map

- `T729` executed targeted scan/inspection and recorded `R719`.

## Stress Test

- Plausible failure mode: calling VMuse HTTP base64 “safe” because it is below shell. `R719` avoids that by classifying it as active lower-level raw-byte protocol and only safe with respect to LLM context because shell wrappers normalize it.

## Residual Risk

- If lower-level VMuse APIs are accessed directly by new code, they can bypass the shell artifact contract. This is a future boundary risk, but the problem only required classification.
