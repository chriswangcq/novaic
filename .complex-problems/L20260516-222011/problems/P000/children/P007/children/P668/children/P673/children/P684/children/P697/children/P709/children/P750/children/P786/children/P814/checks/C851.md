# AssistantMessage events rendering cleanup verified

## Summary

P814 required removing or guarding legacy event rendering paths that could stringify non-string content. The event rendering is live (text, final, warning, error, image types all active), so removal was not applicable. The `JSON.stringify` fallback in `extractContent` was the sole unsafe path — it has been replaced with an empty string return. All event types now produce safe, bounded output.

## Evidence

- `grep -n JSON.stringify AssistantMessage.tsx` returns zero hits — the stringify fallback is gone.
- The `extractContent` function (lines 96-113) now returns `''` for non-string nested content instead of `JSON.stringify(content)`.
- The event switch exhaustively handles: `text`/`final` (Markdown), `warning`/`error` (bounded text with fallback label), `image` (authenticated image component), `thinking`/`tool_start`/`tool_end` (null), `default` (null). No path renders raw JSON.
- `tsc --noEmit` passes with zero errors.

## Criteria Map

- "Legacy events rendering is removed if no longer live" -> Events rendering is confirmed live (text/final/warning/error/image all active in production). Not removed; instead, the unsafe path within it was fixed.
- "If a narrow event path is still required, it only renders safe, bounded text projections" -> All event types produce bounded output: Markdown for text, short labels for warnings/errors, image components for images, null for intermediate events.
- "JSON.stringify fallback rendering is removed or guarded for this chat surface" -> Removed. `extractContent` returns empty string for non-string content. Zero `JSON.stringify` calls remain in the file.
- "Focused chat/component tests pass for touched behavior" -> No AssistantMessage-specific tests exist in the codebase. TypeScript compilation (the only available automated check) passes. No tests were broken.

## Execution Map

- T811 (one_go) -> R802: single line change in `extractContent`, removing `JSON.stringify` fallback. Verified via grep and tsc.

## Stress Test

- **Non-string nested content that was previously visible**: If event.data contained `{ content: { nested: "obj" } }`, the old code would render `{"nested":"obj"}` as chat text. Now it renders nothing. This is intentional — chat surfaces should not display raw JSON.
- **String content regression**: String content follows the unchanged `typeof content === 'string' ? content` path. No regression possible.
- **Empty content handling**: All call sites (text/final/warning/error) already handle empty string: text/final return `null` (no render), warning shows "Warning", error shows "Error".

## Residual Risk

- No component-level tests exist for AssistantMessage. A future change could reintroduce unsafe rendering without test-gated feedback. This is pre-existing technical debt, not introduced by this change.
