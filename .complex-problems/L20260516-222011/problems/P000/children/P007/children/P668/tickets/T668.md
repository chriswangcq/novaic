# Audit Runtime Process Topology and Deployment Scripts

## Problem Definition

Runtime service topology may have drifted after FSM, shell, LogicalFS, sandbox, and worker changes. We need inspect deployment/start scripts, worker/service entrypoints, scheduler/health processes, and related docs to ensure the roles are current and diagnosable.

## Proposed Solution

Search process/start/deploy surfaces, inventory current service and worker roles from source/config, compare them with docs/runbooks/scripts, and apply small corrections when stale names or misleading process descriptions are found. Keep changes low-risk and verify with relevant static guard or script checks.

## Acceptance Criteria

- Locate and inspect deployment/start scripts, service entrypoints, and process topology docs.
- Produce a current process-role summary based on code/config evidence.
- Patch any stale active doc/script/test claim discovered during the audit.
- Run relevant local checks or targeted command validation for changed files.

## Verification Plan

Use `rg --files`, `rg` for worker/service names, targeted reads of scripts/docs/config, and local static checks. If changes are made, verify with focused linters or syntax checks and record command outputs.

## Risks

- Some deployment topology may live outside this repository or in runtime environment config not present locally.
- Broad edits to deploy scripts can be risky; keep patches small and evidence-backed.
- Historical docs may intentionally mention retired topology and should be classified instead of blindly deleted.

## Assumptions

- Live deploy/restart is out of scope for this audit child.
- Current topology can be inferred sufficiently from repository scripts, worker entrypoints, and docs.
