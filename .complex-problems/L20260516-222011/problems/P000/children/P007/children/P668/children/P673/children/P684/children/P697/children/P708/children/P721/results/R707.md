# Device/devicectl surface discovery and contract map result

## Summary

Completed a bounded discovery pass for the Device/devicectl surface without code changes. The active shape is: `devicectl hd ...` is exposed through the Cortex shell capability layer, routed through Business/Device to CloudBridge/VmControl for Host Desktop and VM operations, and media-producing screenshot output is expected to become a `tool-output.v1` artifact manifest instead of raw base64 text.

## Done

- Inventoried device/devicectl-related files across `novaic-device`, `novaic-cortex`, `novaic-agent-runtime`, `novaic-mcp-vmuse`, `docs`, and startup scripts.
- Mapped Device Service as the hardware/Host Desktop/VM proxy and typed CloudBridge/VmControl command boundary.
- Mapped Cortex shell as the CLI-facing projection layer for `devicectl hd ...` commands.
- Identified artifact/display-related test coverage and docs that already expect screenshot output as Blob-backed artifact manifests.
- Separated discovery candidates from remediation so stale boundary wording can be fixed in the remediation ticket.

## Verification

- Ran file inventory over device/devicectl/runtime/cortex/docs surfaces with `find` plus `rg`.
- Ran targeted grep for `devicectl`, `Device Service`, `CloudBridge`, `VmControl`, `hardware`, `hd screenshot`, `screenshot`, `Host Desktop`, `HD`, `vm screenshot`, and `mobile screenshot`.
- Ran targeted grep for `runtime-artifact`, `tool-output.v1`, `blob://runtime-artifact`, and `display` across runtime, cortex, common, vmuse, device, docs, and scripts.
- Verified by inspection that `novaic-cortex/novaic_cortex/shell_capabilities.py` wraps `devicectl hd screenshot` into `tool-output.v1` manifest text and removes the raw screenshot field from stdout.
- Verified by inspection that `novaic-cortex/tests/test_shell_capabilities_blob_contract.py` asserts screenshot stdout does not contain raw `screenshot` data.

## Known Gaps

- `novaic-device/device/gateway_facing_api.py` has stale module wording saying Device APIs are exposed for Gateway to call; this conflicts with the current Gateway-edge and Business竊奪evice architecture wording.

- `novaic-mcp-vmuse` still has standalone MCP image-emitting paths. This may be legitimate for MCP/plugin use, but it must be classified separately from shell/devicectl output in the artifact/display discovery ticket.
