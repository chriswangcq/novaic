# Clean Up Queue Runtime Stale Entrypoint Residue

## Problem Definition

After queue/runtime role mapping, we need to search for stale duplicate entrypoints, retired worker names, misleading old comments, or launch surfaces that contradict the current worker/FSM architecture. Low-risk residues should be cleaned; uncertain residues must be recorded explicitly.

## Proposed Solution

Split into a stale-entrypoint scan and a remediation/verification child. Use the p689/p690 role maps as baseline, run targeted stale-term and reference scans, then patch only proven low-risk residues with focused tests.

## Acceptance Criteria

- Stale/duplicate queue/runtime entrypoint candidates are searched with evidence.
- Low-risk stale residues are removed or clarified; risky leftovers are recorded.
- Production code changes, if any, are minimal and verified.
- Focused tests, compile/import checks, or guard scans pass for changed files.

## Verification Plan

Use targeted `rg` scans over queue/runtime source, scripts, docs, and tests. For changes, run `python -m py_compile` or focused pytest/CI scripts depending on touched files. Save evidence artifacts.

## Risks

- Some old-looking text may be intentional docs/test coverage for legacy prevention.
- Removing entrypoint references without proving inactivity could break launchers.

## Assumptions

- Split is appropriate because scan evidence and remediation/verification have different success criteria.
