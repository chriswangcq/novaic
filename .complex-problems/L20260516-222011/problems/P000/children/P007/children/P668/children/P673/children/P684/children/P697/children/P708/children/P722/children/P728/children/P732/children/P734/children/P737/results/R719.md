# VMuse HTTP and file binary API classification result

## Summary

VMuse HTTP screenshot/file endpoints are an active lower-level Device/VmControl tool protocol surface, not the LLM-facing shell history contract. They still return/accept raw base64 in HTTP JSON for screenshot and binary file movement. Current Cortex shell wraps the HD proxy paths and converts screenshot/file-pull outputs to `tool-output.v1` Blob artifact manifests before they enter shell output. Therefore these APIs are not current LLM context leaks, but they are protocol-level raw-byte surfaces and should be candidates for a future VMuse/Device media contract migration if the architecture wants Blob/LogicalFS below shell too.

## Done

- Inspected `novaic-mcp-vmuse/src/novaic_mcp_vmuse/http_server.py` route registration:
  - `/api/desktop/screenshot` returns `DesktopTools.screenshot(...)` JSON directly.
  - `/api/browser/screenshot` returns `BrowserTools.screenshot(...)` JSON directly.
  - `/api/file/pull` and `/api/file/push` expose file transfer APIs.
- Inspected `http_server.py` handlers:
  - `desktop_screenshot` returns screenshot result containing base64 `screenshot` from `DesktopTools`.
  - `browser_screenshot` returns screenshot result containing base64 `screenshot` from `BrowserTools`.
  - `file_pull` documents and returns `binary=True` content as base64 via `FileTools.read_file`.
  - `file_push` expects base64 content when `binary=True` via `FileTools.write_file`.
- Inspected `novaic-mcp-vmuse/src/novaic_mcp_vmuse/tools/files.py`:
  - Binary read path returns `content: base64.b64encode(raw).decode(...)` and `is_base64: True`.
  - Binary write path decodes base64 before writing bytes.
  - Sudo helper scripts duplicate the same base64 read/write protocol.
- Inspected active proxy chain evidence:
  - `novaic-device/device/agent_vm_proxy.py` forwards `/api/vmuse/{agent_id}/{path}` through `proxy_tool_request`.
  - `novaic-app/src-tauri/vmcontrol/src/api/routes/vmuse.rs` documents and implements `POST /api/vmuse/{agent_id}/{tool}/{operation}` including `desktop/screenshot`.
  - `novaic-device/device/pc_client.py` states `tool_proxy` is used for `/api/vmuse`, `/api/android`, and `/api/hd` tool calls.
  - `novaic-cortex/novaic_cortex/shell_capabilities.py` maps HD shell capability `screenshot` to `desktop/screenshot` and wraps screenshot/file-pull results into Blob artifact manifests.

## Known Gaps

- No code was changed in this classification ticket.
- VMuse HTTP remains an active lower-level raw base64 protocol. This is acceptable only because shell capability wrappers normalize before LLM-facing history/context. If the target architecture requires all realtime media/file IO to be Blob/LogicalFS even below shell, this is a remediation/migration candidate.
