# Display current-round LLM image projection discovery result

## Summary

The current-round display image path is working by design: `display` fetches Blob bytes, public tool history redacts inline image data, durable payload stores `image_ref` plus display metadata, current display perception resolves the Blob ref to provider image content, and `FactoryLLMClient` preserves structured image message content in the request body.

## Done

- Identified `_exec_display` and `_display_content_for_llm` in `tool_handlers.py` as the Blob-fetch/display executor path.
- Identified `_display_public_output` and `_display_durable_payload` as public-history sanitization and durable `image_ref` storage paths.
- Identified `prepare_llm_call` as the injected assembly sequence: expand step refs, sanitize context, then process multimodal messages.
- Identified `step_result_client._resolve_display_image_refs` as the current display-only Blob image-ref resolver.
- Identified `context.process_multimodal_messages` and `multimodal.to_openai_content` / `to_anthropic_content` as provider-native image message conversion.
- Identified `FactoryLLMClient.chat` as preserving structured `messages` content when posting to Factory.

## Known Gaps

- Current-round provider-native image content is represented as base64 data URLs for OpenAI and base64 source blocks for Anthropic. That is the provider API transport form, not plain text history. Factory logs should treat it as structured JSON content, not text.
- If a future provider supports Blob URLs directly, this can be optimized, but current behavior matches the active request contract and tests.
