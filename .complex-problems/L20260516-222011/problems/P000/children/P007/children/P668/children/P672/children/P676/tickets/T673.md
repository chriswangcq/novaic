# Remediate Active Deployment Script Stale Roles

## Problem Definition

P678 classified several active/high-risk launcher surfaces, including local and packaged `start-backends.sh` variants that may still describe or launch an older backend topology. We need inspect and fix concrete low-risk stale script issues, then verify script syntax and relevant deployment guards.

## Proposed Solution

Use P678 remediation inputs to inspect active launcher scripts. Patch stale or misleading local/package start scripts where the current topology is clearly known from `scripts/start.sh` and runtime roster. Verify with shell syntax checks and relevant CI guard scripts. Avoid live deployment/restart.

## Acceptance Criteria

- P678 remediation inputs are inspected with file evidence.
- Stale role/process assumptions in active scripts are fixed or explicitly classified as not fixable locally.
- Changed shell/Python scripts pass syntax checks or relevant lint guards.
- Residual risks outside local repository control are recorded.

## Verification Plan

Run targeted `sed`/`rg` inspections, apply minimal patches, then run `bash -n` on changed shell scripts and relevant guards such as `python3 scripts/ci/check_start_config_contract.py`, `python3 scripts/ci/lint_deploy_fresh_smoke.py`, and `python3 scripts/ci/lint_runtime_worker_supervision.py` as applicable.

## Risks

- Packaged/generated files may require regeneration rather than direct edits.
- Local dev start script may intentionally differ from cloud topology, so fixes should improve clarity and prevent stale claims without over-expanding scope blindly.
- Some runtime behavior depends on app packaging not tested locally here.

## Assumptions

- Live deployment is out of scope.
- Low-risk script changes are acceptable if verified locally.
