# Classify Queue and Runtime Worker Roles

## Problem Definition

Queue service and agent runtime workers include task, saga, outbox, scheduler, health, session coordination, and runtime loop entrypoints. We need a current code-evidence map that distinguishes generic worker infrastructure from runtime/business computation, and catches stale or duplicate entrypoint code when safe.

## Proposed Solution

Use discovery artifacts plus focused reads of queue-service/runtime source and launch scripts to classify roles. Split into bounded children for queue-service worker roles, agent-runtime loop/CLI roles, and stale/duplicate remediation if needed.

## Acceptance Criteria

- Queue/runtime entrypoint files and launch commands are identified with file evidence.
- Worker roles are summarized clearly and mapped to source files.
- Stale or misleading queue/runtime entrypoint code is patched if low-risk, otherwise recorded.
- Relevant import/syntax/static checks are run if code changes occur.

## Verification Plan

Use `rg`, `sed`, and existing scan artifacts to inspect queue/runtime entrypoint modules and launch commands. If edits are needed, run focused Python compile/import or existing tests/guards.

## Risks

- Queue worker names may exist both as generic FSM infrastructure and specific runtime computation.
- Some worker functions may be invoked indirectly by launch scripts or supervisor config.
- Over-cleaning old entrypoints without checking references could break local or packaged startup.

## Assumptions

- This should split because queue-service roles, runtime loop roles, and possible cleanup are separate verification units.
