# Build Entrypoint Discovery Scan Artifacts

## Problem Definition

We need reproducible repository evidence for backend worker/service entrypoints and launch commands before making topology claims or editing stale entrypoint code. The scan must cover scripts, Python modules, package metadata, app resources, service configs, and deployment/worker launch references.

## Proposed Solution

Create scan artifacts in `.complex-problems/L20260516-222011/tmp/` using shell-native pointer-oriented scans. Split the scan into smaller evidence-producing subproblems for script/package metadata, Python module entrypoints, and service/config launch references if the surface is broad.

## Acceptance Criteria

- Candidate entrypoint files and launch commands are captured from source evidence.
- Scan commands are saved so results are reproducible.
- Output is summarized into stable artifact files instead of loading huge raw scans into context.
- No production source files are modified by this discovery child.

## Verification Plan

Run the saved scan commands, count artifacts, inspect head/tail samples, and verify that known expected surfaces such as queue service, runtime workers, Blob, LogicalFS, Sandbox, Cortex, and app backend launchers appear in the candidate set.

## Risks

- Broad regex scans may be noisy and need classification later.
- Entrypoints may be encoded in Python metadata, shell scripts, app resources, or deployment config, so a single scan style is not enough.
- Missing generated/resource paths would bias later classification.

## Assumptions

- This child should remain discovery-only; classification and cleanup belong to sibling problems.
