# Artifact/display/context projection discovery ticket

## Problem Definition

We need an evidence-backed map of how screenshot and artifact outputs move from shell/devicectl output through Blob/artifact manifests, display, LLM request construction, and history replay. The audit must specifically catch active paths that still embed large media bytes as text or fail to project images to the LLM in the current round.

## Proposed Solution

Perform a focused discovery pass over Runtime, Cortex, Blob/artifact, display handlers, shell output projection, and tests/docs. Classify active paths versus legacy/plugin surfaces. Record exact file pointers for shell output contract, artifact manifest persistence, display tool result construction, LLM request image projection, and history replay behavior. List remediation candidates for any active path that still violates the terminal-text-plus-artifact-manifest contract.

## Acceptance Criteria

- Shell/tool output contract for screenshots/artifacts is identified with code/test/doc evidence.
- Blob/artifact URI and manifest-only history behavior are identified with code/test evidence.
- Display tool output and LLM image projection path are identified with code/test evidence.
- Active paths that embed large media bytes as text are listed as cleanup candidates.
- Legacy or standalone surfaces are explicitly classified instead of mixed into active shell/runtime behavior.

## Verification Plan

Run targeted `rg`/file inspections for `tool-output.v1`, `runtime-artifact`, `display`, `image_url`, `base64`, `screenshot`, `_mcp_content`, `tool_call_id`, `history`, and LLM request construction across `novaic-agent-runtime`, `novaic-cortex`, `novaic-common`, `novaic-mcp-vmuse`, docs, and relevant tests. Prefer exact line pointers and existing tests over broad claims.

## Risks

- Display projection may span several packages, so a one-shot grep could miss constructor/adaptor code.
- Standalone MCP/plugin image content may look like a violation but may not be part of the active shell/runtime path.
- Tests can prove intended behavior but not necessarily deployment wiring, so the result must separate implementation evidence from runtime deployment assumptions.

## Assumptions

- No code changes are made in this discovery ticket.
- If a defect is found, remediation is deferred to the later remediation child problem.
- Historical archived context with old base64 is not automatically an active-path bug unless current request assembly still replays it as text.
