# Discover and Classify Deployment Scripts

## Problem Definition

Before editing deployment/start scripts, we need a reliable inventory of candidate files and their status. Without this, remediation could accidentally edit historical docs, generated files, or test fixtures instead of active runtime surfaces.

## Proposed Solution

Run reproducible file and text searches for deployment/start/supervision/smoke/health scripts and configs. Save candidate lists and classify high-signal files into active, test-only, historical, or generated/ignored categories. Do not modify source code in this discovery ticket.

## Acceptance Criteria

- Candidate file list and keyword scan artifacts are saved under the ledger tmp directory.
- Active deployment/start/supervision candidates are identified with file evidence.
- Test-only/historical/generated candidates are separated from active scripts.
- No code/source changes are made by this discovery ticket.

## Verification Plan

Run read-only `rg --files`, targeted `rg`, `find`, and selected `sed`/`head` reads. Confirm `git status --short` after discovery only shows ledger/tmp changes attributable to evidence capture or pre-existing unrelated work.

## Risks

- Keyword searches can miss scripts with unusual names.
- Active deploy behavior may live outside this repository.
- Large repo scans may include ledger/history artifacts; exclude `.complex-problems` where appropriate.

## Assumptions

- Discovery can be completed safely in one bounded pass because it is read-only apart from ledger evidence artifacts.
- Any remediation belongs to sibling problem P676.
