# Split repo inventory into fact gathering and residue scan

## Problem Definition

Repo topology and residue inventory must establish a reliable factual baseline before deeper optimization. It should not mix workspace state discovery, submodule boundaries, and residue pattern interpretation into one opaque pass.

## Proposed Solution

Split this problem into smaller child problems: one for clean-state/topology/submodule inventory, and one for residue hotspot search and triage. These children can produce evidence without implementation edits unless they discover very small safe cleanup work.

## Acceptance Criteria

- Workspace topology and dirty state are captured separately from residue interpretation.
- Submodule boundaries and current branch/pointers are recorded.
- Residue pattern scans identify candidate areas for later child problems.
- No unrelated edits are made during fact gathering.

## Verification Plan

- Use git and file-search commands with bounded output.
- Record exact commands and evidence pointers in results.
- Check that no files changed unless an explicitly justified cleanup is made.

## Risks

- Broad searches can produce noise; triage must distinguish active current paths from stale docs or old ledgers.
- Submodule dirty state can hide accidental changes.

## Assumptions

- The current checkout is the intended audit target.
- Existing `.complex-problems` ledgers are part of audit metadata, not implementation code.
