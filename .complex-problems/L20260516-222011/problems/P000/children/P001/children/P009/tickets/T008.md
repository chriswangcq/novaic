# Split residue hotspot search by pattern class

## Problem Definition

Residue hotspot search spans multiple unrelated risk classes: old direct tool names, ephemeral path leaks, base64/media leakage, compatibility/fallback code, TODO/skipped tests, and stale docs. Searching all patterns in one result would mix active bugs with historical noise.

## Proposed Solution

Split into targeted child problems by residue class. Each child will run bounded `rg` searches, classify hits as active issue / benign historical reference / candidate for specialized audit, and implement only tiny high-confidence cleanup if found.

## Acceptance Criteria

- Residue patterns are searched with explicit bounded commands.
- Hits are triaged, not blindly counted.
- Active code issues are routed to specialized child problems or fixed if tiny and safe.
- Historical ledger/doc hits are identified separately from active implementation residue.

## Verification Plan

- Use `rg` with focused globs and exclusions for `.complex-problems` when appropriate.
- Record evidence pointers and next-step routing.
- Confirm any fix with targeted tests or static checks.

## Risks

- Previous ledger history contains many obsolete terms; searches must exclude or label historical ledgers.
- Docs may intentionally describe old failures; do not delete useful postmortem context without review.

## Assumptions

- Active code and current tests matter more than historical ledger artifacts for optimization decisions.
