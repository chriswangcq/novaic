# Scan and clean business dispatch active residue

## Problem Definition

`novaic-business` may contain active-path residue around dispatch/subscriber code: migration-era comments, fallback/compatibility wording, TODO/FIXME, or old direct-tool names. These can obscure the queue/FSM handoff boundary even if behavior is correct.

## Proposed Solution

Run focused scans over `novaic-business` active code and relevant tests, inspect all matches, classify non-issues versus real cleanup candidates, apply only safe comment/dead-code cleanup, and validate with focused pytest runs.

## Acceptance Criteria

- Residue scans cover active `novaic-business` source and relevant tests for legacy/fallback/compat/migration/TODO/FIXME and old direct-tool terms.
- All hits are classified with active-path risk.
- Safe misleading residue is removed or rewritten to current-state wording.
- Focused business tests pass for touched areas.
- Final targeted scan over touched files confirms no cleaned residue remains.

## Verification Plan

Use `rg` to inventory hits, `sed`/`nl` to inspect context, `apply_patch` for safe edits, then run targeted `pytest` in `novaic-business`. Finish with a post-cleanup `rg` scan and `git diff --stat`.

## Risks

- Some words like compatibility may be legitimate provider/API concepts rather than old dispatch paths.
- Business tests may require local import paths; resolve with `PYTHONPATH` rather than broad environment changes.

## Assumptions

- Current dispatch handoff should flow through queue/FSM contracts, not direct harness tool surfaces.
- Comment cleanup is acceptable when it improves the explicit dependency boundary without behavior changes.
