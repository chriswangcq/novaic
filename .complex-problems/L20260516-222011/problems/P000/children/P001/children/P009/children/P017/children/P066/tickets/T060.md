# Split active code residue audit by implementation boundary

## Problem Definition

`P066` must find stale fallback/compatibility residue in active implementation code without mixing unrelated service boundaries or over-editing risky live paths.

## Proposed Solution

Split the active code audit by implementation boundary: runtime/queue handling, Cortex/common projection and sandbox helpers, and app/business/MCP adapters. Each child should run focused scans, classify active hits, and apply only high-confidence cleanup or route larger findings to follow-up work.

## Acceptance Criteria

- Active implementation areas are scanned with bounded, reproducible commands.
- Hits are classified by service boundary and risk.
- High-confidence small cleanup is applied directly where safe.
- Any larger active compat branch is not hidden; it becomes explicit child/follow-up work.

## Verification Plan

- Run `rg` scans in each child problem for fallback/compat/legacy/migration/TODO/FIXME/pass/skip patterns.
- Inspect representative active hits before editing.
- Run focused tests for touched service areas.
- Close each child with explicit criteria map and residual-risk review.

## Risks

- One giant code scan can hide active-risk branches in a long result.
- Removing live compatibility behavior without service-specific tests could create regressions.

## Assumptions

- Build/cache/ledger paths are not active implementation code and should be excluded from scan conclusions.
