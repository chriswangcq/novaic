# Scan stale fallback compatibility and TODO residue

## Problem Definition

`P017` needs a bounded residue audit for stale fallback/compatibility branches, TODO/FIXME/pass/skip markers, and old migration comments that could preserve deprecated behavior after the newer shell/display/blob/FSM contracts were introduced.

## Proposed Solution

Search active code and active documentation with focused patterns, excluding `.complex-problems` ledger noise and clearly historical archived design records where appropriate. Classify hits into active risk, benign test fixture, intentional guard, historical note, or cleanup candidate. Apply only tiny high-confidence cleanup directly; create specialized child problems for any risky or larger active branch removal.

## Acceptance Criteria

- Searches are bounded and reproducible.
- Hits are classified by active path status and risk.
- Tiny high-confidence cleanup is applied directly if found.
- Larger/riskier issues are routed to child problems rather than buried in a summary.
- Final result states whether active old-compat code remains.

## Verification Plan

- Run focused `rg` scans for fallback, compat, TODO/FIXME, pass, skip, migration, legacy, old path, and similar terms.
- Review representative active hits before deciding whether to edit or split.
- Run focused tests only if code changes are made.
- Record exact result and a strict success check.

## Risks

- Broad scans can produce historical noise and tempt accidental churn.
- Removing a compatibility branch without proving it is inactive could break a live path.

## Assumptions

- The user's current preference is no backward-compatibility residue unless explicitly justified by a current active contract.
- Historical roadmap material may be preserved if not active implementation guidance.
