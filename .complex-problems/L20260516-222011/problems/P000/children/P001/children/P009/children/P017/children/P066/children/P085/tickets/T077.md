# Classify and clean non-monitor App active residue

## Problem Definition

`P085` must finish the active App residue audit outside ActivityTimeline/monitor code. The parent scan found fallback/legacy/base64/compatibility hits in settings, device polling, WebRTC, chat audio, types/tests, and UI helper code that were not classified by the earlier App monitor ticket.

## Proposed Solution

Run a bounded `novaic-app/src` scan, inspect each non-monitor hit, classify it as current behavior/test guard/benign adapter/stale residue, and apply only safe cleanup for stale comments or misleading names. Avoid broad UI behavior changes unless a hit is clearly harmful.

## Acceptance Criteria

- `novaic-app/src` is scanned for fallback, compat, legacy, migration/migrate, TODO/FIXME, base64/data URL, and direct-tool residue.
- Each remaining non-monitor hit is classified with file-level evidence.
- Stale comments or names are cleaned when safe.
- Focused App tests/lint are run for touched areas, or explicit no-code-change verification is recorded.

## Verification Plan

- Run targeted `rg` scans before and after cleanup.
- Inspect representative source around every active hit.
- Run focused `npm run test:unit` tests for touched App areas and any existing guard tests relevant to blob/base64/compatibility.

## Risks

- Some frontend fallback/base64 strings are legitimate browser/UI mechanics, such as polling defaults, WebRTC cursor bytes, tiny silent audio placeholders, or guard tests. These should be classified, not deleted blindly.
