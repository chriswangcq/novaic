# Split factual inventory by state and entrypoints

## Problem Definition

The workspace inventory should be factual, bounded, and reviewable. It includes both git/submodule state and repo layout/test/deploy entry points, which are related but independent evidence surfaces.

## Proposed Solution

Split into child problems: one for git branch/submodule/dirty-state inventory, and one for top-level repo layout plus test/deploy entrypoint discovery. Each child will run read-only commands and record evidence.

## Acceptance Criteria

- Git/submodule state and layout/entrypoints are not conflated.
- Commands are read-only.
- Evidence is bounded and stable.
- No implementation edits occur in this factual inventory step.

## Verification Plan

- Confirm `git status --short` before and after fact gathering.
- Use `git submodule status`, bounded `find`, `rg --files`, and selected package/project files.
- Record exact output summaries in result bodies.

## Risks

- Large mono-repo output can flood context; use head/sed and targeted lists.
- Submodules may be on expected detached/pinned states; avoid treating all non-main pointers as bugs.

## Assumptions

- Read-only shell commands are safe for this step.
