# Remove Cortex tests top-level package ambiguity

## Problem Definition

Cortex tests currently rely on generic `tests.*` imports while multiple subprojects expose a top-level `tests` package. When runtime and Cortex tests are selected together, Python may resolve `tests` to the wrong subproject and fail before executing the intended test module.

## Proposed Solution

Move Cortex shared test helpers into a package-specific helper namespace, update Cortex tests to import that namespace, and remove the stale `novaic-cortex/tests/__init__.py` package marker so Cortex test modules are collected as standalone tests rather than as `tests.*` modules.

## Acceptance Criteria

- Cortex helper imports no longer use `from tests...`.
- `novaic-cortex/tests` no longer declares itself as a top-level Python package unless a concrete remaining need is proven.
- The combined focused command for runtime tool surface policy and Cortex schema limits passes in one pytest process.
- Package-local Cortex tests that depend on moved helpers still pass.

## Verification Plan

- Search Cortex tests for `from tests` and `import tests` residue.
- Run the previously failing combined focused command.
- Run a focused Cortex helper-dependent test slice to ensure moved helper imports work.

## Risks

- Many Cortex tests may import shared helpers; update breadth must be mechanical but carefully verified.
- Removing `tests/__init__.py` changes collection names; any hidden code that relies on package-style test module names may need cleanup rather than compatibility fallback.

## Assumptions

- Test helper code is not production API and can be moved to a package-specific helper namespace.
- It is acceptable to remove old test package markers rather than preserve backwards-compatible `tests.*` imports.
