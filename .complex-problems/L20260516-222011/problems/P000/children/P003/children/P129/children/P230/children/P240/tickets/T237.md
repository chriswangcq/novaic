# Audit and harden shell capability payload guidance

## Problem Definition

Most former interface tools now live behind shell capabilities, so shell help and CLI guidance are the practical contract the model sees. The guidance must keep terminal output bounded and route full inspection through explicit payload/artifact commands instead of raw large text or base64.

## Proposed Solution

Audit `shell_capabilities.py`, runtime shell output contracts, and related schema/help tests. Correct stale wording if found, and verify the shell/payload output contract with focused tests.

## Acceptance Criteria

- Shell capability guidance and payload-help paths are mapped with concrete file/function pointers.
- Guidance points agents to explicit `cortex payload read/search/summarize/qa` commands for full payload inspection.
- Guidance does not encourage pasting raw large outputs/base64 into replies or normal context.
- Focused shell capability and output contract tests pass after any corrections.

## Verification Plan

- Inspect shell capability source and tests with `rg`/`sed` pointers.
- Run focused tests around shell output contract, tool schemas, and payload inspection API.
- Record any residual wording gap as follow-up instead of calling the ticket successful.

## Risks

- Shell guidance spans Cortex schema docs and runtime shell normalization; checking only one layer could miss stale wording.

## Assumptions

- Existing tests should encode most of the intended contract, but source text still needs manual audit.
