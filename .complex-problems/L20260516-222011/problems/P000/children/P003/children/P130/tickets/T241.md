# Audit provider-native media boundary

## Problem Definition

Image bytes are allowed at the final provider API boundary, but must not be confused with Cortex/tool history text. The code must clearly separate history projection from provider-native multimodal payload construction.

## Proposed Solution

Audit runtime multimodal context assembly, display projection, and LLM factory provider adapters. Verify provider request tests prove image content is passed as provider-native media structures and not plain text tool history. Correct comments/tests if ambiguity remains.

## Acceptance Criteria

- Provider-native media construction is mapped with file/function pointers.
- Cortex history/tool projection paths are mapped separately and do not inject image bytes as text.
- Provider tests prove OpenAI/Anthropic/Gemini-style media payloads use native content parts where applicable.
- Ambiguous comments/tests are corrected or explicitly classified as provider-boundary-only.

## Verification Plan

- Inspect runtime context/multimodal conversion and LLM factory provider code.
- Run focused runtime multimodal tests and LLM factory provider media tests.
- Run residue scans for base64/image data appearing as tool history text.

## Risks

- Provider adapters differ by vendor; a fix for one can leave another path ambiguous.

## Assumptions

- Provider boundary tests can be focused and do not require live external provider calls.
