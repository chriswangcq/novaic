# Ticket: Classify or fix media/base64 residue

## Problem Definition
The codebase recently moved toward a shell text plus artifact manifest contract, with provider-native image transport handled only at the LLM factory boundary. Residual `base64`, `data:image`, `inlineData`, and `image_url` occurrences may be legitimate provider/test references or stale paths that still flatten media into text. They must be classified or fixed.

## Proposed Solution
Run a repository-wide targeted residue scan across runtime, cortex, common, and LLM factory code. Build a classification table for each finding cluster. Patch any ambiguous active path so media is either artifact-manifest text in shell output or provider-native media at the LLM boundary, never accidental base64 text.

## Acceptance Criteria
- Every relevant residue cluster is classified as provider-boundary, test fixture, help text, shell manifest, or fixed stale logic.
- No active runtime context path is found flattening screenshot/image base64 into tool text sent to the LLM.
- No shell-facing CLI emits raw image base64 in normal stdout.
- Any code change has focused tests or a documented no-test reason.

## Verification Plan
- Use `rg` scans for `data:image`, `/9j/`, `base64`, `inlineData`, `image_url`, and `_mcp_content` image payload patterns.
- Run focused existing tests around shell output contracts, display chat history, factory multimodal conversion, and Cortex blob boundaries.
- If patches are needed, rerun the directly affected test modules.

## Risk
False confidence if residue is only sampled; classification must be cluster-based and cite file paths.
