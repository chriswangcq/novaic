## Topology Map

Queue Service process entrypoint:

- `novaic-agent-runtime/queue_service/main.py` builds the FastAPI app, opens `queue.db`, constructs `TaskQueue`, `SagaRepository`, `SagaOrchestrator`, `SessionLedgerRepository`, `SessionOutboxDispatcher`, and `SessionRepository`, then registers task/saga/recovery/session routes.
- `main.py` uses `QueueServiceDependencies.system()` as the dependency boundary for clock and ID providers. This is the correct direction for deterministic repository behavior.
- Startup calls `session_repo.rebuild()`, so restart recovery still lives in the queue service process rather than in an external worker.

HTTP/API surface:

- `novaic-agent-runtime/queue_service/routes.py` owns Task Queue, Saga, recovery, and session API route factories.
- Task publish/claim/complete/fail routes still expose the durable queue primitives to workers.
- Saga creation goes through policy checks such as `reject_session_owned_saga_type`, which helps prevent non-session paths from bypassing the session harness for wake sagas.

Generic FSM substrate:

- `novaic-agent-runtime/queue_service/fsm/core.py` is business-agnostic and IO-free: `FsmStateSnapshot`, `FsmEvent`, `FsmEffect`, `FsmDecision`, and `decide_transition` are pure reducer primitives.
- `novaic-agent-runtime/queue_service/fsm/runner.py` owns the mechanical durable write order: append event, materialize state, append outbox effects. It intentionally does not generate time/IDs or make business decisions.
- This is a good explicit dependency boundary: business reducers compute decisions; repositories/runners provide persistence mechanics; dependency providers inject time and IDs.

Session DSL/business decision layer:

- `novaic-agent-runtime/queue_service/session_fsm.py` is the session-specific FSM DSL/reducer layer.
- It models `no_active`, `starting`, `active`, `ending`, `suspected_dead`, and `recovering` states.
- Dispatch actions are `start_wake`, `attach_input`, and `buffer_pending`.
- Finalize actions are `accept_finalize` and `reject_finalize`.
- Effects are explicit specs such as `create_wake`, `attach_input`, `buffer_input`, and `finalize_session`.

Session repository/coordinator:

- `novaic-agent-runtime/queue_service/session_repo.py` is the main queue session coordinator. It is not just a thin adapter: it appends input events, reads session state, decides dispatch/finalize/recovery behavior, records session transitions, builds wake/restart plans, and publishes or defers outbox effects.
- It now receives `clock`, `scope_id_provider`, `session_ledger`, and `outbox_dispatcher` through the constructor, so the largest hidden time/ID boundary has already been made explicit.
- It still contains orchestration complexity around suspected-dead recovery, live attach, occupied-state buffering, pending projections, and session rebuild. This is the main area to audit next for remaining imperative/compat residue.

Durable outbox roles:

- `novaic-agent-runtime/queue_service/session_outbox.py` dispatches durable session side effects. Supported effects include wake saga creation, recovery archive scope, and attach-input publish.
- Wake saga creation still eventually calls `saga_orchestrator.create()` inside the session outbox dispatcher, but the durable outbox row is the authority and idempotency key carrier.
- `novaic-agent-runtime/queue_service/session_outbox_worker.py` is a business-specific handler plugged into the generic worker substrate; it drains session outbox rows through `SessionOutboxDispatcher`.
- `novaic-agent-runtime/queue_service/saga_outbox_worker.py` similarly drains saga outbox rows through `SagaOrchestrator.drain_pending_effects()`.

Generic worker substrate:

- `novaic-agent-runtime/queue_service/worker/generic_worker.py` is business-agnostic employee mechanics: poll, handle, report, metrics, shutdown, sleep policy, and reporter error behavior.
- Business-specific code is provided through `JobSource`, `JobHandler`, and `JobReporter` ports rather than being hardwired into the loop.

Runtime worker/process roster:

- `novaic-agent-runtime/task_queue/workers/runtime_roster.py` is the canonical process roster and worker-mode SSOT.
- Current modes: `task-worker`, `saga-worker`, `session-outbox-worker`, `saga-outbox-worker`, `health`, and `scheduler`.
- Runtime processes also include the subscriber role, so not every supervised process is an FSM worker, but worker command modes are centralized.

## Optimization Candidates Found

- Session coordination remains the densest business/orchestration unit. The next tickets should verify whether `session_repo.py` still contains legacy imperative branches that should move behind narrower FSM decisions or explicit effect handlers.
- Wake saga creation is durable-outbox mediated, but the actual side effect still occurs in `SessionOutboxDispatcher` via `saga_orchestrator.create()`. That may be acceptable if the outbox row is the durable command, but it needs focused audit against the user's desired "outbox owns side effects" model.
- The roster is centralized, but a focused check should confirm deploy/start/health paths all consume the same roster and there is no stale duplicate worker list.
