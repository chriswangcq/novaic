## Classification

Shell CLI / device artifacts:

- `devicectl hd screenshot` and `devicectl hd file-pull` decode device base64 at the device boundary, upload bytes to Blob, and return `tool-output.v1` artifact manifests.
- Shell capability help explicitly says media/file outputs are `blob://` artifacts and raw base64 should not be printed/pasted.
- Tests assert raw screenshot/file base64 and original device keys do not appear in shell stdout.

Cortex step-result/payload projection:

- `step_result_projection` parses `tool-output.v1` as text plus artifacts; history/current projections are text-only, while `display_perception` is the explicit visual path.
- Unknown dict fallback is bounded diagnostic text only.
- `/v1/payload/read|search|summarize|qa` remain explicit bounded payload inspection endpoints and return text slices/summaries, not automatic large context injection.

Runtime display perception:

- `display` may inline small image bytes into `_mcp_content` for the shell-outside visual perception path.
- Runtime multimodal utilities convert only current-turn `display_perception` messages into model image input; tests assert historical replay is placeholder/text and does not preserve raw image data.
- This is intentional multimodal model input, not shell terminal output or historical ordinary text context.

Tests/fixtures:

- Remaining base64/data-image hits in tests are fixture values and assertions for the intended behavior: shell manifests stay text, current display can become model image input, history replay stays sanitized.

## Residual Risk

The display boundary intentionally remains capable of sending image bytes to the LLM for current-turn perception. That boundary must stay small and explicit; regressions are covered by no-historical-tool-image-injection, shell output contract, and tool output projection tests.
