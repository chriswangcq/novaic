# Active session read path audit result

## Summary

Active session read/list paths derive from `SessionLedgerRepository.get_state/list_active_states`, which is backed by `tq_session_state` through the generic FSM store. No legacy `tq_active_sessions` or independent active pointer read path was found in production queue/task code.

## Done

- Searched active-session read APIs and legacy table names in production `queue_service` and `task_queue`.
- Inspected `SessionRepository.get_active_session`, `list_active_session_states`, `_active_session_state_in_current_transaction`, and `_active_session_from_session_state`.
- Inspected `SessionLedgerRepository.list_active_states` and `/sessions` diagnostic route caller.

## Verification

- `get_active_session()` calls `_active_session_state_in_current_transaction()`, which calls `session_ledger.get_state(session_key)`.
- `list_active_session_states()` calls `session_ledger.list_active_states()`, which filters `state = active` with non-null active saga/scope columns.
- `/sessions` route is diagnostic and delegates to `session_repo.list_active_session_states()`.
- `tq_active_sessions` had no production hits in `queue_service`/`task_queue`.

## Known Gaps

- None for active session read paths.
