# Ticket: Final Cortex compatibility guard matrix

## Problem Definition

After the Cortex lifecycle, archive/diagnostic, API, CLI, and runtime bridge cleanup children have closed, perform a final Cortex compatibility verification pass. The goal is to prove that no live Cortex path still accepts missing/stale generation, revives old active-state lookup, or leaks large/binary tool payloads into LLM context through compatibility-shaped surfaces.

## Proposed Solution

- Run narrow and widened guards over Cortex/runtime code for compatibility phrases and old-state lookup patterns.
- Run guard scans for missing/stale generation handling, active-state lookup, context projection bypass, payload/media projection, and shell artifact contracts.
- Classify every remaining hit as current path, test fixture, docs/comment, operational endpoint, or unresolved risk.
- Rerun focused Cortex/runtime tests covering changed lifecycle, context event, payload, shell/tool-output, and bridge boundaries.
- Spawn follow-up subproblems for any dangerous or unclassified hit.

## Acceptance Criteria

- Guard artifacts are saved and every remaining hit category is classified.
- No live Cortex path accepts missing/stale generation where generation is required.
- No live Cortex path revives old active-state lookup as authoritative state.
- No live shell/tool-result path exposes raw base64/large payloads as ordinary context text.
- Focused Cortex/runtime boundary tests pass.
- Any unresolved dangerous hit is split instead of being waved through.

## Verification Plan

- Use `rg`/source slices for compatibility/generation/active lookup/media guard scans.
- Run focused pytest suites from the closed child problems plus any final guard tests needed.
- Record a matrix with hit classification, evidence paths, test commands, and residual risk.

## Risks

- Some compatibility words may remain in historical tests, docs, or ledger artifacts. Exclude `.complex-problems` from source scans and classify source/test hits carefully.
- A final verification pass can be broad; split if scan or test failures reveal real implementation work.
