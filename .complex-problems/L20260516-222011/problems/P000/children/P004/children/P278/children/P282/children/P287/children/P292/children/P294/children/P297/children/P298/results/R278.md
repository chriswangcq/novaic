# Result: Dispatch start-wake helper design

## Summary

Designed the smallest safe extraction for duplicated dispatch start-wake construction in `SessionRepository.dispatch()`.

## Current Duplicate Shape

Ordinary no-active branch (`session_repo.py:287-356`):

- Runs inside the initial global transaction.
- Calls `_decide_live_dispatch(active_session=None, ...)` and requires `START_WAKE`.
- Allocates `scope_id` and computes `session_generation` inside the transaction.
- Reads unconsumed input event ids inside the transaction.
- Builds `build_dispatch_wake_plan(...)` with no recovery marker/projection.
- Records `DISPATCH_WAKE_START_QUEUED` via `session_ledger.record_transition(...)`.
- Returns result immediately.

Recovery start branch (`session_repo.py:381-459`):

- Runs after initial transaction because suspected-dead observation is first recorded durably.
- Repeats `START_WAKE` decision check.
- Allocates `scope_id`, computes next generation through a small transaction helper, and reads unconsumed input ids through a small helper.
- Builds `build_dispatch_wake_plan(...)` with optional recovery marker/projection.
- Optionally appends recovery archive effect.
- Records `DISPATCH_WAKE_START_QUEUED` through `_record_session_transition_after_transaction(... require_outbox=True)`.
- Returns result.

## Common Logic

The shared logic is:

- Validate start decision for `active_session=None`.
- Allocate scope id and generation.
- Load unconsumed input event IDs.
- Build `DispatchWakePlan`.
- Construct result `{"action": "wake_start_queued", "scope_id": scope_id, "delivery": "outbox_pending"}`.
- Record `DISPATCH_WAKE_START_QUEUED` with decision trace, active scope id, durable outbox effect(s), and `require_outbox=True`.
- Attach `outbox_id` to result when present.

## Path-Specific Differences

- Transaction context:
  - ordinary path can run in the already-open transaction;
  - recovery path must run after the suspected-dead observation transaction and therefore uses `_record_session_transition_after_transaction`.
- Recovery extra effects:
  - recovery path may add `RECOVERY_ARCHIVE_SCOPE` effect;
  - ordinary path has no recovery archive effect.
- Recovery metadata:
  - recovery path passes `recovery_marker` and `recovery_pending_projection` to `build_dispatch_wake_plan`.

## Chosen Helper Design

Add a private helper such as `_queue_start_wake_transition(...)` on `SessionRepository` with parameters:

- identity/input: `session_key`, `agent_id`, `subagent_id`, `user_id`, `trigger_type`, `metadata`, `idempotency_key`, `created_at`.
- allocation values: `scope_id`, `session_generation`, `input_event_ids`.
- recovery values: `recovery_marker=None`, `recovery_pending_projection=None`.
- recording mode: `record_in_current_transaction: bool`.

The helper should:

- perform the `START_WAKE` decision check once;
- build the dispatch wake plan once;
- build optional recovery archive effect once;
- call either `_record_session_transition_in_current_transaction` or `_record_session_transition_after_transaction` based on `record_in_current_transaction`;
- require outbox for both modes;
- return the same result dictionary.

This keeps only allocation/timing differences at call sites, while centralizing the transition shape and result shape.

## Implementation Guidance

- Ordinary no-active branch should reduce to computing allocation values inside the existing transaction and calling the helper with `record_in_current_transaction=True`.
- Recovery branch should compute allocation values using after-transaction helpers and call the helper with `record_in_current_transaction=False`.
- Remove the duplicate inline construction blocks.
- Keep recovery archive effect logic inside the helper so future changes cannot update ordinary/recovery effect recording differently.
