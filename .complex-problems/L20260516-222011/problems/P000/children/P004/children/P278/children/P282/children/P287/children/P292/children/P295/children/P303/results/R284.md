# Result: Finalize transaction flow audit found a race gap

Mapped finalize path:
- `task_queue/sagas/wake_finalize.py` builds `session_ended` payload with `finalize_reason`, `generation`, and `remaining_stack`.
- `task_queue/handlers/session_handlers.py` validates those fields and calls `saga_client.session_ended(...)`.
- `queue_service/routes.py` exposes the same required contract on `/session-ended`.
- `queue_service/session_repo.py::session_ended(...)` calls pure `decide_session_finalize(...)`, records finalize accepted/rejected through `SessionLedgerRepository`, and derives pending input projection.

Clean evidence:
- The public finalize contract requires `finalize_reason`, `generation`, and `remaining_stack` at route/handler/saga boundaries.
- Rejected finalize records an explicit rejection event and leaves active state intact.
- Accepted finalize records a `session_finalized` event with metadata including reason, generation, remaining stack, and ended scope.

Gap found:
- In `session_ended`, Step 1 runs in a global transaction and `record_session_finalized(...)` clears state to `no_active`.
- If pending input exists, the method exits that transaction, then creates restart scope/generation and records `SESSION_RESTART_PENDING` in a second transaction through `_record_session_transition_after_transaction(...)`.
- Between those two transactions, a concurrent dispatch can observe `no_active` and start a separate wake, racing the intended restart from pending inbox.
- This violates the desired finalize/restart atomicity: active should not be externally observable as `no_active` when the same finalize operation has already decided to restart pending input.

Recommended follow-up:
- Make finalize-with-pending record restart intent/state inside the same transaction that accepts finalize, or leave state in `ending` until restart transition is durably recorded.
- The follow-up must preserve durable outbox behavior and generation checks.
