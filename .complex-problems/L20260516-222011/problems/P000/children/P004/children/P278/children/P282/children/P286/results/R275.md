# Result: Session schema table inventory

## Summary

Aggregated child results R273 and R274. Current Queue Service session schema and active code usage are clean around the intended SSOT model: `tq_session_state` is the authoritative materialized session state, `tq_session_events` is the append-only event/inbox log, and `tq_session_outbox` is the durable side-effect ledger. `tq_active_sessions` is absent from fresh DDL and has no production runtime references in the scanned code paths.

## Child Results Used

- R273: Session DDL definition scan.
- R274: Session table active usage classification.

## Findings

- `novaic-agent-runtime/queue_service/db/schema.py` defines only three session tables: `tq_session_events`, `tq_session_state`, and `tq_session_outbox`.
- `SessionLedgerRepository` configures the generic FSM store to use those three tables and is the session-specific adapter for event/state/outbox operations.
- `SessionRepository.get_active_session()` and `list_active_session_states()` read active sessions through `session_ledger.get_state()` / `list_active_states()`, deriving active pointers from `tq_session_state`.
- `session_rebuild.py` reconstructs active state through `session_ledger.mark_active_states_no_active()` and `record_active_session(...)`.
- `session_projection.py` is pure projection logic and does not own SQL state.
- `tq_active_sessions` appears only in tests/residue guards asserting its removal, not in production runtime code under `queue_service` or `task_queue`.

## Optimization Outcome

No production cleanup was needed for this slice. The active-session table removal and state SSOT cutover appear complete at the schema/table-usage layer. Remaining audit work should focus on mutation/finalize/generation/outbox behavior, not table existence.
