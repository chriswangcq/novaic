# Result: Session ledger mutation API inventory

## Summary

Completed a read-only inventory of `SessionLedgerRepository` mutation APIs. The ledger is the session-specific adapter over the generic FSM SQLite store and owns session event/state/outbox row shape. Time and ID providers are injected at the ledger boundary. I did not find direct business dispatch branching inside the ledger, but the ledger does intentionally own session-domain event payload wrappers and generation helper semantics.

## Mutation API Inventory

`SessionLedgerRepository` constructor dependencies:

- `clock`: injected clock provider stored as `_clock`.
- `event_id_provider`: injected event ID provider stored as `_new_event_id`.
- `outbox_id_provider`: injected outbox ID provider stored as `_new_outbox_id`.
- `store`: optional injected `FsmSqliteStore`; defaults to configured generic store.

Direct event APIs:

- `append_event(...)`: writes generic session event rows and handles event idempotency via store lookup.
- `append_input_received(...)`: session-domain wrapper for `input_received` event payload shape.
- `record_input_consumed(...)`: writes `input_consumed` events and marks input events consumed.
- `record_pending_projection(...)`: writes `pending_projection_observed` event with deterministic pending projection.
- `record_session_finalize_rejected(...)`: writes `session_finalize_rejected` event.
- `record_session_finalized(...)`: writes `session_finalized` event and then materializes ending/no_active state.

State APIs:

- `upsert_state(SessionStateRecord)`: generic state upsert wrapper.
- `record_active_session(...)`: session-domain active state shape owner.
- `mark_active_states_no_active(...)`: bulk startup/rebuild helper to demote active states.
- `get_state(...)`, `list_active_states(...)`: read materialized state.
- `session_generation(...)`, `next_generation(...)`, `active_generation(...)`: generation calculation helpers based on current materialized state.

Outbox APIs:

- `append_outbox(...)`: writes durable outbox rows and enforces allowed initial statuses `pending` or `observed`.
- `list_outbox(...)`, `get_outbox(...)`, `list_pending_outbox(...)`: read outbox rows.
- `mark_outbox_published(...)`, `mark_outbox_failed(...)`: update outbox lifecycle.

Atomic transition API:

- `record_transition(...)`: builds a `FsmTransitionRecord` and delegates the mechanical event/state/outbox write order to `FsmTransitionRunner`. It also optionally records input consumption after the transition.

## Boundary Classification

- Storage mechanics: delegated to `FsmSqliteStore` and `FsmTransitionRunner`.
- Session row shape: owned by `SessionLedgerRepository` and `SessionStateRecord`.
- Business decision logic: not found as dispatch/finalize branching in the ledger; those decisions live in `session_fsm.py` and `session_repo.py`.
- Domain payload wrappers: present and intentional in the ledger for input/finalize/projection/active-state ownership. Existing tests explicitly assert this boundary.

## Tests / Guards Observed

- `test_pr272_session_active_state_ledger_boundary.py` asserts active state row shape is owned by the ledger and repo/outbox do not construct `SessionStateRecord` or call `.upsert_state()`.
- `test_pr268_session_input_ledger_boundary.py` asserts input payload shape and consumed markers are owned by the ledger and not duplicated in repo/outbox.
- `test_pr235_session_ledger.py` verifies session ledger event/state/outbox writes are atomic under an explicit DB transaction.

## Optimization Candidates

No production bug was found in this slice. One infrastructure note: `FsmSqliteStore` owns a SQLite busy retry policy with `time.sleep`; this is generic infrastructure policy rather than queue business logic, but if deterministic unit tests ever need to assert retry timing, sleeper injection would make that boundary cleaner.
