# Scan session mutation residue

## Problem Definition

Search the project for direct session table SQL, old repository wrappers, compatibility branches, and mutation shortcuts outside the intended ledger/repository/outbox boundary. Classify every relevant match and create cleanup follow-ups if production residue remains.

## Proposed Solution

Split the scan into focused child problems: direct session SQL/table access, old wrapper/compatibility/mutation shortcuts, and final classification/check. Use `rg`-based evidence with explicit categories instead of relying on intuition.

## Acceptance Criteria

- All searched patterns are listed with representative matches or no-match evidence.
- Matches are classified as legitimate adapter SQL, test guard, documentation, risky active path, or removable residue.
- Production residue, if found, becomes a follow-up cleanup problem rather than being waved away.
- Scan does not modify code unless a child/follow-up explicitly owns a cleanup.

## Verification Plan

- Run targeted `rg` patterns across runtime and relevant repo areas.
- Inspect source slices for ambiguous matches.
- Record classifications and source pointers.

## Risks

- Pattern-only scans can create false positives or miss semantically equivalent residue; ambiguous matches need source inspection.

## Assumptions

- The intended active boundary is session ledger/repository/outbox ownership, with tests and docs allowed to mention legacy names as guards or historical context.
