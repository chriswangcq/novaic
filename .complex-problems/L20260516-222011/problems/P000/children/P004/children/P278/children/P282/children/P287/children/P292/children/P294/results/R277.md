# Result: Session dispatch transaction flow audit

## Summary

Completed a read-only audit of `SessionRepository.dispatch()`. The dispatch path is largely aligned with the intended model: input is appended before decisions, live routing uses `decide_session_dispatch`, active attach is backed by durable session outbox, and start-wake creates durable outbox effects. I found no direct `TaskTopics.SESSION_ATTACH_INPUT` publish from `SessionRepository`.

However, I found an optimization/cleanup candidate: start-wake transition construction exists in two active branches, one inside the initial transaction for ordinary no-active start and one after the initial transaction for suspected-dead recovery start. Both use durable outbox, but the duplicated construction increases drift risk.

## Dispatch Flow Map

Initial transaction, `session_repo.py:106-356`:

- Appends input event first via `_append_input_event_in_current_transaction()`.
- Reads current state via `session_ledger.get_state(session_key)` and derives active session from `tq_session_state`.
- Active session branch:
  - If suspected-dead marker exists, records `session_suspected_dead_observed` transition and leaves recovery start for after the transaction.
  - Otherwise uses `_decide_live_dispatch()` / `decide_session_dispatch()`.
  - Attachable input builds `attach_request` only; it does not publish while holding the global transaction.
  - Non-attachable active input records `dispatch_buffered` and pending projection in the same transaction, then returns buffered.
- Occupied non-active state branch (`starting`, `ending`, `suspected_dead`, `recovering`): uses `decide_session_dispatch()` and records buffered transition/pending projection.
- Ordinary no-active branch: uses `_decide_live_dispatch()`, allocates scope/generation, builds `build_dispatch_wake_plan(...)`, records `dispatch_wake_start_queued` through `session_ledger.record_transition(...)`, and returns with outbox id.

After initial transaction:

- Attach path calls `_publish_attach_request_after_transaction()`.
  - It computes active generation in a separate transaction.
  - It builds `build_attach_input_effect(...)` with `expected_session_generation`.
  - It records transition/outbox via `_record_session_transition_after_transaction(... require_outbox=True)`.
  - It then calls `outbox_dispatcher.publish_attach_input_effect(outbox_id)`; tests show failure leaves retryable pending outbox rather than losing the input.
- Recovery start path builds another `build_dispatch_wake_plan(...)`, possibly adds recovery archive effect, records durable outbox transition with `require_outbox=True`, and returns queued wake start.

## Boundary Classification

- Input append: ledger-owned, transactional.
- Dispatch decision: pure FSM decision through `decide_session_dispatch`, with trace persisted.
- Buffer path: ledger transition plus pending projection; no external publish.
- Start-wake path: durable session outbox effect via `build_dispatch_wake_plan` and `session_ledger.record_transition`.
- Attach path: durable session outbox effect, then synchronous publish attempt through `SessionOutboxDispatcher`; failure stays retryable in outbox.
- Direct publish bypass: not found in `SessionRepository`; `test_pr248_attach_outbox_cutover.py` asserts no `_publish_attach_input_task` and no `TaskTopics.SESSION_ATTACH_INPUT` in repo source.

## Optimization Candidate

Duplicate start-wake construction should be cleaned up:

- Ordinary no-active branch: `session_repo.py:287-356` builds and records start-wake transition inside the initial transaction.
- Recovery branch: `session_repo.py:381-459` builds and records a very similar start-wake transition after the initial transaction.

This is not a data-loss bug because both paths use durable outbox and `require_outbox` where needed, but it is a maintainability/drift risk in exactly the area that used to fail from old/new path divergence. A small follow-up should extract one helper/plan path so ordinary start and recovery start share the same durable outbox recording logic where possible.
