# Result: Session rebuild projection flow audit

Audited rebuild/projection paths.

Findings:
- `session_rebuild.py` is a startup adapter boundary. It receives `db`, `SessionLedgerRepository`, and explicit `updated_at`; it does not read clock/env/id globals itself.
- Rebuild runs under a global transaction, marks active session-state rows no_active, then records active sessions from saga rows whose state is pending/running/launched. It does not use the removed `tq_active_sessions` table.
- `session_projection.py` contains pure pending-input projection helpers and has no env/time/uuid hidden inputs.
- Pending projection event shape is owned by `SessionLedgerRepository`, with tests guarding that `session_repo.py` no longer owns pending projection event shape.
- Active state row shape is owned by `SessionLedgerRepository`, with tests guarding repo/outbox do not construct `SessionStateRecord` directly.

Source guard run:
- No `tq_active_sessions` in production queue_service Python source.
- Rebuild has explicit `updated_at` and no direct os/uuid/time hidden inputs.
- Pending projection has no direct os/uuid/time hidden inputs.

No new gap found in this rebuild/projection audit slice.
