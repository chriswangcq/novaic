# Cortex scope_end identity contract

## Problem Definition

`handle_cortex_scope_end()` performs multiple mutations: it reads wake meta, archives a scope path, transitions input notifications to processed, removes Cortex bridge entries, and marks participant status completed. The handler currently requires `scope_id/agent_id/user_id`, but it does not require the positive `session_generation` now available in wake-finalize context, and `wake_finalize._build_cortex_scope_end_payload()` does not pass generation into this handler.

## Proposed Solution

1. Inspect whether Cortex archive is safe by immutable `scope_id/wake_scope_path` alone.
2. For any mutation that depends on current wake/session identity rather than immutable scope path, add an explicit positive `session_generation` contract to the payload and handler.
3. Update `wake_finalize._build_cortex_scope_end_payload()` to propagate generation if the handler needs it.
4. Add focused tests for missing generation and zero generation when invoking `handle_cortex_scope_end()` from finalize payloads.
5. Preserve existing guarantees:
   - archive happens before notification processed-transition.
   - processed-transition stays best-effort after archive success.
   - structural scope_end never writes summary text.

## Acceptance Criteria

- The Cortex scope_end handler either rejects missing/non-positive generation or documents with code-backed evidence why the mutation is safe without generation.
- If generation is required, `wake_finalize` passes it into the Cortex scope_end payload.
- Existing scope_end ordering, summary, and notification tests still pass.
- Tests cover the stale/malformed identity risk relevant to the chosen contract.
- No compatibility fallback treats missing generation as zero or valid.

## Risks

- Requiring generation in scope_end may break recovery archive paths if P351 has not yet propagated generation. If discovered, record a child/follow-up rather than adding fallback.
- Cortex archive itself may only need immutable scope identity, while notification processing may be session-lifecycle semantics; the result must distinguish those cases instead of making a blanket claim.

## Assumptions

- P349 guarantees normal React-generated wake_finalize contexts have positive `session_generation`.
- Recovery-created scope_end payloads are handled separately by P351 if they lack generation.
