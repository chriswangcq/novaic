# Split-first comprehensive optimization audit

## Problem Definition

The root problem is broad and high-impact: review the current multi-repo NovaIC project for optimizable points, implement high-confidence fixes, delete residue, and verify the system. The scope spans runtime/tool output contracts, Cortex context and projection, shell CLI capabilities, queue/FSM state boundaries, LogicalFS/sandbox/blob layering, tests, docs, and deployment-facing process health.

Because this is not a small low-risk task, it must not be handled as one one-go ticket. The solution path is to split into independent audit/remediation child problems, solve each frontier leaf with evidence, then only summarize at the parent level after all children close.

## Proposed Solution

Use a split-first ledger plan with dedicated child problems for each audit surface. Each child problem will inspect code and tests, identify concrete optimization opportunities, implement safe changes when found, run targeted checks, and record residual risks. The parent ticket only coordinates and summarizes child results.

Initial child surfaces should include:

- Repo topology, dirty-state, branch/submodule, and deployment-process sanity.
- Runtime shell/tool-output/display projection contract and residue cleanup.
- Cortex context assembly, event-source, prompt/history projection, and payload boundary.
- Shell capability CLI coverage and hidden-tool residue.
- Queue/FSM/session/outbox boundaries and old imperative branch residue.
- LogicalFS/sandbox/blob layering and direct-backdoor residue.
- Test, docs, and architecture guard coverage for the above contracts.

## Acceptance Criteria

- Root ticket is classified as split, not one_go.
- Child problems cover all major audit surfaces rather than hiding work in a single broad execution.
- Each child has its own result and check before the parent records a result.
- High-confidence fixes are implemented and verified, not only described.
- If a child uncovers deeper work, it creates spawn/follow-up problems instead of compressing risk into a summary.
- Final parent check maps all root success criteria to child evidence.

## Verification Plan

- Use `ledger.py next` after each state-changing action.
- Validate and render the ledger before final reporting.
- Run targeted tests for touched repositories.
- Run `git diff --stat`, focused diff review, and residual residue searches.
- Confirm the repo is in a coherent state with no accidental unrelated edits.

## Risks

- The workspace is a large multi-repo tree; a superficial audit would miss active old paths.
- Some checks may be expensive; use targeted tests first and escalate when changes touch shared contracts.
- Existing submodule boundaries may require separate commits later, but this ticket should first produce correct local code and evidence.
- Over-adding code is a risk; prefer simplification and deletion when old logic is obsolete.

## Assumptions

- Current branch `codex/reliable-evolution-fsm` is the intended working branch.
- The user wants fixes performed locally now, but commit/deploy/push are not requested in this message.
- Backward compatibility with obsolete paths is not required unless a current test or deployment contract proves it is active.
