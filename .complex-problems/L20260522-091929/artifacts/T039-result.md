# Entangled Postgres DDL dialect result

## Summary

P043 added dialect-aware Entangled DDL and schema inspection. SQLite DDL remains the default and is unchanged for existing callers; Postgres DDL can now map field types, add `entangled_rowid`, inspect columns through `information_schema`, and let schema registration choose the proper path from the database backend.

## Done

- Added `FieldDef.sql_type_for(dialect)` and dialect-aware column DDL helpers.
- Added Postgres field mappings:
  - `TEXT -> text`
  - `INTEGER -> bigint`
  - `REAL -> double precision`
  - `BLOB -> bytea`
  - `JSON -> jsonb`
  - `BOOL -> boolean`
  - `TIMESTAMP -> text`
- Added Postgres JSON/default rendering and timestamp-text default rendering.
- Suppressed Postgres FK enforcement from field `ref` clauses for first-cutover safety.
- Added dialect-aware `SqlEntityDef.create_table_sql`, `index_sqls`, and `alter_add_column_sqls`.
- Added `entangled_rowid bigint generated by default as identity` to Postgres dynamic entity tables and indexes.
- Added `table_columns()` to SQLite and Postgres DB adapters.
- Updated `SqlEntityStore.ensure_schema_unlocked` to use backend-specific DDL and table-column discovery.
- Added focused DDL dialect tests.

## Verification

- `python -m pytest tests/test_postgres_ddl_dialect.py tests/test_postgres_database_boundary.py` passed: `11 passed`.
- Full Entangled test suite passed: `86 passed`.
- py_compile passed for touched SQL modules.

## Known Gaps

- Entity-store CRUD/list/upsert SQL conversion is intentionally deferred to P044.
- Sync-version and transition support-table Postgres behavior is intentionally deferred to P045.
- No production Entangled runtime or data migration was changed.
