# Port Entangled sync-version and transition-log persistence to Postgres

## Problem Definition

Entangled support tables still use SQLite DDL and upsert semantics. `entangled_sync_versions` must become a monotonic Postgres upsert, and `subagent_state_transitions` must use a generated identity ID while preserving append-only row shape and transition atomicity.

## Proposed Solution

1. Add backend-aware DDL for `entangled_sync_versions`.
2. Change version bump persistence so Postgres uses `GREATEST(existing.version, excluded.version)` and SQLite retains existing syntax.
3. Add backend-aware DDL for `subagent_state_transitions`, using `bigint generated by default as identity` in Postgres.
4. Keep transition append/list row shapes unchanged.
5. Add a helper or documented SQL for resetting Postgres transition identity above a migrated max ID.
6. Add tests for SQLite DDL compatibility, Postgres DDL generation, monotonic upsert SQL, transition DDL, rollback behavior, and row-shape output.

## Acceptance Criteria

- `entangled_sync_versions` Postgres DDL uses `state_key text primary key` and `version bigint`.
- Postgres version persistence is monotonic and cannot regress versions.
- SQLite version persistence remains unchanged.
- `subagent_state_transitions` Postgres DDL uses generated identity `bigint`.
- Transition append/list output remains API-compatible.
- Migration sequence reset expectation is represented.
- Full Entangled tests pass.

## Verification Plan

Run focused support-table tests, full Entangled pytest, and py_compile. Do not run production migration or cutover in this child.

## Risks

- Sync-version regression would cause missed deltas or bad client sync.
- Transition ID sequence reset must be handled by migration tooling before production cutover.

## Assumptions

- P038 database backend boundary exposes `backend_name`.
- Migration tooling will call sequence reset after importing production transition rows.
