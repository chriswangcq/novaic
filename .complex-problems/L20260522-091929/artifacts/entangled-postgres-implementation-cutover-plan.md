# Entangled Postgres Implementation and Cutover Plan

Inputs:

- P001 Postgres foundation: `novaic-postgres` on `api.gradievo.com`, local-only `127.0.0.1:5432`, database/user `novaic_entangled`.
- P018 inventory: `.complex-problems/L20260522-091929/artifacts/entangled-sqlite-inventory.md`.
- P019 semantic mapping: `.complex-problems/L20260522-091929/artifacts/entangled-postgres-semantics.md`.

Scope: requirements and cutover planning only. P020 does not cut over production and does not change Entangled code or runtime configuration.

## Target Policy

- Production target database: `novaic_entangled` on the existing `novaic-postgres` Docker service.
- Production Entangled must not keep a maintained SQLite fallback after cutover.
- Integration/test environment should run Postgres.
- Development unit tests may use narrow fakes or adapter-level test doubles, but they must not require a second production SQLite implementation path.
- Any remaining SQLite use after cutover must be explicitly named as test-only or rollback-only residue.

## Phase 0: Preconditions

Required before implementation starts:

- P018 live inventory exists and is current enough for schema design.
- P019 semantic mapping is accepted.
- `novaic_entangled` database and role can connect from the Entangled runtime host/container.
- Secret storage path is chosen, likely under `/opt/novaic/postgres/secrets/novaic_entangled_password` or a service-specific env file with `0600 root:root`.
- Decide timestamp storage policy:
  - either preserve timestamp-like columns as `text` for first cutover;
  - or use `timestamptz` with adapter serialization proving identical wire output.
- Decide FK enforcement policy:
  - first cutover should use no FK enforcement or `NOT VALID` constraints until a data audit passes, because SQLite runs with FK checks off.
- Decide `rowid` replacement:
  - recommended: add `entangled_rowid bigint generated by default as identity` to dynamic PG entity tables and migrate SQLite `rowid` values.

Do not proceed if any of these decisions are unresolved.

## Phase 1: Config and Dependency Boundary

Implementation requirements:

- Add Entangled config for `NOVAIC_ENTANGLED_DB_BACKEND=postgres` and `NOVAIC_ENTANGLED_DATABASE_URL` or equivalent DSN.
- Keep current `--db-path` only for legacy/rollback mode during implementation; do not let production default silently fall back to SQLite.
- Add a startup guard:
  - if production backend is unset or `sqlite`, fail fast after the cutover ticket marks Postgres required;
  - before cutover, allow explicit `sqlite` only for current production runtime.
- Add Postgres driver dependency and a small DB adapter boundary.
- Keep `Database` interface stable enough for existing callers: `execute`, `fetchone`, `fetchall`, `transaction`, `get_connection`, rowcount, and dict-like rows.

Verification:

- Unit tests can instantiate adapter with a fake DSN or test container.
- SQLite-specific imports do not leak into Postgres runtime path except legacy rollback module.

## Phase 2: Postgres Adapter

Implementation requirements:

- Use a connection pool.
- Return dict-like rows.
- Convert placeholders safely. Prefer query builder/helper methods over global string replacement.
- Implement transaction context with commit/rollback and optional advisory lock mapping:
  - `transaction("global")`: transaction-scoped advisory lock for compatibility.
  - `transaction(lock_type, resource_id=...)`: transaction-scoped advisory lock unless the specific operation uses row locks.
- Classify transient Postgres exceptions and log them with operation context.
- Implement `RETURNING` support for auto-integer inserts.
- Preserve rowcount behavior.

Verification:

- Adapter contract tests for `fetchone`, `fetchall`, `rowcount`, rollback on exception, advisory lock serialization, and auto-id returning.

## Phase 3: Postgres DDL Generator

Implementation requirements:

- Add Postgres dialect for `FieldDef` and `SqlEntityDef`.
- Map:
  - `TEXT` to `text`;
  - `INTEGER` to `integer` or `bigint`, with identity for auto primary keys;
  - `REAL` to `double precision`;
  - `BLOB` to `bytea`;
  - `JSON` to `jsonb`;
  - `BOOL` to `boolean`;
  - `TIMESTAMP` per Phase 0 decision.
- Add `entangled_rowid` to dynamic tables where stream/list cleanup semantics depend on SQLite `rowid`.
- Replace `PRAGMA table_info` with catalog inspection.
- Preserve `CREATE TABLE IF NOT EXISTS`, `ALTER TABLE ADD COLUMN`, and create-index-after-column ordering.
- Represent SQLite autoindexes through PG primary keys and unique constraints.
- Enforce identifier validation or safe identifier quoting.

Verification:

- Generate PG DDL for all live P018 tables.
- Diff expected table/column/index inventory against P018 schema.
- Verify additive schema registration can add a new indexed column.

## Phase 4: Entity Store SQL Conversion

Implementation requirements:

- Convert CRUD/list/list_stream/upsert SQL to Postgres-safe SQL.
- Preserve fail-fast non-auto id behavior.
- Convert `lastrowid` to `RETURNING`.
- Convert `datetime('now')` expressions to the Phase 0 timestamp policy.
- Replace SQLite `rowid` with `entangled_rowid`.
- Preserve:
  - `_scope_where` user/key-param/parent filters;
  - `filters`, `in_filters`, `not_in_filters`;
  - `default_not_in_filters`;
  - hidden-field masking and `has_<hidden>` behavior;
  - JSON/BOOL/TIMESTAMP input/output conversion;
  - `ON CONFLICT(id_field) DO UPDATE`;
  - rowcount-driven update/delete/CAS behavior.

Verification:

- Unit tests for create/get/update/delete/upsert/list/list_stream/append/batch/update_where/delete_where/cleanup/cas_update.
- Tests for stream pagination with duplicate cursor timestamp values.
- Tests for JSONB round trip and bool output shape.

## Phase 5: Sync Versions and Transition Logs

Implementation requirements:

- Create `entangled_sync_versions(state_key text primary key, version bigint not null default 0 check (version >= 0))`.
- Use monotonic upsert:
  - `version = GREATEST(entangled_sync_versions.version, EXCLUDED.version)`.
- Load all versions before client sync decisions are made.
- Preserve state-key string generation exactly.
- Map `subagent_state_transitions.id` to `bigint generated by default as identity`.
- Keep `subagent_state_transitions.created_at` as bigint milliseconds for first cutover.
- Keep transition update plus history append atomic.
- Use `SELECT ... FOR UPDATE` on the target `subagents` row, or a documented advisory-lock compatibility bridge.

Verification:

- Sync-version migration equality: every `state_key` and `version` matches SQLite before cutover.
- Monotonic upsert test where an older version cannot overwrite a newer version.
- Transition test: status update and history insert commit or rollback together.
- Sequence reset test: next transition id is greater than migrated max id.

## Phase 6: Migration Tooling

Implementation requirements:

- Build an offline migration command that:
  - stops Entangled writers before final export;
  - reads SQLite with `sqlite3 -readonly`;
  - exports all tables, including SQLite `rowid` for entity tables that need `entangled_rowid`;
  - transforms JSON/TIMESTAMP/BOOL per PG schema policy;
  - imports into a clean `novaic_entangled`;
  - resets identity sequences;
  - writes a migration report with counts and checksums where feasible.
- No dual-write migration. Use a single cutover window to avoid two sources of truth.
- Include a dry-run mode against a staging/test Postgres database.

Pre-migration checks:

- Entangled health/ready green.
- Capture fresh row counts for every SQLite table.
- Capture `entangled_sync_versions` full key/value set.
- Capture max `subagent_state_transitions.id`.
- Confirm DB file holders before stopping service.

Post-import checks:

- Every table row count equals SQLite snapshot.
- Every sync-version key/value equals SQLite snapshot.
- `subagent_state_transitions` count and max id equal SQLite snapshot.
- `entangled_rowid` count equals table count where applicable.
- Representative JSON columns decode.
- Representative timestamp columns return expected wire format.
- Schema/index inventory matches expected PG plan.

## Phase 7: Test Environment Validation

Required integration tests:

- Start Entangled in PG mode against test `novaic_entangled`.
- Register all production schemas.
- Run REST smoke tests:
  - list `agents`;
  - get/upsert singleton-style entity such as `agent-state`;
  - append/query stream entity such as `messages` or `environment-notifications`;
  - update/delete a disposable test entity row.
- Run WebSocket smoke tests:
  - connect to `/v1/sync`;
  - receive schema frame with all expected live entities;
  - entangle list/form/stream representatives;
  - verify full/head sync;
  - perform write and verify delta/push;
  - restart Entangled and verify reconnect behavior uses persisted versions safely.
- Run compatibility tests against the app client if available.

Exit criteria:

- No SQLite file is opened in PG mode.
- No schema registration failure.
- No timestamp or JSON shape regression in API responses.
- No sync-version reset.

## Phase 8: Production Cutover Window

Recommended production order:

1. Announce/enter maintenance window for Entangled-backed realtime state.
2. Freeze or stop upstream writers that publish entity changes:
   - gateway/business/subscriber paths that write to Entangled;
   - Entangled service itself last, after writers stop.
3. Confirm no active writes and record final health/readiness.
4. Stop Entangled.
5. Confirm no process holds `/opt/novaic/data/entangled.db*`.
6. Run final SQLite export and PG import into `novaic_entangled`.
7. Run post-import checks.
8. Start Entangled in PG mode.
9. Re-register schemas from upstream services.
10. Run health/ready checks.
11. Run REST and WS smoke tests.
12. Resume upstream writers.
13. Watch logs for DB, schema, sync, and WebSocket errors.

Do not cut over if:

- row counts differ without a documented reason;
- any sync-version value differs or decreases;
- `entangled_rowid` replacement is absent for stream tables;
- schema registration fails;
- ready is not HTTP 200;
- WS schema/full/delta/reconnect smoke tests fail;
- Entangled in PG mode still opens `/opt/novaic/data/entangled.db*`.

## Phase 9: Rollback Boundaries

Before PG writers start:

- Rollback is simple. Stop PG-mode Entangled, restart SQLite-mode Entangled with the original `/opt/novaic/data/entangled.db`, and keep PG import as failed-attempt evidence.

After PG writers start but before client-visible deltas:

- Prefer stopping Entangled and replaying or discarding PG-only writes based on the migration report.
- If any sync versions advanced in PG, do not restart SQLite unless the version delta is understood and accepted.

After client-visible deltas or external side effects:

- Rollback is not symmetric.
- Options:
  - keep PG and fix forward;
  - export PG changes back to SQLite with explicit replay tooling;
  - accept loss of PG-only changes and force full resync, only if product owner approves.
- This boundary must be called out in the production runbook before cutover.

## Stabilization and Cleanup

Stabilization window:

- Keep old SQLite files rollback-only for a defined period, recommended 7-14 days.
- Mark them clearly:
  - `entangled.db.NON_CURRENT_POSTGRES_CUTOVER.md`;
  - mode `0600`;
  - owner `root:root`;
  - no process should hold them.
- Monitor:
  - Entangled health/ready;
  - schema registration errors;
  - PG connection pool errors;
  - sync-version monotonicity;
  - WS reconnect/full-sync rates;
  - app-side stale data reports.

Cleanup criteria:

- PG mode stable through the window.
- Fresh PG backup exists.
- No process opens `/opt/novaic/data/entangled.db*`.
- All downstream services point to PG-mode Entangled.
- User-approved cleanup ticket exists.

Cleanup action:

- Move old SQLite files to `/opt/novaic/residue-archive/entangled-sqlite-postgres-cutover-<timestamp>/`.
- Keep a short marker in `/opt/novaic/data/` explaining that Entangled is PG-backed.
- Update any operator docs and residue classification docs.
