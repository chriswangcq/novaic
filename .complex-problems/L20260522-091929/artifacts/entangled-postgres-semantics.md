# Entangled Postgres Semantic Requirements

Inputs:

- P018 live inventory: `.complex-problems/L20260522-091929/artifacts/entangled-sqlite-inventory.md`
- Local source under `Entangled/packages/server-python/entangled/`

Scope: requirements only. No production Entangled migration, runtime change, or code edit was performed.

## Current SQLite Shape

Entangled is a dynamic SQL entity service. The live DB currently has:

- 22 registered entities reported by `/v1/health`.
- 29 SQLite tables including entity tables, support tables, `entangled_sync_versions`, `subagent_state_transitions`, and `sqlite_sequence`.
- `entangled_sync_versions`: 67 rows, max version 5319.
- `subagent_state_transitions`: 184 rows, id range 15-542.
- SQLite foreign key clauses in DDL, but runtime `PRAGMA foreign_keys = OFF`.
- WAL mode and multiple thread-local handles from the same host process.

The Postgres migration must preserve application semantics, not just table names.

## Database Adapter

Current SQLite owner:

- `sql/database.py`
- Interface used by callers: `connect`, `close`, `execute`, `executemany`, `fetchone`, `fetchall`, `fetch_all`, `commit`, `rollback`, `transaction`, `get_connection`, `vacuum`.
- Current connection behavior: thread-local `sqlite3.Connection`, `row_factory = sqlite3.Row`, `check_same_thread=False`, WAL, `foreign_keys=OFF`, `synchronous=NORMAL`, `busy_timeout=3000`.
- Current transaction behavior: Python FIFO locks from `DatabaseLockManager` wrap a connection transaction, then commit or rollback.

Postgres requirements:

- Introduce an explicit Postgres adapter behind the same minimal DB interface, backed by a connection pool.
- Production Entangled should use DSN-based Postgres config for `novaic_entangled`; do not keep a production SQLite fallback path.
- Preserve dict-like row output so `dict(row)`, `row["field"]`, `fetchone`, and `fetchall` callers do not change.
- Replace SQLite `?` placeholders with Postgres placeholders inside the adapter/repository layer, not by scattered call-site string hacks.
- Preserve `rowcount` for update/delete/CAS semantics.
- Replace `lastrowid` with `INSERT ... RETURNING <id_field>` for auto-integer primary keys.
- Treat SQLite busy handling as obsolete. Postgres transient errors should be classified explicitly: deadlock, lock timeout, connection loss, serialization failure if a stronger isolation level is ever used.
- Use Postgres `READ COMMITTED` unless a specific operation documents a stronger need.
- Do not rely on Python FIFO locks for correctness. For operations using `transaction(lock_type, resource_id=...)`, either:
  - map global/resource locks to `pg_advisory_xact_lock(hash(lock_type, resource_id))`, or
  - replace them with row locks and unique constraints where the affected row is known.
- Keep `transaction("global")` serializing schema registration, sync-version writes where needed, and cross-table subagent transition updates until narrower row-lock proofs exist.

## Field and Type Mapping

Current owner:

- `sql/field_def.py`

SQLite field storage:

| FieldKind | SQLite storage | Current Python behavior |
|---|---|---|
| `TEXT` | `TEXT` | unchanged |
| `INTEGER` | `INTEGER` | unchanged |
| `REAL` | `REAL` | unchanged |
| `BLOB` | `BLOB` | unchanged |
| `JSON` | `TEXT` | `json.dumps` on input, `json.loads` on output with fallback to original string |
| `BOOL` | `INTEGER` | bool to `0/1`, output to `bool(value)` |
| `TIMESTAMP` | `TEXT` | no automatic DB parse; `default="NOW"` in Python uses canonical ISO UTC `Z` |

Postgres requirements:

- `TEXT` maps to `text`.
- `INTEGER` maps to `integer` or `bigint`; auto-integer primary keys should use `bigint generated by default as identity` plus `RETURNING`.
- `REAL` maps to `double precision` unless a schema opts into numeric precision later.
- `BLOB` maps to `bytea`.
- `JSON` should map to `jsonb`, with adapter output preserving current Python shape. Valid JSON strings should become dict/list/scalar on output as today; invalid legacy JSON strings must not crash reads unless the existing SQLite path would have crashed.
- `BOOL` should map to `boolean`, with adapter output preserving Python bools.
- `TIMESTAMP` requires a wire-compatibility decision before implementation:
  - preferred long-term storage is `timestamptz`;
  - first cutover must preserve the JSON/API wire shape expected by clients, so the adapter must convert returned datetimes to the same string format currently returned, or the first PG schema should preserve timestamp-like fields as text until a separate timestamp normalization is tested.
- SQLite `datetime('now')` defaults must be converted deliberately. Do not rely on accidental DB-specific timestamp text.
- JSONB defaults must use typed defaults such as `'{}'::jsonb` or `'[]'::jsonb`.

## Generated DDL and Schema Registration

Current owners:

- `sql/entity_def.py`
- `sql/entity_store.py`
- `app/schema.py`

Current behavior:

- Schema registration POSTs JSON specs, converts them to `SqlEntityDef`, validates the batch, then under `store.db.transaction("global")` calls `store.register(defn)` and `store.ensure_schema_unlocked(defn)`.
- `ensure_schema_unlocked` currently:
  - executes `CREATE TABLE IF NOT EXISTS`;
  - reads columns through `PRAGMA table_info`;
  - applies `ALTER TABLE ADD COLUMN` for missing columns;
  - creates indexes after columns.
- Column-before-index ordering is intentional and must be preserved.
- `SqlEntityDef.create_table_sql()` can emit single-column or composite primary keys.
- Field-level `index=True` emits `CREATE INDEX IF NOT EXISTS idx_<table>_<column> ON <table>(<column>)`.
- Live DDL includes foreign key clauses, but SQLite runtime has foreign key enforcement disabled.

Postgres requirements:

- Provide a Postgres DDL generator instead of reusing SQLite DDL strings.
- Replace `PRAGMA table_info` with `information_schema.columns` or `pg_catalog` inspection.
- Preserve idempotency:
  - create table if missing;
  - add only missing columns;
  - create indexes only after missing columns are added.
- Quote identifiers through a safe identifier API or keep existing validation strict enough that all table/column names are safe unquoted identifiers. Do not interpolate unvalidated names.
- Preserve table-level `UNIQUE`, `CHECK`, and primary-key constraints.
- Foreign keys must not become accidentally stricter than production SQLite. First implementation should either:
  - create FK constraints as `NOT VALID` and validate after a data audit, or
  - omit FK enforcement initially and document a later hardening pass.
- Composite primary keys must map to Postgres composite primary keys.
- SQLite autoindexes map to primary-key and unique constraints, not separate named indexes.

## Entity Store Queries

Current owner:

- `sql/entity_store.py`

### Scope and Security Filters

Current behavior:

- `_scope_where` builds `user_id = ?`, key-param filters, and parent-ownership subqueries.
- Missing parent entity registration raises a `ValueError`.

Postgres requirements:

- Preserve the same scope filter semantics and parent subqueries.
- Convert placeholders consistently.
- Keep field-name validation before dynamic filter/order SQL is built.
- Add tests for user-scoped, key-param scoped, and parent-scoped entities.

### CRUD

Current behavior:

- `_sql_get`: scoped `SELECT *`.
- `_sql_create` and `append`: caller must provide a non-auto primary key; Entangled does not mint ids for non-auto-int keys.
- Auto-integer primary keys use `cur.lastrowid`.
- `_apply_defaults` fills non-null fields with explicit defaults before insert/update checks.
- `_check_required` raises named missing-field errors before SQL.
- `_sql_update`, `update_where`, `batch_update`, `cas_update`: build `SET col = ?`, optionally update `updated_at = datetime('now')`, use rowcount for result.
- `_sql_delete`, `delete_where`, `cleanup`: rowcount drives deletion result and notification behavior.
- `_sql_upsert`: `INSERT ... ON CONFLICT(id_field) DO UPDATE SET ...`.

Postgres requirements:

- Preserve fail-fast missing-id behavior for non-auto primary keys.
- For auto-int IDs, use `RETURNING` and set the output id exactly where SQLite used `lastrowid`.
- Convert all `datetime('now')` update expressions to an adapter-defined expression that preserves client-visible output.
- Preserve `_apply_defaults` and `_check_required` in Python; do not push this behavior entirely into DB defaults.
- Preserve rowcount-driven responses for update/delete/CAS.
- Keep upsert conflict target on the entity `id_field`, but verify entities with additional unique constraints do not need separate conflict paths.
- Use one transaction per existing write operation.
- Keep notification calls outside or after the DB write path as they are today; migration must not publish deltas for failed DB transactions.

### List and Stream Pagination

Current behavior:

- `list` orders by validated `default_order` or request `order_by`.
- `list_stream` supports `filters`, `in_filters`, `not_in_filters`, `before_id`, `after_id`, `limit`, `order_by`, `cursor_field`, and default-not-in filters.
- `list_stream` and `exists_before` use SQLite `rowid` as a stable tie-breaker when comparing rows with equal cursor field values.
- `cleanup` can order by `rowid DESC` when no better default exists.

Postgres requirements:

- Replace SQLite `rowid` with an explicit stable per-table tie-break column.
- Recommended first-cut requirement: every PG entity table gets an internal `entangled_rowid bigint generated by default as identity` column, and migration copies SQLite `rowid` into it for existing rows using `OVERRIDING SYSTEM VALUE` or a by-default identity sequence.
- Update stream queries from `rowid` to `entangled_rowid`, preserving `(cursor_field, rowid)` ordering semantics.
- For new rows after migration, ensure `entangled_rowid` increases monotonically.
- Add indexes for common stream order pairs, for example `(agent_id, timestamp, entangled_rowid)` or the entity's key params plus cursor field plus `entangled_rowid`.
- `after_id` currently compares primary key strings/ints directly, not cursor positions. Preserve or explicitly fix only with client compatibility tests.

## Sync Version Persistence

Current owners:

- `sql/persistence.py`
- `server/sync.py`
- `app/factory.py`
- `app/ws.py`

Current behavior:

- `entangled_sync_versions(state_key TEXT PRIMARY KEY, version INTEGER NOT NULL DEFAULT 0)`.
- Startup ensures the table, initializes sync engine, then `load_all_sync_versions` hydrates registry versions.
- `SyncRegistry.hydrate_versions` sets each state current version to the max of current and persisted version.
- Version bump callback upserts `state_key, version` after record_op.
- State keys are either bare entity names or `entity:<json sorted params>`.
- Reconnect can use versions and op-log; if op-log gap exists, full/head sync is required.

Postgres requirements:

- Use `state_key text primary key`, `version bigint not null default 0`, and `CHECK (version >= 0)`.
- Upsert must be monotonic:
  - `INSERT ... ON CONFLICT(state_key) DO UPDATE SET version = GREATEST(entangled_sync_versions.version, EXCLUDED.version)`.
- Load all versions before clients can depend on sync decisions.
- Keep state-key string generation exactly stable; do not change JSON sorting, separators, entity naming, or param shape.
- Add a migration check that SQLite and Postgres have identical state-key counts and identical version values before cutover.
- Add a startup guard that refuses PG mode if persisted versions are absent while entity tables contain data, unless an explicit first-migration flag is present.

## Subagent Transition Semantics

Current owners:

- `sql/subagent_state.py`
- `sql/state_transitions.py`
- `app/state_transitions.py`
- `app/subagent_state.py`

Current behavior:

- `transition()` runs under `db.transaction("global")`.
- It selects the `subagents` row, validates state transition, updates `subagents.status` plus allowed extras and `updated_at`, then appends exactly one `subagent_state_transitions` row for non-noop transitions.
- Self-loop transitions may update extras but do not append transition history.
- `subagent_state_transitions.id` is `INTEGER PRIMARY KEY AUTOINCREMENT`.
- `metadata_json` is text JSON; `created_at` is integer milliseconds.
- Listing transitions orders by `id ASC`.

Postgres requirements:

- Run transition under one Postgres transaction.
- Prefer `SELECT ... FOR UPDATE` on `subagents` scoped by `(subagent_id, agent_id)` instead of only a global advisory lock.
- If the global lock remains for first cutover, use transaction-scoped advisory lock and document it as a compatibility bridge.
- Map transition `id` to `bigint generated by default as identity`; migrate existing ids and reset sequence above max id.
- Keep `created_at` as bigint milliseconds for first cutover to preserve API shape.
- Map `metadata_json` to `jsonb` only if output conversion returns the same Python/JSON shape; otherwise preserve text for first cutover.
- Keep `ORDER BY id ASC` for history.
- Ensure update plus append remains atomic.

## WebSocket and Client Compatibility

Current owners:

- `app/ws.py`
- `server/sync.py`
- `server/ws_handler.py`
- `server/protocol.py`

Compatibility risks and requirements:

- Schema push on connect uses in-memory `store.get_schema()`. PG migration must preserve schema registration order and schema dict shape so client schema handling and hash behavior do not change.
- Full sync depends on `list` / `list_stream` ordering and serialization. PG must return the same row shapes, hidden-field masking, JSON values, bool values, timestamp strings, and id values.
- Delta sync depends on write success followed by notifier behavior. PG writes must commit before deltas are published.
- Reconnect depends on monotonic `entangled_sync_versions`. Versions must not reset or decrease during migration.
- Op-log gaps after restart already trigger full/head sync. PG migration should not pretend to provide deltas when the op-log is empty and persisted versions are ahead.
- Stream entities require special smoke tests because of the SQLite `rowid` tie-break replacement.

Required smoke tests before PG production cutover:

- Connect over `/v1/sync` and verify schema frame contains all 22 live entities.
- Entangle representative list entity: `agents`.
- Entangle representative form/singleton-style entity: `agent-state` or `agent-binding`.
- Entangle representative stream entity: `messages` or `environment-notifications`.
- Perform create/update/delete or append through service APIs and verify both REST response shape and WS delta shape.
- Restart Entangled against PG, reconnect, and verify full/head sync behavior with persisted versions.

## Implementation Blockers

- A Postgres adapter must exist before any runtime cutover.
- SQLite `rowid` replacement must be designed and migration-tested before stream/list cleanup behavior can be trusted.
- Timestamp storage and wire format must be decided and tested.
- Foreign-key enforcement strategy must be explicit because SQLite currently runs with FK checks off.
- Sync-version upsert must be monotonic.
- Migration tooling must compare every table count, every sync version value, and representative JSON/timestamp decode results before PG mode starts.
