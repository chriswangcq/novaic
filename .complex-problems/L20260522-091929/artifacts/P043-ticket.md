# Implement Entangled Postgres DDL dialect

## Problem Definition

`FieldDef`, `SqlEntityDef`, and `SqlEntityStore.ensure_schema_unlocked` are SQLite-oriented: SQLite storage types, `datetime('now')`, inline `AUTOINCREMENT` style assumptions, and `PRAGMA table_info`. P043 must add a Postgres DDL/schema-inspection path while leaving SQLite behavior intact.

## Proposed Solution

1. Add dialect-aware DDL methods to `FieldDef` for SQLite and Postgres column fragments.
2. Add dialect-aware methods to `SqlEntityDef` for Postgres `CREATE TABLE`, indexes, and missing-column `ALTER TABLE` statements.
3. Add a stable `entangled_rowid bigint generated by default as identity` internal column to Postgres dynamic entity tables, except where it would conflict with an existing field.
4. Keep JSON fields as `jsonb`, BOOL as `boolean`, BLOB as `bytea`, REAL as `double precision`, INTEGER as `bigint` for first migration safety, and TIMESTAMP as `text` to preserve current wire shape.
5. Add a dialect helper on the database boundary so schema registration can choose SQLite `PRAGMA` or Postgres catalog inspection.
6. Update `ensure_schema_unlocked` to use dialect-specific DDL and existing-column discovery.
7. Add tests for field DDL, entity DDL, catalog-inspection dispatch, `entangled_rowid`, and unchanged SQLite DDL.

## Acceptance Criteria

- SQLite DDL output remains backward compatible.
- Postgres DDL maps every `FieldKind` to the agreed first-cutover type.
- Postgres `CREATE TABLE` includes `entangled_rowid` for dynamic entity tables.
- Postgres missing-column detection does not use `PRAGMA`.
- Postgres index DDL is generated after missing-column alter statements.
- Unsafe identifiers are still rejected by existing validation.
- Focused tests and full Entangled tests pass.

## Verification Plan

Run new DDL dialect tests, existing Entangled tests, and py_compile for touched modules. Do not run production migration or change production runtime in this child.

## Risks

- Adding `entangled_rowid` universally may conflict with a future entity field if not guarded.
- Using `jsonb`/`boolean` changes storage type, so later entity-store output conversion must preserve API shape.
- Foreign-key references should not become stricter than current SQLite runtime behavior.

## Assumptions

- P038 adapter boundary exposes `backend_name`.
- Full query behavior and row output conversion are handled in P044.
- FK hardening remains deferred.
