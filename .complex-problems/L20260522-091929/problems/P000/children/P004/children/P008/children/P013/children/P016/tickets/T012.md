# Define Queue Session Outbox Idempotency Postgres Replay Map

## Problem Definition

Queue replay safety depends on session event/state transitions, durable outbox rows, and task idempotency ledger behavior. The SQLite implementation serializes many of these operations in local transactions, while outbox publication can intentionally happen outside the original session transition and be retried. The Postgres migration needs an explicit replay map so no session inputs are lost, no effects are double-published beyond the current idempotency contract, and duplicate idempotency keys keep current API behavior.

## Proposed Solution

Produce a durable mapping artifact that translates the current session, outbox, and idempotency behavior to Postgres:

- Session dispatch/session-ended/rebuild flows and their event/state/outbox writes.
- Session event consumption and pending-input projection behavior.
- Session, saga, task, and worker lease outbox pending selection, publish/ack, failure retry, and dead-letter semantics.
- Task idempotency acquire, in-progress contention, completed-result return, lease expiry, complete, and release semantics.
- Live pending outbox migration requirements from the P012 inventory.
- Crash-window outcomes for publish-before-ack, ack retry, duplicate idempotency keys, and session input during finalization.

The output is a design artifact only; runtime code changes remain later work.

## Acceptance Criteria

- Session dispatch, buffering, attach, suspected-dead observation, session-ended, pending projection, event consumption, and rebuild are mapped to concrete Postgres transaction patterns.
- Outbox pending selection and status updates are mapped to `FOR UPDATE SKIP LOCKED` or another explicit PG-safe rule.
- Idempotency ledger behavior is mapped using unique constraints, transaction predicates, and API-compatible conflict handling.
- Pending live outbox rows from P012 have a clear cutover/replay requirement.
- Crash windows are documented with expected outcomes.
- Ambiguous replay behavior and implementation blockers are listed.

## Risks

- If pending outbox rows are not migrated or deliberately drained, cutover can lose effects.
- If publish/ack crash windows are under-specified, PG workers can duplicate effects unexpectedly.
- If idempotency conflict behavior changes, callers may see failures instead of existing completed results.
- If session finalization and dispatch are not serialized on session key, wake inputs can be lost or attached to the wrong active scope.

## Assumptions

- P015 already owns task/saga/lease claim and recovery concurrency.
- P017 will own JSONB, timestamp, and PG exception/defer policy details.
- Server runtime is Postgres-only after cutover; SQLite remains a migration source, not a runtime fallback.
