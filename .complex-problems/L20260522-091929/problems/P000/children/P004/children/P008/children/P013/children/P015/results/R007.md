# Task Saga Lease Postgres Concurrency Mapping Result

## Summary

Completed the P015 design mapping for Queue task, saga, and worker lease concurrency semantics. The result translates SQLite process-local transaction behavior into Postgres row-lock, uniqueness, and transaction rules without changing production code or data.

## Done

- Mapped task publish, claim, complete, fail/retry, heartbeat, release, stale recovery, and cancel flows to concrete Postgres transaction patterns.
- Mapped saga create, claim, heartbeat, stale recovery, launch, append step result, completion, failure, pending cancel, and cancel-all flows to concrete Postgres transaction patterns.
- Mapped worker lease acquire, heartbeat, release, and timeout behavior, preserving `lease_id` and unique `(machine_type, machine_id)` ownership semantics.
- Defined candidate-selection rules using `FOR UPDATE SKIP LOCKED`, state-row locks, lease-row locks, and unique constraints.
- Documented race outcomes for duplicate claim, completion vs recovery, heartbeat vs recovery, cancel vs claim, lease uniqueness conflicts, and saga step-result vs completion.
- Classified current Python locks as SQLite-era correctness/fairness helpers that must be replaced by database-level rules for PG correctness.

## Verification

- Read and cited source evidence from `queue_service/queue_db.py`, `saga_repo.py`, `lease_ledger.py`, `lease_fsm.py`, `fsm/runner.py`, and `novaic-common/common/db/database.py`.

- Verified the artifact contains concrete sections for task lifecycle mapping, saga lifecycle mapping, race-case outcomes, Python lock classification, and implementation blockers.
- No runtime code, production service, or production database was changed.
