# Map Queue Postgres Semantics

## Problem Definition

Queue migration cannot be implemented safely until the current SQLite correctness semantics are translated into explicit Postgres rules. The risky parts are not the table DDL alone: they are the transactional boundaries around task/saga/session/lease FSM writes, claim/recovery races, outbox replay, idempotency, JSON dependency queries, and SQLite busy/defer behavior.

## Proposed Solution

Produce a semantics mapping document that uses the P012 inventory and local source code as evidence. The document will define, for each queue subsystem, the required Postgres behavior before implementation starts:

- Task FSM writes: publish, claim, complete, fail/retry, heartbeat, release, recover stale, cancel, idempotency acquire/complete/release.
- Saga FSM writes: create, claim, heartbeat, recover stale, launch, append step result, complete, fail, cancel.
- Session FSM writes: dispatch, buffer, attach, suspected-dead recovery, session-ended, event consumption, pending projection, rebuild.
- Worker lease writes: acquire, heartbeat, release, timeout, unique `(machine_type, machine_id)` ownership.
- Outbox behavior: pending selection, publish/ack, failure/dead-letter, and idempotent replay.
- JSON behavior: dependency readiness, agent filtering, context/step result access, and JSONB/index strategy.
- Busy/lock behavior: replace SQLite lock strings and process-local FIFO locks with Postgres transaction isolation, row locks, `FOR UPDATE SKIP LOCKED`, partial unique indexes, retry/defer policy, and optional advisory locks only where row locks are insufficient.

The output should explicitly call out any blockers that must be resolved before code implementation.

## Acceptance Criteria

- Task, saga, session, and worker-lease FSM writes are mapped to concrete Postgres transaction patterns.
- Claim/recovery flows have race-safe candidate selection rules using `FOR UPDATE SKIP LOCKED`, compare-and-update constraints, unique indexes, or advisory locks with clear scope.
- Outbox replay and idempotency behavior preserve current API semantics, including duplicate idempotency keys returning existing results where the SQLite implementation does so today.
- SQLite JSON functions and text timestamp assumptions are translated to Postgres JSONB/time choices.
- SQLite-only assumptions, ambiguous semantics, and implementation blockers are listed clearly.
- The mapping is stored as a durable artifact for P014 cutover design and later implementation tickets.

## Verification Plan

Review the mapping against:

- P012 inventory artifact and live row-count evidence.
- `queue_service/queue_db.py`, `saga_repo.py`, `session_repo.py`, `fsm/sqlite_store.py`, route busy handling, ledger adapters, outbox workers, and `common/db` locks.
- Existing tests that guard projection table ownership, FSM store behavior, outbox behavior, and queue claim/recovery boundaries.

The result must include a criteria map showing which source behaviors have a PG equivalent, and which remain blockers.

## Risks

- Under-specifying claim/recovery races could create duplicate task/saga execution after migration.
- Treating Python FIFO locks as equivalent to database locks would be unsafe once multiple processes use Postgres.
- Changing timestamp types without a conversion rule could break retry, heartbeat, and stale recovery comparisons.
- Migrating pending outbox rows without a replay rule could either lose effects or publish them twice.
- JSONB translation mistakes could break saga dependency readiness or agent-scoped cancellation.

## Assumptions

- Server runtime will be Postgres-only; SQLite remains acceptable only as a one-time migration source or isolated unit-test fixture/fake.
- Queue schema ownership should stay inside the queue service boundary unless a later ticket deliberately centralizes DB adapters in shared common code.
- The current API behavior is the compatibility contract; internal SQL can change if callers observe the same outcomes.
- P014 will own the operational cutover plan; this ticket owns semantic mapping only.
