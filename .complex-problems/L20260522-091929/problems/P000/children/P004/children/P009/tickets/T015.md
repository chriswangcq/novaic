# Design Entangled Postgres Migration Requirements

## Problem Definition

`entangled.db` is still a current state owner. It stores dynamically registered entity tables, schema-driven CRUD data, `entangled_sync_versions`, and append-only transition history used by sync/client behavior. Migrating it to Postgres must preserve dynamic schema registration, JSON/boolean/timestamp serialization, user/key-param scoping, sync-version monotonicity, WebSocket schema/full/delta expectations, and rollback clarity.

## Proposed Solution

Produce a read-only migration requirements package for Entangled before any implementation or production cutover.

1. Inventory the live Entangled SQLite database on the `api` host:
   - file path, owner/mode/size/mtime, open process holders, health/readiness endpoints, and service command line.
   - complete table list, schema DDL, indexes, triggers, and row counts.
   - current `entangled_sync_versions` keys and max/min/version counts.
   - registered entity tables versus internal raw tables such as `subagent_state_transitions`.

2. Map the local code paths that must change:
   - `entangled.sql.database.Database` connection/transaction behavior.
   - `SqlEntityStore.ensure_schema*`, CRUD/list/list_stream/get/create/update/upsert/delete, JSON/boolean/timestamp in/out conversion, filter/order/pagination behavior.
   - `SqlEntityDef` and `FieldDef` DDL/type/index generation.
   - `persistence.ensure_sync_versions_table`, `load_all_sync_versions`, and version bump upsert behavior.
   - raw SQL tables in `state_transitions.py` and route/service startup paths in `app.factory` / `app.state`.

3. Define Postgres requirements:
   - DSN-based database config for `novaic_entangled` and no long-term SQLite fallback in production.
   - adapter boundary that keeps the existing app/store API stable while swapping SQL placeholder, row, transaction, and type behavior.
   - `jsonb` for JSON fields, `boolean` for bool fields, `timestamptz` or explicitly documented text preservation for timestamp fields, and `bigserial`/identity for append-only transition ids.
   - idempotent schema registration with `CREATE TABLE IF NOT EXISTS`, additive column migration, and index creation ordered safely after columns.
   - Postgres-native upserts for sync versions and entity writes.
   - transaction and lock mapping from Python/FIFO SQLite-era locks to Postgres row/table/advisory locks where needed.

4. Define migration and cutover requirements:
   - offline export/import or dual-write-free single-window migration from SQLite to Postgres.
   - pre/post row counts for every table, schema/table/index parity checks, sync-version equality checks, and sample JSON decode checks.
   - WebSocket/client smoke tests: schema push after restart, full sync for representative list/form/stream entities, delta after create/update/delete, and reconnect behavior.
   - rollback boundaries before PG writes, after PG writes with no client-visible deltas, and after client-visible deltas.
   - cleanup criteria for retiring the old SQLite file after a stabilization window.

5. Write one durable artifact summarizing the requirements, blockers, verification matrix, and recommended next implementation tickets.

## Acceptance Criteria

- A durable Entangled Postgres requirements artifact exists under the ledger artifacts directory.
- The live Entangled SQLite schema, indexes, row counts, process holders, health/readiness state, and sync-version summary are captured from the `api` host.
- Entity-store code paths and raw SQL code paths are mapped to specific Postgres requirements.
- Schema registration and `entangled_sync_versions` monotonicity/upsert behavior are documented.
- Sync/client compatibility risks are listed with concrete smoke tests.
- A no-cutover implementation and rollback plan exists with pre/post row and version checks.
- No production Entangled data, schema, config, or runtime mode is changed by this ticket.

## Verification Plan

- Re-run read-only remote inventory commands after writing the artifact and confirm Entangled health/readiness remains unchanged.
- Verify the artifact cites the live table inventory, sync-version summary, and specific code paths inspected.
- Verify the plan contains explicit cutover blockers and rollback boundaries.
- Record the result with no production mutation claim and then run a skeptical success check against the P009 criteria.

## Risks

- Dynamic entity tables may differ from local documentation, so live SQLite inventory must be treated as authoritative.
- SQLite DDL generated by `SqlEntityDef` may hide assumptions that need a proper dialect layer rather than simple string replacement.
- Timestamp fields currently use canonical ISO `Z` strings; converting to `timestamptz` could subtly change wire output unless the out-bound serializer is tested.
- WebSocket clients may depend on current full/delta ordering or schema hash behavior across restart.
- Sync versions must never reset or go backwards, or reconnecting clients may miss updates or rebuild incorrectly.
- After PG writes begin, rollback to SQLite may require either discarding PG-only writes or replaying them back into SQLite; this boundary must be explicit before cutover.

## Assumptions

- The target Postgres database is the existing `novaic_entangled` database in the `novaic-postgres` container on the `api` host.
- Development machines can keep SQLite only for narrow unit-test fakes if needed, but production and integration behavior should target Postgres.
- P009 is requirements and planning only; implementation, data migration, production cutover, and residue cleanup will be separate tickets.
