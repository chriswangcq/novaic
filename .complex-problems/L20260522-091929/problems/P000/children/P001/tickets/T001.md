# Install local Postgres Docker foundation

## Problem Definition

`api` needs a Postgres foundation before any service migrates from SQLite. This ticket must provision infrastructure only; it must not switch application data paths yet.

## Proposed Solution

Create `/opt/novaic/postgres` with a Docker Compose stack using the official Postgres image, a persistent data volume, root-readable secrets, an init script that creates per-service roles/databases, health checks, and backup/restore helper scripts. Bind Postgres to `127.0.0.1:5432` only. Verify container health, database/role creation, local-only listener, backup script output, and existing service health.

## Acceptance Criteria

- `novaic-postgres` container is running and healthy on the `api` host.
- Postgres binds only to `127.0.0.1:5432` and is not exposed on `0.0.0.0`.
- Databases/roles exist for `novaic_llm_factory`, `novaic_entangled`, `novaic_queue`, `novaic_gateway`, and `novaic_cortex`.
- Passwords are in root-readable secret files, not embedded in world-readable compose files.
- Backup helper exists and can produce a dump artifact or dry-run proof.
- Existing NovAIC services and `llm-factory` remain healthy.

## Verification Plan

- Run `docker compose up -d` and inspect container health.
- Run `psql` inside the container to list roles and databases.
- Run `ss -ltnp` to confirm local-only binding.
- Run the backup script and verify a non-empty dump artifact.
- Check `novaic.service`, `nginx`, `docker`, and `novaic-llm-factory` health.

## Risks

- Credentials can leak if written into compose or logs.
- Port 5432 can be accidentally exposed publicly.
- Existing Docker networking can be disturbed if the compose stack is misconfigured.
- Backup scripts can create unbounded disk usage if retention is omitted.

## Assumptions

- Running Postgres before app cutover is safe because no application will point at it yet.
- The `api` host has enough disk for a local Postgres volume and backups.
- Official Postgres Docker image can be pulled for ARM64.
