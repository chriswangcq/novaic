# Migrate Frozen Queue SQLite Snapshot To Production Postgres

## Problem Definition

Production Queue is frozen with a verified SQLite backup at `/opt/novaic/backups/queue-cutover/20260523T011125Z/queue.db`. The Queue data must be migrated into production Postgres database `novaic_queue` and verified before Queue services restart on Postgres.

## Proposed Solution

Inspect the production runtime and migration tooling, prepare the Postgres target with the deployed Queue schema, run the migration from the frozen backup into `novaic_queue`, and save a redacted migration report. Verify source-to-target row mappings and semantic invariants for queue tasks, sagas, FSM projections, outbox/session state, leases, and idempotency before allowing the restart ticket to proceed.

## Acceptance Criteria

- Production runtime code used for migration is known and compatible with the staged/pushed Queue Postgres migration code.
- Migration runs from the verified backup snapshot, not from a live mutable SQLite writer path.
- Production Postgres target state is checked before writes and backed up or proven safe to overwrite/merge.
- Row count comparisons are recorded for all migrated Queue tables.
- Semantic checks cover task state, saga state, FSM projections, outbox/session rows, worker leases, and idempotency semantics.
- Warnings, skipped rows, or target conflicts are either resolved or recorded as blockers.
- A redacted migration report is saved under `.complex-problems/L20260522-091929/artifacts/`.

## Verification Plan

Run migration preflight, schema initialization, source count queries, target count queries, migration execution, target count checks, invariant queries, and redaction scan on generated reports. Do not proceed to service restart if any checksum, count, semantic, or warning blocker remains.

## Risks

- Production Postgres may already contain Queue data from previous attempts and require a controlled backup/truncate or merge decision.
- The production runtime checkout may lag behind the pushed migration fixes and need a safe fast-forward before migration.
- Migration tooling may expose a gap for a table or invariant that was not covered in staging.
- Queue remains frozen until migration and restart are completed or rollback is selected.

## Assumptions

- Operator approval already covers continuing through the cutover without another prompt.
- The verified backup directory `/opt/novaic/backups/queue-cutover/20260523T011125Z` is the source of truth for migration.
- The production Postgres container `novaic-postgres` and database `novaic_queue` are the intended target.
