# Independently Verify Migrated Queue Postgres State

## Problem Definition

The migration report says production Queue data is validated, but the cutover should not proceed to restart until an independent post-migration verification compares the frozen SQLite backup and the live Postgres target for row counts and Queue semantics.

## Proposed Solution

Run a separate verification script from the clean cutover runtime that opens the frozen SQLite backup and production `novaic_queue`, computes all Queue table counts, semantic aggregates, schema version, and selected consistency checks, and saves a redacted verification report. Treat any mismatch or query error as a cutover blocker.

## Acceptance Criteria

- Source SQLite and target Postgres row counts match for all 16 Queue migration tables.
- Schema version is `18` on both source and target.
- Task, saga, session, outbox, worker lease, and idempotency aggregate checks match.
- Verification is run independently after the final migration execution, not only copied from the migration report.
- Verification artifacts are saved locally and remotely and scan clean for credentials.

## Verification Plan

Execute independent source/target count and aggregate queries, compare results, write JSON and Markdown reports, copy artifacts to the ledger, and run redaction scan.

## Risks

- A verification query may reveal a semantic gap not covered by the migration report.
- Postgres type normalization may differ from SQLite string formats and require careful comparison.

## Assumptions

- P133 completed with report status `validated` and no errors.
- Queue production remains frozen, so target state should not change during verification.
