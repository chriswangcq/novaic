# Prepare Runtime And Target For Queue Production Migration

## Problem Definition

The frozen Queue SQLite backup cannot be migrated safely until the production runtime code, migration CLI, Postgres connection path, target database contents, and schema initialization are inspected and prepared without leaking credential values.

## Proposed Solution

Inspect the production Queue runtime checkout and migration scripts, fast-forward only if needed to the pushed cutover commit, locate the approved production Queue Postgres connection path, initialize or verify the Queue schema in `novaic_queue`, inspect target table counts, and make an auditable target backup/clear decision before migration execution.

## Acceptance Criteria

- Runtime code version and git cleanliness are recorded.
- Migration tool presence and CLI usage are recorded.
- Postgres connection path is usable and not exposed in artifacts.
- Existing target row counts are captured before migration.
- Target schema is initialized or verified.
- Any existing target rows are either proven safe, backed up and cleared, or recorded as a blocker.
- A redacted preparation report is saved under ledger artifacts.

## Verification Plan

Run remote git/version probes, migration CLI help/probe, schema inventory/count queries against `novaic_queue`, optional pg_dump backup and controlled truncate when required, then scan the generated report for credential patterns.

## Risks

- Production runtime may have local uncommitted changes that should be preserved instead of overwritten.
- Target Postgres may contain partial Queue data from a previous attempt.
- Connection details must be handled operationally without writing secrets into reports.

## Assumptions

- The validated cutover code is available from the pushed `main` branch in `novaic-agent-runtime`.
- The target database is `novaic_queue` in the existing `novaic-postgres` container.
- Operator approval allows executing these preparation steps without another prompt.
