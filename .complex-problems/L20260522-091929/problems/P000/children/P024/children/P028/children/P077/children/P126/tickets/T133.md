# Smoke Production Queue Postgres Runtime

## Problem Definition

After restart, production Queue Postgres mode needs behavior-level proof. Health/readiness alone is insufficient; safe Queue API, lifecycle, worker process, and outbox checks must confirm the system is operating without falling back to SQLite or entering error loops.

## Proposed Solution

Run a production-safe smoke script against loopback Queue Service and Postgres. Check `/health` and `/ready`, create isolated smoke task/session/saga/idempotency rows through safe Queue APIs or direct repository calls where APIs are not suitable, verify worker/outbox process liveness and recent logs, record post-smoke table counts and outbox histograms, and save redacted artifacts.

## Acceptance Criteria

- Health/readiness pass with `database_backend=postgres`.
- Safe task smoke exercises a representative lifecycle path and records resulting state.
- Safe saga/session/idempotency smoke records representative lifecycle rows or a documented safe equivalent.
- Worker/outbox processes are alive and recent logs show no tracebacks or error loops.
- Post-smoke DB counts and outbox histograms are saved.
- Smoke artifacts are redacted and contain no credential values or credential-file paths.

## Verification Plan

Use a bounded remote smoke script, then copy JSON and Markdown summaries into ledger artifacts and scan for credential patterns. Treat API failures, log tracebacks, SQLite holders, or outbox error loops as blockers.

## Risks

- Production smoke rows can be visible in operational history, so IDs must be clearly prefixed and low-impact.
- Some worker behavior may depend on external services; if unsafe, use direct Queue DB/repository checks and record the rationale.

## Assumptions

- Queue production has already restarted in Postgres mode.
- The smoke may create clearly labeled `codex-smoke` Queue rows as part of validation.
