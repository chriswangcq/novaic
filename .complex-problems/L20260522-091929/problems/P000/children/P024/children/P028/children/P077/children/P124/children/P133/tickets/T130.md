# Execute Frozen Queue SQLite Backup Migration To Production Postgres

## Problem Definition

The production Queue data is still frozen in the verified SQLite backup and must be copied into the prepared empty `novaic_queue` Postgres schema. This is the state-changing migration step before semantic verification and service restart.

## Proposed Solution

Run the pushed migration CLI from the clean cutover runtime `/opt/novaic/services/novaic-agent-runtime-pg` against `/opt/novaic/backups/queue-cutover/20260523T011125Z/queue.db` and the production Queue Postgres credential file. Save the generated redacted migration report under the remote backup directory and local ledger artifacts. Treat any non-validated status, warning, skipped row, or command error as a cutover blocker.

## Acceptance Criteria

- Migration source path is the frozen backup, not `/opt/novaic/data/queue.db`.
- Migration command runs from clean runtime commit `6c464b8b0796b3f1b31729d4d36afe81778a0fd0`.
- CLI exits successfully and report status is `validated`.
- Report records copied row counts for all Queue tables.
- Report contains no unresolved errors or credential values.
- Remote and local migration report artifacts are saved.

## Verification Plan

Run the migration CLI, parse the JSON report status/errors/copied rows, verify the source path and runtime commit, copy the report into ledger artifacts, and run a redaction scan.

## Risks

- Insert failures can leave partial rows if transaction semantics do not roll back as expected.
- JSON payload conversion may still expose a production-only malformed row.
- Long migration runtime could keep Queue frozen longer than expected.

## Assumptions

- P132 prepared an empty target schema and dry-run status `ready`.
- Operator approval remains active for the production freeze window.
