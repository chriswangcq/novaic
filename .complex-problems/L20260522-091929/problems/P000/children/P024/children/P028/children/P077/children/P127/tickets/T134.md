# Archive Old Queue SQLite Active Path After Postgres Cutover

## Problem Definition

After successful Postgres migration, restart, and smoke tests, `/opt/novaic/data/queue.db` must no longer look like an active Queue database path. It should be preserved as rollback evidence but removed from the live path so stale code cannot silently fall back to SQLite.

## Proposed Solution

Verify no process holds the SQLite file, archive `queue.db` and any sidecar WAL/SHM files into the cutover backup directory, leave a clear README/manifest at the archive location, verify the live path is gone, rerun Queue health/ready and lsof, and save a redacted cleanup report.

## Acceptance Criteria

- `lsof /opt/novaic/data/queue.db` shows no holders before archive.
- `/opt/novaic/data/queue.db` and sidecars are moved to a timestamped rollback-only archive under `/opt/novaic/backups/queue-cutover/20260523T011125Z/`.
- Archive manifest records original paths, destination paths, file sizes, and hashes where available.
- Live `/opt/novaic/data/queue.db` path no longer exists after archive.
- Queue `/health` and `/ready` still pass on Postgres after archive.
- Cleanup report is saved and redacted.

## Verification Plan

Run lsof/stat/hash before archive, move files, write manifest, verify old paths absent, run health/ready/lsof after archive, copy reports into ledger artifacts, and scan for credential patterns.

## Risks

- If any process still opens SQLite, moving the file can break a hidden fallback path; this must block cleanup.
- Rollback becomes a deliberate restore action rather than an accidental live-path fallback.

## Assumptions

- Postgres migration, restart, and smoke checks have passed.
- The timestamped backup already contains a verified copy of the SQLite database.
