# Phased Postgres unification with residue cleanup

## Problem Definition

Production state is currently split across multiple SQLite files on the `api` host. Some files are active state owners (`queue.db`, `entangled.db`, `llm_factory.db`), some are small current service-local state (`gateway.db`, `cortex/operational.sqlite3`), and some appear to be residue (`business.db`, empty `device.db` tables). The user wants a unified Postgres direction and explicit cleanup of old residue without leaving future agents with misleading stale paths.

## Proposed Solution

Split the work into safe phases:

1. Provision a local-only Postgres Docker service with persistent storage, per-service databases/users, health checks, and backup tooling, without switching any application yet.
2. Produce an evidence-backed classification of every SQLite file and stale code path: active owner, migrate candidate, local projection, archive/delete residue, or deferred risk.
3. Migrate the lowest-risk application first (`llm-factory`) behind a reversible adapter and data migration, then verify production health.
4. Plan and later migrate `entangled` and `queue` separately, because they own entity sync and FSM/outbox/lease semantics.
5. Clean old residue only after evidence: archive/remove empty DB files, update startup/code comments that imply obsolete ownership, and leave explicit notes for any intentionally retained local projection.

## Acceptance Criteria

- Postgres service exists on `api` with volume, local-only exposure, health check, restart policy, and backup/restore commands.
- Per-service database/user boundaries are created rather than one shared schema for all services.
- SQLite inventory has evidence for each file: file size, tables, row counts, runtime owner, and code references.
- Empty or obsolete residue is archived/removed or documented so it no longer looks like a current state owner.
- No application is cut over without a backup, rollback path, and health verification.
- `queue.db` migration is not attempted until FSM/outbox/lease SQL semantics are explicitly mapped.

## Verification Plan

- Validate ledger state after every action.
- On the `api` host, verify Docker/Postgres container health and local-only port binding.
- Verify created databases/users with `psql` queries.
- Verify backups by creating at least one dump artifact or dry-run command output.
- Verify service health after any runtime change: `novaic.service`, `nginx`, Docker containers, and relevant `/health` endpoints.
- For residue cleanup, verify no live process holds or writes the archived/deleted DB file and that service restart does not recreate misleading residue unless intentionally retained.

## Risks

- Production state loss if SQLite migration is rushed without transactional backups.
- Queue semantics can change if SQLite locking/JSON behavior is naively translated to Postgres.
- Removing empty DB files can be misleading if startup code immediately recreates them.
- Introducing Postgres before code is adapted can create another unused infrastructure component if not tracked to closure.
- Credentials may leak if Docker env files or compose files are world-readable.

## Assumptions

- The `api` host remains the consolidation target.
- Postgres should initially be reachable only from the host and local Docker network.
- `llm-factory` is the first real application migration candidate because it is already Dockerized and has low data volume.
- The immediate safe first execution is infrastructure plus inventory/cleanup planning, not full conversion of `queue.db`.
