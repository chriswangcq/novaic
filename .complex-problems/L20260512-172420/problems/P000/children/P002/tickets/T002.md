# Preserve Explicit Display Media Through Runtime Wrapper

## Problem Definition

The `display` executor returns `_mcp_content` image objects, but the generic runtime wrapper currently treats that dict as an unstructured result and serializes it into `tool-output.v1.text`. This puts image base64 into ordinary tool text and prevents the display-perception path from creating structured multimodal input.

## Proposed Solution

Teach the runtime wrapper to preserve structured `_mcp_content` only for the explicit `display` tool. Keep generic dict results text-based. Then verify that Cortex projection parses display media and runtime multimodal conversion creates `image_url` while sanitizing the tool text.

## Acceptance Criteria

- `_ok(..., tool_name=\"display\", result={\"_mcp_content\": [...]})` returns public `content` with `_mcp_content`, not `tool-output.v1.text`.
- Non-display unstructured dict results remain `tool-output.v1` text.
- The full display path through Cortex projection and runtime multimodal processing emits structured `image_url`.
- The final tool message has placeholder image metadata and no raw base64.

## Verification Plan

- Add a regression test for `_ok` display preservation and the full projection/conversion path.
- Run display tool and multimodal context tests.

## Risks

- Preserving `_mcp_content` too broadly could re-open arbitrary image injection; constrain the preservation to `tool_name == \"display\"`.

## Assumptions

- Current explicit `display` is the only direct LLM-facing visual perception tool.
