# Split Shell Text and Display Media Contracts

## Problem Definition

The current runtime tool-output contract still mixes terminal text, raw shell payloads, artifact/media manifests, and direct display perception. This caused base64 image data to appear as ordinary tool text in LLM requests and also leaves full shell stdout/stderr duplicated in diagnostics. The user wants a tighter final shape: `shell` is a human-terminal preview only, while full replay belongs in Cortex RO steps/payloads and `display` is the explicit visual perception tool.

## Proposed Solution

Split the implementation into child problems:

- Redesign and implement the `shell` output projection as terminal text preview plus bounded diagnostics only.
- Redesign and implement `display` wrapping so structured image content survives runtime wrapping and reaches the existing display-perception multimodal converter.
- Add regression tests for the full real path and update/remove tests that assert the old raw diagnostics behavior.
- Run focused and repository-level checks, then commit and deploy.

## Acceptance Criteria

- `shell` no longer puts full raw stdout/stderr into `tool-output.v1.diagnostics`.
- `shell` still reports exit code, truncation flags, output lengths, and changed-file summary in terminal text/metadata.
- `display` no longer becomes `tool-output.v1.text` containing JSON/base64 after `_ok()`.
- The full current-display path produces structured `image_url` content while the tool text has placeholders/no base64.
- Tests fail on the previous bug shape and pass after the implementation.
- Commit and deploy are completed with documented verification.

## Verification Plan

- Run unit tests for shell output contract, display tool handling, step result projection, and multimodal context processing.
- Add a full-path regression test using `_exec_display`/`_ok` or equivalent wrapper + Cortex parser + runtime multimodal conversion.
- Run the nearest project-wide test matrix or canonical subset if full matrix is too expensive.
- Deploy with the repo deploy script and run smoke/status checks against production services.

## Risks

- Tightening shell diagnostics could break tests or callers that read `raw_shell_result` from the LLM-facing envelope.
- Display media preservation must not reintroduce historical image injection.
- Deployment may need careful smoke checks because runtime/cortex contracts cross service boundaries.

## Assumptions

- Cortex step/payload storage already retains the complete raw shell execution observation outside the LLM-facing preview.
- The only direct visual perception path that should become multimodal input is explicit current `display`.
