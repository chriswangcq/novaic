# Structured public activity titles for Agent Monitor

## Problem Definition

Agent Monitor title projection currently mixes two layers: durable runtime activity records and frontend presentation inference. The frontend reads `record.text` for reasoning records and infers public titles from keywords like `回复`, `新消息`, or `上下文`. This creates incorrect UI-owned step titles that do not correspond to a real runtime event and are hard to reason about, test, or localize.

The desired long-term shape is explicit: runtime activity projection writes public presentation fields for the monitor, while private reasoning text remains detail content. The frontend should render public title fields when present and fall back only to structured phase/status labels. It must not inspect reasoning text to invent titles.

## Proposed Solution

Introduce an explicit `public_title` field on `agent-activity-records` and route monitor row titles through that field. Runtime projection will populate `public_title` for known user-facing records such as message observation, shell read/reply actions, summary/bookkeeping, and generic reasoning. The app entity contract and TypeScript normalization will carry the field. `ActivityTimeline` will prefer `public_title`, use structured fallbacks for legacy records, and remove all reasoning-text keyword title inference.

The implementation should keep reasoning `text` visible as detail text because that is useful agent thinking content, but title generation must no longer depend on that text.

## Acceptance Criteria

- `agent-activity-records` contract/schema includes `public_title` as a public field.
- Runtime `activity_projection` writes `public_title` on emitted records.
- Frontend `ActivityTimelineRecord` and normalization include `public_title`.
- `activityTimelinePublicTitle()` uses `public_title` first, then structured tool/phase/status fallbacks.
- Reasoning text containing `回复`, `新消息`, `上下文`, or `skill_end` does not create title labels like `正在组织回复` unless that exact label is explicitly supplied as `public_title`.
- Existing user-facing read/reply/summary/tool display behavior remains covered by tests.
- Runtime projection tests verify `public_title` is materialized.
- Frontend tests verify the screenshot regression and no keyword-title inference.
- Changes are committed in affected subrepos and the parent repo records the updated gitlinks.

## Verification Plan

- Run focused frontend ActivityTimeline tests.

- Use the ledger success check to map all acceptance criteria to evidence.

## Risks

- Existing records without `public_title` must still render acceptably through structured fallbacks.
- Overloading `public_title` with UI copy could become a localization issue later; this change still improves correctness by making the boundary explicit.
- Backend contract updates may require schema sync in Business or generated contracts if the local system has multiple contract sources.
- Parent repo commit must not accidentally include unrelated submodule or generated artifact churn.

## Assumptions

- `agent-activity-records` is append/upsert tolerant to new fields after contract/schema update.
- Chinese UI labels are currently acceptable as public title values because existing monitor titles are already Chinese.
- The current uncommitted frontend hotfix belongs to this ticket and will be folded into the long-term implementation.
