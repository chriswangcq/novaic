# Consume public_title in ActivityTimeline

## Problem Definition

The frontend must stop inventing monitor row titles from private reasoning text. It needs to consume the new explicit `public_title` field from Entangled activity records and keep only structured fallbacks for legacy records that lack the new field.

## Proposed Solution

Add `public_title` to app-side contracts, entity types, normalized timeline records, and title projection. Make `ActivityTimeline` prefer `record.public_title` when non-empty. Keep existing read/reply low-level shell fallbacks for old records, keep phase/status fallback for reasoning records, and remove all reasoning-text keyword inference. Update tests to cover explicit public titles, legacy fallback, and the screenshot regression.

## Acceptance Criteria

- App entity contract includes `public_title` for `agent-activity-records`.
- `AgentActivityRecordEntity` and `ActivityTimelineRecord` include optional `public_title`.
- `normalizeActivityRecord()` carries `public_title` without raw entity spreading.
- `activityTimelinePublicTitle()` prefers explicit `public_title`.
- Reasoning `text` containing `回复`/`上下文` no longer creates synthetic titles.
- Explicit `public_title="正在组织回复"` can still render that exact title, proving title authority is data, not keyword inference.
- Focused frontend tests pass.

## Verification Plan

- Inspect frontend type/normalization/render tests.
- Patch app contract, types, hook, ActivityTimeline, and tests.
- Run focused ActivityTimeline/hook/contract tests.
- Run focused ESLint on changed source file.

## Risks

- Legacy records without `public_title` still need acceptable fallback.
- Tests that intentionally covered old heuristic behavior must be updated carefully to preserve read/reply public behavior.

## Assumptions

- Existing Entangled rows may omit `public_title`; optional frontend typing and fallback are required.
