# 项目文档重写总方案

## Problem Definition

docs/ 目录 651 个文件经多轮架构迁移积累，结构反映历史路径而非当前架构。需要归档旧文档、从代码重新构建全套文档。

## Proposed Solution

分阶段执行：

1. **归档旧文档**：将 docs/ 整体移到 docs-archive/，保留 git 历史
2. **代码调研**：逐个子模块阅读代码，提取架构事实（目录结构、API 路由、进程角色、依赖关系）
3. **设计新文档结构**：确定文档层级和文件清单
4. **编写文档**：从代码事实出发编写每个文档
5. **适配 CI guards**：确保 lint 脚本在新结构下通过
6. **验证**：交叉检查文档与代码一致性

## Acceptance Criteria

- 旧文档完整归档，不丢失历史
- 新文档基于代码编写，非旧文档复制
- 文档层级清晰（总览 → 服务 → 运维 → 参考）
- 每个服务有独立文档
- 运维手册实用、无历史迁移叙述
- 文档总量 20-40 个文件
- CI guards 通过
- 中文编写

## Verification Plan

1. 验证 docs-archive/ 包含全部旧文件
2. 逐个新文档检查代码引用是否正确
3. 运行所有 docs 相关 CI guard 脚本
4. git diff --stat 确认变更范围合理

## Risks

- 旧文档中可能有运维关键信息（密钥路径、生产配置）需要保留
- CI guard 脚本可能硬编码了旧文档路径
- 部分文档内容（如生产部署步骤）无法纯从代码推断

## Assumptions

- 上一个账本（L20260516-222011）的审计结果可作为参考
- 代码是文档的 ground truth
- 生产运维信息可从旧文档中提取关键部分
