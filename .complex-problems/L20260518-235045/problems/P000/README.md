# 项目文档重写：归档旧文档，从代码重建全套文档

## Problem

当前 docs/ 目录有 651 个 markdown 文件，经历了多次架构迁移后积累了大量历史文档。其中 527 个是 roadmap/tickets 历史工单，其余约 120 个覆盖架构、运维、参考等。这些文档是在多轮迭代中逐步添加的，存在以下问题：

1. 文档结构反映的是历史迁移路径，不是当前架构的自然组织方式
2. 大量文档描述的是"从X迁移到Y"的过程，而非当前系统的工作方式
3. roadmap/tickets 527个工单文件已完成，作为历史记录价值大于作为当前文档的价值
4. 文档之间存在重复、交叉引用混乱、层级不清晰

用户要求：结合上一个账本（L20260516-222011）的审计结果和对代码的重新阅读，从零重写项目文档。旧文档全部归档当做丢弃。

## Success Criteria

- 旧 docs/ 目录完整归档到一个明确的归档位置（如 docs-archive/），不丢失历史
- 新文档从代码实际状态出发编写，不是旧文档的复制粘贴
- 新文档结构清晰：有明确的层级（总览 → 服务详解 → 运维手册 → 参考），读者能快速定位
- 每个服务/子模块有独立的架构文档，内容基于代码目录结构和实际 API/路由
- 运维文档（本地开发、部署、排障）保留实用内容，去除历史迁移叙述
- roadmap/tickets 历史工单归档，不在新文档体系中
- 新文档总量控制在合理范围（预计 20-40 个文件），每个文件有明确职责
- CI guard 脚本（lint_docs_status_consistency.py 等）在新文档结构下仍然通过，或做必要适配
- 新文档使用中文编写，与项目团队语言一致
