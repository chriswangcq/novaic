# 边缘与基础设施服务代码调研方案

## Problem Definition

需要调研项目中 8 个边缘/基础设施子模块的代码结构：novaic-gateway（Rust）、Entangled（novaic-app）、novaic-device、novaic-llm-factory、novaic-blob-service（如存在）、novaic-sandbox-service、novaic-logicalfs、novaic-common。这些服务支撑核心 agent 执行链路但不直接参与 agent 循环。

## Proposed Solution

启动多个并行调研 agent，每个 agent 负责 2-3 个子模块的代码阅读。分三组：
1. Gateway（Rust, 需要读 Cargo.toml + src/）+ Device + LLM Factory
2. Sandbox Service + LogicalFS + Blob Service
3. Common 共享库 + Entangled (novaic-app)

每个 agent 按标准维度调研：入口文件、目录结构、API 路由、核心模块划分、服务间调用关系、配置和启动参数。

## Acceptance Criteria

- gateway：Cargo.toml 依赖、src/ 目录结构、路由定义、auth 中间件、WebSocket handler、blob proxy 路由
- device：入口文件、API 路由、CloudBridge/VmControl 集成方式、设备管理 API
- llm-factory：入口文件、API 路由、provider 路由机制（如何选择 LLM provider）
- sandbox-service：入口文件、API 路由、进程隔离和执行机制
- logicalfs：模块结构、Snapshot/Authority/Store 抽象、与 Cortex workspace 的集成
- common：包结构、ServiceConfig、HTTP client、共享 contracts、工具定义 (llm_builtin)、wake assembler
- entangled (novaic-app)：入口文件、WebSocket 协议、Entity Store 实现
- 每个服务的默认端口和启动命令

## Verification Plan

对照 Success Criteria 逐条检查，确认每个子模块的所有要求维度均有代码直接阅读的事实支撑。

## Risks

- Gateway 是 Rust 代码，文件组织与 Python 服务不同，需要关注 Cargo workspace 结构
- 部分子模块可能代码量较小（如 blob-service、logicalfs），调研可能很快完成
- Entangled 可能嵌入 novaic-app 而非独立子模块，需要确认实际位置

## Assumptions

- 所有子模块代码存在于 mono-repo 的子目录中
- Rust 服务可以通过 Cargo.toml + src/ 目录推断结构
- novaic-app 包含 Entangled 服务的实现
