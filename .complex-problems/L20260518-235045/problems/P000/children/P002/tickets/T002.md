# 代码调研与新文档结构设计

## Problem Definition

需要深入阅读 14 个子模块的代码，提取架构事实，设计全新的文档目录结构。这个调研直接决定后续文档编写的质量。

## Proposed Solution

1. 逐个子模块调研：
   - 入口文件和启动方式
   - 目录结构和模块划分
   - HTTP API 路由（如果是 HTTP 服务）
   - 进程角色和工作流程
   - 与其他服务的依赖关系
2. 整合调研结果，设计新文档结构
3. 产出文件清单，每个文件明确职责

## Acceptance Criteria

- 14 个子模块核心结构摸底完成
- 产出新文档目录结构方案
- 方案与代码实际状态交叉验证
- 层级清晰：总览 → 服务架构 → 运维手册 → 参考

## Verification Plan

1. 每个子模块的调研结果与代码目录对照
2. 文档结构方案覆盖所有子模块
3. 无遗漏的核心组件

## Risks

- 子模块代码量大（agent-runtime 3857 文件、app 4290 文件），需要聚焦核心

## Assumptions

- 聚焦入口文件、路由定义、核心模块即可，不需要逐行阅读
