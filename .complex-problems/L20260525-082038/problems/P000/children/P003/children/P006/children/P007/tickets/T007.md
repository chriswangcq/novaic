# Propagate session_generation into initial react_think

## Problem Definition

Production now dispatches user-message notifications into Queue, but the first `react_think` saga spawned by `subagent_wake` lacks the required `session_generation`. `ReactThinkInput.from_context` fails before context assembly, so the message pipeline stops with `ReactThinkInput.session_generation is required`.

## Proposed Solution

Update `task_queue/sagas/subagent_wake.py::_build_trigger_think_payload` to copy `session_generation` from the wake context using the existing `require_positive_session_generation` helper. Add focused unit coverage proving the initial `react_think` trigger carries the generation and still rejects missing/invalid values. Run the Runtime test slice, commit/push, deploy through Release Controller, then verify production no longer emits the `ReactThinkInput.session_generation is required` failure for a fresh message path.

## Acceptance Criteria

- Initial `react_think` trigger context includes `session_generation`.
- No fallback/default generation is introduced.
- Missing or invalid generation fails fast in the payload builder.
- Runtime regression tests cover the wake-to-think propagation.
- Release Controller deploys the fix to staging and prod.
- Production verification shows the previous `ReactThinkInput.session_generation is required` failure is gone for new messages.

## Verification Plan

Run focused Runtime tests around `subagent_wake`, session generation contracts, and explicit Runtime contracts. After deployment, check prod task-worker logs and queue/saga state for a fresh user-message path.

## Risks

- Existing stale failed tasks may remain in queue stats; verification must focus on post-deploy logs and new tasks.
- The next stage may expose another downstream failure after `react_think` can proceed.

## Assumptions

- `session_generation` is already present in the `subagent_wake` saga context at dispatch time, as proven by successful `session.init` payload construction.
- Failing fast on missing generation remains the desired explicit-boundary behavior.
