## Problem
Cortex currently owns `logical_fs.py`, which mixes Cortex workspace semantics, agent capability env, subagent RW layout, local materialization, mount planning, and RW diff flushing. The desired final shape is a LogicalFS substrate/service that serves sandboxd with mountable filesystem views while treating Cortex as a state source through explicit snapshot and patch contracts.

## Success Criteria
- LogicalFS core/service contracts are snapshot/view/patch based and do not import Cortex, agent runtime, agentctl, subagent, or sandbox core execution substrate.
- Cortex produces `WorkspaceSnapshot` and applies `WorkspacePatch`; it does not materialize process filesystems itself.
- Sandbox execution path uses LogicalFS as the view provider and sandboxd as the process executor.
- Agent/capability env and RW layout are explicit inputs from Cortex/runtime, not generated by LogicalFS from hidden state.
- Old Cortex-owned LogicalFS implementation is removed or reduced to a thin adapter with no materialization/diff logic.
- Tests, residue scans, and deployment wiring prove the active path is the new LogicalFS boundary.
