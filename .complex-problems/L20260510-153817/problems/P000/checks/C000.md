# Cortex incremental context assembly research check

## Summary

The research problem is satisfied: current implementation was inspected from source, mutation points were mapped, six design options were compared, and a phased recommendation was recorded.

## Evidence

- Source pointers inspected: Runtime prepare handler, context read/append handlers, CortexBridge prepare path, Cortex API prepare/append/step/scope endpoints, Workspace mutation methods, ContextEngine and StepTreeBuilder.
- Result R000 documents current call-time assembly, mutation points, six options, pros/cons, and recommended phased direction.

## Criteria Map

- Identify current call-time assembly path -> covered under "Current Implementation" in R000.

- Provide multiple design options -> six options listed.
- Compare correctness/latency/complexity/recovery/fit -> each option includes pros, cons, and best use.
- Recommend direction -> phased recommendation selects folded-prefix projection plus active-path on-demand, with DFS fallback.

## Execution Map

- T000 -> completed via source inspection and design analysis, recorded as R000.

## Stress Test

- Stale projection failure mode -> recommendation requires generation checks and DFS fallback.
- Notification boundary failure mode -> recommendation preserves notification-as-hint semantics; no raw IM bodies are introduced into projections.
- Projection worker lag failure mode -> async option requires wait/fallback/repair handling rather than silent stale reads.
- Over-eager write-through failure mode -> synchronous write-through is explicitly not recommended as primary path.

## Residual Risk

- This is design research only; implementation tickets would still need detailed schemas for generation metadata, projection files, repair paths, and tests.
