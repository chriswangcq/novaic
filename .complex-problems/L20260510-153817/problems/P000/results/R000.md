# Cortex incremental context assembly research result

## Summary

Researched the current novaic-cortex LLM context assembly implementation and compared several ways to move work off the LLM call path or make assembly incremental. Recommendation: do not jump directly to full synchronous write-through context; use a durable projection model, with a practical first phase of historical folded-prefix projection plus active-path on-demand expansion.

## Current Implementation

### Runtime call path

- Runtime handler `handle_cortex_prepare_llm_context` first calls `handle_context_read`, then calls `bridge.prepare_for_llm`, then combines Cortex messages with tool schemas.
- `handle_context_read` currently appends Environment notification system hints into the wake `context.jsonl` if `meta.input_message_ids` contains unseen ids. This means LLM context preparation is not purely read-only today; it can append notification hints before preparing.
- `CortexBridge.prepare_for_llm` posts to `/v1/context/prepare_for_llm`.

### Cortex prepare path

- `/v1/context/prepare_for_llm` constructs `ContextEngine` from the agent-root scope path.
- `ContextEngine.prepare_messages_for_llm()` reads root `context.jsonl`, builds a DFS `StepTree` from `steps/_index.jsonl`, merges context/tool/scope data, hoists primary system prompts, and budget-compacts returned messages.
- `StepTreeBuilder` recursively reads child `meta.json`, `context.jsonl`, `summary.md`, `steps/_index.jsonl`, and tool step JSON as needed.

### Mutation points

- `create_scope(parent_path=...)` appends a scope entry to parent `steps/_index.jsonl`, writes child `meta.json`, and initializes child `steps/_index.jsonl`.
- `complete_child_scope()` writes `summary.md` and archives the scope.
- `append_context()` / `append_context_batch()` append messages to `context.jsonl`.
- `write_step()` writes step JSON, possibly payload files/blob payload records, then appends to `steps/_index.jsonl`.
- `append_input_message_ids()` mutates root `meta.input_message_ids`, which later drives notification hint insertion.

## Option 0: Keep Current On-Demand DFS Assembly

Description: Keep rebuilding the prompt from source files when Runtime asks for LLM context.

Pros:

- Very correct and simple: workspace files are source of truth.
- Crash recovery is natural: no projection to repair.
- No stale cache class of bugs.
- Fits current tests and mental model.

Cons:

- Prepare latency grows with historical tree size.
- Re-reads mostly unchanged old scopes on every LLM round.
- `handle_context_read` side effect means prepare is already not a pure read path.
- Expensive reads happen at the most latency-sensitive time.

Best use:

- Baseline fallback and correctness oracle. Should remain as a rebuild/repair path even if projections are added.

## Option 1: Call-Time Memoization With Generation/Watermarks

Description: Cache the prepared context result and reuse it when a root-level generation has not changed.

Pros:

- Minimal conceptual change.
- Easy rollback: cache miss falls back to current DFS assembly.
- Good if many repeated prepares happen without intervening writes.

Cons:

- Need precise invalidation over all mutation points: context append, step write, scope begin/end, input ids, engine config.
- Computing a reliable generation may itself require tree reads unless mutation endpoints maintain it.
- Does not solve first prepare after a mutation.

Best use:

- Safe short-term optimization after adding an explicit `context_generation` bump in all mutation endpoints.

## Option 2: Synchronous Write-Through Materialized Context

Description: Every workspace mutation immediately updates a `prepared_context` materialized view under the same lock.

Pros:

- Prepare becomes very fast: read projection plus maybe tool schemas.
- Strong consistency if mutation and projection are atomic.

Cons:

- Fold behavior is nonlocal: `skill_end` must replace or suppress a range of previously expanded messages.
- Tool result insertion depends on assistant `tool_calls`, so step writes may need to patch earlier assistant turns.
- Any projection bug blocks core writes or creates silent prompt corruption.
- Write amplification is high, and this pushes complex LLM context semantics into every mutation path.

Best use:

- Not recommended as first implementation. It is too easy to create hidden stale state.

## Option 3: Async Durable Projection Worker

Description: Mutation endpoints append `ContextSourceEvent` records to a durable outbox/ledger. A projector maintains a materialized `prepared_context` projection. Prepare checks projection generation and either reads it, waits briefly, or falls back to DFS rebuild/repair.

Pros:

- Matches the broader FSM/saga/outbox direction.
- Externalizes projection failure; replay can rebuild from source events.
- Moves heavy work off the LLM call path.
- Can expose projector lag, last applied generation, and repair status as metrics.

Cons:

- More infrastructure: event schema, projector worker, leases, replay, generation checks.
- Event ordering must be exact per agent-root scope.
- Prepare must handle eventual consistency safely.

Best use:

- Long-term robust architecture if context assembly becomes a real hot path.

## Option 4: Hybrid Historical Folded-Prefix Projection + Active-Path On-Demand

Description: Materialize only the stable historical prefix: closed wake/skill summaries and structural folds. At prepare time, read this prefix and assemble only the active open path on demand.

Pros:

- Removes the expensive historical DFS scan without trying to incrementally maintain live, mutable open-scope internals.
- Closed scopes are stable by definition, so projection invalidation is much simpler.
- Keeps active agent state fresh and source-derived.
- Fits Cortex semantics: completed skills become durable summaries; active wake remains live context.

Cons:

- Prepare still does some work for active path and recent open scopes.
- Needs careful repair if a wake is auto-archived or recovered.
- Needs projection generation metadata and source fallback.

Best use:

- Recommended first real optimization. It captures most expected savings with much lower semantic risk.

## Option 5: Prewarm Next Context After Mutations

Description: Keep current DFS builder, but schedule a background/prewarm prepare after context/step/scope mutations. The LLM call reads a warmed snapshot if generation matches.

Pros:

- Low risk because the builder stays the same.
- Easy to adopt incrementally.
- Good UX: moves latency out of the user-visible LLM request when idle time exists.

Cons:

- Can waste work during rapid tool loops.
- Still needs generation matching.
- Does not reduce total compute unless paired with caching/projection.

Best use:

- Good companion to Option 1 or Option 4.

## Option 6: Full Event-Sourced Context Source Model

Description: Make context assembly source itself an append-only event stream rather than filesystem DFS files. Files become one projection; prepared LLM context becomes another projection.

Pros:

- Most mathematically clean.
- Perfect fit with durable outbox and replay.
- Clear separation: source events vs derived views.

Cons:

- Largest migration.
- Requires replacing or wrapping several current workspace file assumptions.
- Easy to overbuild before proven bottlenecks.

Best use:

- Final architecture if Cortex workspace state moves broadly toward event sourcing.

## Recommendation

Use a phased design:

1. Keep current DFS assembly as the correctness oracle and fallback.
2. Add explicit per-root `context_generation` / `projection_generation` watermarks on every mutation point.
3. Implement Option 4 first: historical folded-prefix projection for closed scopes, active path assembled on demand.
4. Add Option 5 prewarm after mutations if prepare latency is still visible.
5. If projection lag/repair patterns become important, evolve Option 4 into Option 3 async durable projector.

Avoid Option 2 as the main path. It looks simple but makes each write endpoint responsible for complex future prompt semantics.

## Key Design Rules

- Prepared context must be a projection, not the source of truth.
- Source of truth remains Cortex workspace scope tree or a future event ledger.
- Projection reads must be generation-checked.
- Projection failure must never silently produce empty or stale prompt context.
- Environment notifications remain hints, not message bodies. Moving assembly earlier must not violate the “agent must actively read IM” boundary.
- Recovery/finalize must be able to rebuild projection from source.
