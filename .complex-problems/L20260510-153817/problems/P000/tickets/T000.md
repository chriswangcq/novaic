# Research incremental Cortex context assembly options

## Problem Definition

The user wants a grounded analysis of whether Cortex LLM context assembly can move from full call-time DFS assembly to a non-call-time or incremental model, based on the current code rather than speculation.

## Proposed Solution

- Inspect the active prepare chain in Cortex and Runtime.
- Inspect workspace mutation points: append context, write steps, create/complete scopes, notification merge.
- Build a current-state model of when data changes and when prepared context is consumed.
- Design multiple alternatives: keep on-demand, memoized snapshot, synchronous incremental materialization, async outbox projection, and hybrid active-path cache.
- Compare options by correctness, latency, complexity, recovery, and boundary clarity.

## Acceptance Criteria

- The current implementation is described with exact code pointers.
- At least four viable options are compared with pros and cons.
- The recommendation distinguishes near-term safe improvement from long-term “perfect” architecture.
- No code implementation is performed in this ticket.

## Verification Plan

- Use source inspection with `rg`, `sed`, and line-numbered snippets.
- Confirm no code files are changed by the research ticket.
- Record the final recommendation in the ledger result and summarize to the user.

## Risks

- Incremental context can create stale snapshots unless invalidation/generation rules are explicit.
- Moving work off the LLM call path can hide failures unless projection state is observable.

## Assumptions

- Agent-root DFS remains the semantic source of truth unless deliberately replaced by an event/projection model.
- The user is asking for design research, not immediate implementation.
