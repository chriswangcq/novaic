# Diagnose and repair cross-network WebRTC media path

## Problem Definition

Cross-network host desktop WebRTC sessions can reach a connected state but show a black screen. The failure may be in ICE/TURN advertisement, candidate routing, browser receiver behavior, keyframe/media pump startup, or production coturn exposure. The repair needs to be correct for remote NAT use while preserving same-LAN sessions.

## Proposed Solution

Trace the end-to-end WebRTC path from gateway-provided ICE server configuration through frontend offer creation, Rust peer setup, candidate trickle, media track setup, keyframe delivery, and coturn/container deployment. Identify the deterministic reason remote sessions connect without usable media. Patch the narrow broken layer, add guards/tests where feasible, then verify with code checks and local/remote configuration evidence.

## Acceptance Criteria

- ICE server configuration is traced from production environment to frontend/Rust peer use.
- Remote/non-LAN candidate failure mode is identified with concrete code or deployment evidence.
- Code or deployment configuration is fixed so relay-capable sessions do not silently black-screen after connection.
- Same-LAN behavior remains compatible.
- Focused tests, type checks, or Rust checks cover the touched code path.
- Any remaining deployment or network requirement is explicitly recorded.

## Verification Plan

- Inspect app frontend WebRTC offer creation and Rust peer/candidate handling.
- Inspect backend/gateway ICE server APIs and production compose/coturn environment.
- Run targeted unit/type/Rust checks for changed layers.
- Use local logs/config queries to confirm the repaired path advertises TURN/relay where needed.
- Record residual risk if a real off-LAN media test cannot be fully simulated locally.

## Risks

- WebRTC failures can be environment-sensitive; code checks may not prove all NAT combinations.
- TURN credentials or public port exposure may require production configuration changes as well as code changes.
- Aggressive ICE policy changes could break same-LAN performance if not scoped carefully.

## Assumptions

- The intended long-term path is direct WebRTC with TURN relay fallback, not a separate video proxy service.
- Nginx remains ingress-only; coturn should be exposed as the media relay layer.
- The current production machine can host coturn or already has the host-infra container intended for this role.
