# Harden display image LLM injection

## Problem Definition

Current-round `display` tool output can be visually valid in Cortex but still fail to reach the model as provider-native image input if any boundary formats, sanitizes, or logs it incorrectly. The observed failure mode was a `display` tool result becoming text-only `OK` in the Factory request body, with no extra `user(image_url)` message.

## Proposed Solution

Add an explicit end-to-end contract across the Runtime preparation path:

- Keep the Cortex projection contract strict for `display_perception`: image payloads must remain image blocks, while missing image data should be visible as a diagnostic text receipt instead of an ambiguous `OK`.
- Add Runtime tests that start from LLM input messages with a display `step_ref`, run `prepare_llm_call`, and assert the final messages include a provider-native image message even when a `system` message follows the tool result.
- Add Factory log snapshot redaction so raw base64 is not dumped into logs/UI, while preserving a clear inspectable marker that an image was sent.
- Run focused Runtime/Cortex/Factory tests and inspect diffs for legacy or misleading fallbacks.

## Acceptance Criteria

- Runtime `prepare_llm_call` converts a current-round `display` step-ref image into a `user` message with `image_url`.
- Runtime preserves the image injection when a `system` message follows the display tool result.
- Factory request logging redacts image data into metadata rather than storing raw base64.
- Cortex display projection no longer hides empty display projection as a plain `OK` without diagnostics.
- Focused tests pass in the touched repos.

## Verification Plan

- Add/adjust unit tests for Runtime full preparation path.
- Add/adjust unit tests for Factory log snapshot redaction.
- Add/adjust Cortex projection tests for display projection fallback clarity.
- Run the focused test files.
- Run `git diff --stat` and review the touched diffs.

## Risks

- Provider request format must still receive real image data; redaction must apply only to logs, not transport.
- Over-tightening the display projection fallback could break text-only `display` outputs, so the test should distinguish image-capable display from normal text.

## Assumptions

- `display` is the only direct tool that should produce current-turn visual input.
- Shell remains terminal-text only and should not directly inject images.
- It is acceptable for Factory logs to show placeholders for image data instead of raw base64.
