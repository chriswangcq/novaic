# Make Release Controller CI/CD explicitly triggered

## Problem Definition
The Release Controller is the centralized backend/factory CI/CD service, but the live service still has autonomous branch polling enabled. In an agent-driven workflow, a push should not implicitly start staging deployment; the release action should be explicit and auditable through the controller API.

## Proposed Solution
Patch the live API-host Release Controller config to disable autonomous polling, restart the service, update architecture/runbook docs to make `/v1/triggers` the normal staging entrypoint, and add repository guards that reject docs/config drifting back to autonomous polling as the normal backend/factory path.

## Acceptance Criteria
- Live `/v1/status` reports polling disabled and not running.
- Repository docs state staging backend/factory CI/CD starts from explicit `/v1/triggers`, not branch polling.
- Sample config keeps `polling_enabled=false`.
- CI guard tests cover the manual-trigger release contract.
- Focused Release Controller tests pass.

## Verification Plan
Use the live status API before and after restart, run focused pytest guards/unit tests, commit and push the doc/guard changes, then verify that the push does not create a new automatic release run.
