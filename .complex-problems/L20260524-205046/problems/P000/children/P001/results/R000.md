# Persisted execution result model implemented

## Summary

Implemented `PlanExecutionResult` as a serializable release-controller model and added optional `execution_result` persistence to `ReleaseRun`. Finalized runs now store the command execution result written by `execute_planned_release`, so run fetch/list/status responses expose durable per-step execution details for new runs.

## Done

- Moved `PlanExecutionResult` into the model layer with `from_mapping()` and `to_mapping()` while keeping `release_controller.runner.PlanExecutionResult` imports compatible.
- Added `ReleaseRun.execution_result` with backward-compatible read behavior for old run JSON that lacks the field.
- Updated `execute_planned_release()` to persist success, dry-run, and failed execution results when writing the final run record.
- Extended release-controller tests for model round-trip, state round-trip, API persisted run/list/status output, poller persistence, dry-run skipped results, and failed partial execution persistence.
