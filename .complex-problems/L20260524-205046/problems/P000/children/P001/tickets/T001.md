# Implement persisted execution result model

## Problem Definition

`PlanExecutionResult` is currently an in-memory runner result and is returned by immediate controller responses, but completed `ReleaseRun` records do not store it. Historical run APIs therefore lose command-level stdout, stderr, exit codes, skipped flags, and partial failure context.

## Proposed Solution

Promote `PlanExecutionResult` into the serializable model layer beside `CommandResult`, preserving the existing runner import surface. Add optional `execution_result` to `ReleaseRun`, parse it when present, omit it for old records, and write it during finalization in `execute_planned_release`. Extend unit tests so serialization, state persistence, service endpoints, poller-triggered runs, dry-runs, and failed partial execution all prove the persisted field works.

## Acceptance Criteria

- `PlanExecutionResult` has explicit `to_mapping()` and `from_mapping()` behavior using `CommandResult` for nested command records.
- Existing imports from `release_controller.runner import PlanExecutionResult` remain valid.
- `ReleaseRun` accepts old JSON without `execution_result` and new JSON with it.
- `execute_planned_release()` persists the execution result for success, dry-run, and failure.
- Run-returning endpoints expose `execution_result` because it is part of persisted `ReleaseRun` mappings.
- Tests cover model serialization, state round-trip, service persisted fetch/list, poller persistence, failed partial result persistence, and dry-run skipped result persistence.

## Verification Plan

Run the release-controller test suite, targeted model/state/service/poller tests, release-controller CI guard tests, root tests if practical, and syntax checks for touched Python modules.

## Risks

- Captured stdout/stderr can grow run JSON size; this ticket only preserves existing controller output behavior and does not add redaction or truncation.
- Moving the model can break imports if `runner.py` and `__init__.py` are not kept compatible.
- Old records must not fail deserialization when the new field is absent.

## Assumptions

- Persisting the same execution payload already returned by the immediate API is acceptable for this step.
- Deployment verification is handled by the separate deployment ticket under the root problem.
