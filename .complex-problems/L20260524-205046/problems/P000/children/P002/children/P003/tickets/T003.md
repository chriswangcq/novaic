# Pause polling and publish the persistence commit

## Problem Definition

The persistence implementation must reach `origin/main`, but the currently deployed controller still lacks run-record execution persistence. If polling remains enabled, it may process the new branch head before the upgraded controller image is deployed.

## Proposed Solution

Capture remote controller status and release pointers, disable `polling_enabled` in the remote controller config, restart/verify the controller is healthy with polling disabled, stage only the intended release-controller persistence files, commit them, push to `origin/main`, and re-check remote status remains healthy and paused.

## Acceptance Criteria

- Remote `/v1/status` before the change is saved or summarized with current branch head and current release pointers.
- Remote config has polling disabled and `/v1/status` confirms polling is disabled before pushing.
- The local commit contains only release-controller persistence source/test files, not unrelated dirty files or problem-ledger artifacts.
- The commit is pushed to `origin/main` and the commit hash is recorded.
- Remote controller remains healthy and polling disabled after the push.

## Verification Plan

Use `curl`/SSH to inspect remote status/config, `git status` and `git diff --cached` to confirm the staged set, then `git commit`, `git push origin main`, and remote `/health` plus `/v1/status` checks after push.

## Risks

- Remote config editing can accidentally drop quality gates or branch rules if rewritten incorrectly.
- Existing unrelated dirty files in the local workspace must not be staged.
- If polling cannot be paused, pushing must stop until the controller is safe.

## Assumptions

- Remote controller config is at `/opt/novaic/release-controller/config.json` and container restart uses the existing Docker deployment on API host.
