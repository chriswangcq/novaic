# Deploy persisted-result controller image and verify remote record

## Problem Definition

The persistence commit is pushed and remote polling is paused, but the API-host controller still runs the old image. The host must run a new controller image built from the persistence commit, and remote run inspection must prove persisted execution results are available after the response.

## Proposed Solution

Build the release-controller Docker image on the API host from commit `745e20cbaefdd12139c344e8838d9e3e0e782ba2`, push it to the local registry, deploy it by digest with `deploy release-controller-image`, then execute a safe remote verification run. Prefer a dry-run trigger first to prove persisted `execution_result.results`; then restore polling and run or observe a controlled poll if needed so branch heads advance under the upgraded controller. Finish by checking health, polling `last_error`, and prod/staging pointers.

## Acceptance Criteria

- Remote worktree is checked out at `745e20cbaefdd12139c344e8838d9e3e0e782ba2` for image build.
- A new controller image digest is pushed to `127.0.0.1:5000/novaic/release-controller`.
- `deploy release-controller-image <digest>` succeeds and remote `/health` is healthy.
- A remote run fetched from `/v1/runs/{run_id}` includes persisted non-empty `execution_result.results`.
- Polling is restored with `/v1/status` reporting `last_error=null`.
- Prod and staging pointers are recorded after the deployment; any staging movement to the new commit is intentional and visible.

## Verification Plan

Run remote git checkout/build/push/image-inspect, local deploy command, remote controller health/status checks, remote dry-run or poll trigger, explicit persisted run fetch, and final status pointer/polling inspection.

## Risks

- Building on the API host can take time and may fail if the worktree or Docker cache is inconsistent.
- A real poll will deploy staging to the new commit; if this happens, it must be verified and recorded.
- Restoring polling before verification would reintroduce ambiguity about which controller processed the commit.

## Assumptions

- Commit `745e20cbaefdd12139c344e8838d9e3e0e782ba2` is available from `origin/main` on the API host.
- The existing deploy script supports `release-controller-image` with an immutable digest.
