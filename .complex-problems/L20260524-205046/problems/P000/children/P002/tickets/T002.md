# Deploy execution-result persistence to API-host controller

## Problem Definition

The release-controller persistence change is only local until it is committed, pushed, built into an immutable controller image, deployed to the API host, and verified against the remote controller state API.

## Proposed Solution

Pause remote Release Controller polling, commit and push only the release-controller persistence code, build and push a new `novaic/release-controller` image on the API host from that commit, deploy that image by digest through the existing controller-image deploy path, run a safe remote verification trigger, inspect `/v1/runs/{run_id}` for persisted `execution_result.results`, then restore polling and verify status has `last_error=null` and release pointers remain intended.

## Acceptance Criteria

- Remote polling is disabled before the new persistence commit can be auto-processed.
- The source commit contains only intended release-controller persistence files, not unrelated dirty workspace residue.
- The deployed controller image is built from the persistence commit and addressed by immutable digest.
- Remote verification creates or observes a run whose persisted `/v1/runs/{run_id}` response includes `execution_result.results`.
- Remote polling is re-enabled and reports healthy status.
- Current prod/staging release pointers are checked before and after deployment.

## Verification Plan

Use local tests/guards from P001 as source confidence, then run remote controller health/status checks, image digest inspection, deployment status checks, and a safe dry-run or poll run whose persisted run record is fetched after completion.

## Risks

- If polling is not paused, the old controller may process the new commit before the new image is deployed.
- A real poll can update staging to the new commit; this is acceptable only if it is explicit and verified.
- Remote config edits must preserve existing branch rules, quality gates, and release pointers.

## Assumptions

- SSH access to the API host and the local registry are available as in prior deployment work.
- The existing `deploy release-controller-image <digest>` path is the intended deployment mechanism for controller image updates.
