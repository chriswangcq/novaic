# Add persisted execution results to ReleaseRun

## Problem Definition

Release Controller has enough execution data in memory but does not store it in run records. This makes historical run inspection and CI gate auditing weaker than the live response.

## Proposed Solution

Move or expose `PlanExecutionResult` as a serializable model alongside `CommandResult`, add `execution_result` to `ReleaseRun`, update `execute_planned_release` to store it when finalizing a run, and extend tests for model round-trip, state persistence, service endpoints, poller runs, failed runs, and dry-run results. Then commit/push the change, upgrade the API-host controller image, and verify a dry-run or safe run persists execution results in `/v1/runs/{run_id}`.

## Acceptance Criteria

- `ReleaseRun.to_mapping()` includes `execution_result` when present.
- `ReleaseRun.from_mapping()` accepts old records without `execution_result` and new records with it.
- `execute_planned_release()` writes final runs with execution results for success, failure, and dry-run.
- API endpoints that return runs expose `execution_result` from persisted state.
- Tests cover success, failure, dry-run, and backward-compatible old records.
- API-host Release Controller is upgraded and a verification run proves persisted execution results are visible after the immediate response.

## Verification Plan

Run release-controller tests, release-controller CI guard tests, root pytest, deploy syntax checks, local model/state/service/poller tests, remote controller upgrade, and remote `/v1/runs/{run_id}` inspection for `execution_result`.

## Risks

- Persisted stdout/stderr can make run JSON files larger or expose sensitive output if future commands print secrets.
- Moving `PlanExecutionResult` between modules can break imports if not re-exported carefully.
- Old run records must remain readable.

## Assumptions

- Persisting the current captured stdout/stderr fields is acceptable for now because the controller already returns them in immediate API responses.
- A future hardening task can add output size limits or secret redaction if needed.
