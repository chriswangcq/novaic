# Delete migrated server SQLite code paths

## Problem Definition

Postgres cutover is complete for server persistence, but current code still contains SQLite runtime switches, server database file arguments, migration/admin CLIs, and tests that assert SQLite compatibility. These paths are misleading because they look executable after the cutover.

## Proposed Solution

Split the cleanup by active service boundary. First remove production startup and runtime selector residue. Then retire old migration/admin scripts and update tests/residue guards so Postgres-only server persistence is the current path. Keep only references that are clearly client-local cache or historical archive, and label any remaining low-level SQLite substrate as a separately closed follow-up if it cannot be deleted without replacing a still-active abstraction.

## Acceptance Criteria

- Startup scripts no longer pass server SQLite file paths for migrated services.
- Current server runtime entry points reject or omit SQLite backend selection for migrated services.
- SQLite migration/admin utilities for old server DB files are deleted from current executable paths.
- Tests no longer encode SQLite fallback as desired server behavior.
- Remaining SQLite references are enumerated and justified by category.

## Verification Plan

- Run targeted ripgrep residue scans across current service directories.
- Run focused unit tests for changed service boundaries.
- Run ledger success check with evidence from diffs, scans, and tests.
- Commit and push each touched submodule plus the root pointer/ledger changes.

## Risks

- Some low-level SQLite adapters may still be used as generic test stores and require a separate Postgres-backed replacement rather than blind deletion.
- Existing unrelated root worktree changes must be preserved while editing shared scripts.
- Production runtime may rely on env/secret files not present locally, so local verification may be limited to import/unit/residue checks unless remote deploy smoke is performed.

## Assumptions

- Server-side production persistence is Postgres-only after the migration.
- Client-local SQLite caches are out of scope for this cleanup pass.
- Historical docs archives can remain if they are not imported, executed, or referenced by current runbooks.
