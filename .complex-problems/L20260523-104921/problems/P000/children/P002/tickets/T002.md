# Make Queue Service Postgres-only at runtime

## Problem Definition

Queue Service still carries active-looking SQLite runtime fallback: CLI `--db-backend sqlite`, env backend switches, queue.db construction, SQLite migration CLIs, and SQLite lock boundary helpers. These make the old queue.db path look supported after the Postgres cutover.

## Proposed Solution

Remove the Queue runtime backend switch and make Postgres the only server runtime database. Delete old SQLite-to-Postgres migration modules/tests from current executable paths. Replace worker DB assembly with explicit Postgres DSN inputs, and update the supervised worker roster/startup commands for outbox workers that open the Queue DB directly.

## Acceptance Criteria

- `main_novaic.py queue-service` no longer accepts `--db-backend sqlite` or sets `NOVAIC_QUEUE_DB_BACKEND`.
- `queue_service/main.py` no longer constructs `queue.db` or reports SQLite readiness.
- `task_queue/workers/assembly_factories.py` no longer reads `NOVAIC_QUEUE_DB_BACKEND` or constructs `queue.db`.
- Old Queue SQLite migration CLI/modules/tests are deleted from current paths.
- Focused Queue Postgres tests and residue scans pass.

## Verification Plan

- Run targeted Queue runtime tests after updating expectations.
- Run `python -m py_compile` on changed Queue modules.
- Run `rg` for `NOVAIC_QUEUE_DB_BACKEND`, `queue.db`, `migrate_sqlite_to_postgres`, and `sqlite_boundary` in Queue runtime paths.

## Risks

- Many older tests use `common.db.Database(tmp_path / queue.sqlite3)` as a cheap unit-test substrate; removing all of that in one step may require a larger Postgres test fixture refactor.
- Some repository SQL helpers still generate SQLite dialect branches for old tests; this ticket focuses on server runtime fallback, not every low-level historical dialect branch.

## Assumptions

- Queue server production and integration-test environments have `/opt/novaic/postgres/secrets/novaic_queue_dsn` or an equivalent DSN file.
