# Queue runtime SQLite fallback removed

## Summary

Executed the Queue runtime cleanup slice. Queue server startup and direct-DB outbox worker assembly are now Postgres-only, old queue SQLite migration code is deleted, and the dedicated SQLite lock/busy compatibility boundary is gone from runtime paths.

## Done

- Removed `--db-backend` and `NOVAIC_QUEUE_DB_BACKEND` from `main_novaic.py queue-service`.
- Removed `queue.db` path construction, SQLite readiness reporting, and SQLite schema fallback from `queue_service/main.py`.
- Simplified `queue_service.db.create_queue_database()` to return `QueuePostgresDatabase` only.
- Removed Queue SQLite migration modules: `queue_service/db/migration.py` and `queue_service/db/migrate_sqlite_to_postgres.py`.
- Removed `queue_service/db/sqlite_boundary.py` and the `sqlite_lock_timeout_kwargs` call sites in `queue_service/queue_db.py` and `queue_service/saga_repo.py`.
- Removed SQLite busy classification from `queue_service/transient_errors.py`.
- Added explicit `--postgres-dsn-file` options for the session/saga outbox workers and updated `runtime_roster.py` so supervised worker commands pass `$QUEUE_POSTGRES_DSN_FILE`.
- Deleted the Queue SQLite migration test files and updated focused tests to assert Postgres-only runtime behavior.

## Known Gaps

- `queue_service/fsm/sqlite_store.py` remains as a generic FSM store used by task/saga/session/lease ledgers and many tests. It is no longer a server runtime backend selector, but it is still misleading by name and implementation and should be handled by a separate follow-up or the residue guard child problem.
- Some low-level SQL helpers still branch on a `backend_name` default of `sqlite` for old unit-test fixtures; the runtime entry points no longer expose this path.
- Full-suite verification is blocked by unrelated existing failures: one collection error for `_exec_payload_qa`, missing root docs expected by PR-340 tests, `main_task.py` residue guard, and other pre-existing fixture expectations.
