# Startup entry points moved to Postgres launch args

## Summary

The current startup entry point no longer launches migrated server services through old SQLite database files. The script now names the Postgres DSN secret files explicitly and passes Postgres launch flags for Entangled, Gateway, Queue Service, and Cortex.

## Done

- Added `/opt/novaic/postgres/secrets` DSN variables in `scripts/start.sh`.
- Replaced Entangled `--db-path /opt/novaic/data/entangled.db` with `--db-backend postgres`, `--postgres-dsn-file`, and `--service-token-file`.
- Added Gateway `--db-backend postgres --postgres-dsn-file`.
- Added Queue Service `--db-backend postgres --postgres-dsn-file`.
- Replaced Cortex `--operational-sqlite-path /opt/novaic/data/cortex/operational.sqlite3` with Postgres operational-store flags.
- Relaxed `novaic-cortex/novaic_cortex/main_cortex.py` so Postgres mode no longer requires the deprecated SQLite path argument.

## Verification

- `bash -n scripts/start.sh` passed.
- `python -m py_compile novaic_cortex/main_cortex.py` passed.
- Focused scan of `scripts/start.sh` found no `entangled.db`, `operational.sqlite3`, `queue.db`, `gateway.db`, `device.db`, `business.db`, `--db-path`, or `--operational-sqlite-path` launch residue.
- Remote API machine path inventory confirmed the Postgres DSN files exist under `/opt/novaic/postgres/secrets` for Entangled, Gateway, Queue, and Cortex.

## Known Gaps

- Cortex still has a deprecated `--operational-sqlite-path` parser option and SQLite store implementation; deeper deletion belongs to the Cortex/Entangled child problem.
- Queue worker subprocesses still rely on the runtime code's database assembly behavior; Queue runtime fallback deletion belongs to the Queue child problem.
