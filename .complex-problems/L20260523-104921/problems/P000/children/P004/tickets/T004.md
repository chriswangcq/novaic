# Remove Cortex and Entangled SQLite server runtime fallbacks

## Problem Definition

Cortex and Entangled still expose SQLite fallback paths through service CLIs, default configs, and old migration scripts. After the Postgres cutover these are misleading current paths even when production startup already aims at Postgres.

## Proposed Solution

Make Cortex and Entangled server entry points Postgres-only where practical, delete old SQLite migration CLIs from current paths, and update root startup to match. Preserve only lower-level store code that cannot be safely deleted in this pass without replacing active abstractions, and record it explicitly as residue for the guard/follow-up phase.

## Acceptance Criteria

- Cortex startup CLI no longer accepts or requires an operational SQLite path.
- Entangled startup CLI no longer advertises SQLite db-path startup as normal server mode.
- Old Cortex/Entangled SQLite migration CLIs are deleted from current executable paths.
- Root `scripts/start.sh` matches the new Postgres-only CLI surface.
- Focused py-compile/residue scans pass, and any remaining SQLite store substrate is explicitly categorized.

## Verification Plan

- Run py-compile on changed Cortex/Entangled modules.
- Run focused tests for Postgres boundary modules if available.
- Run residue scans over `novaic-cortex`, `Entangled`, and `scripts/start.sh` for startup/migration fallback tokens.

## Risks

- Cortex's Postgres operational store currently subclasses the SQLite store implementation, so deleting all SQLite-named substrate requires a larger refactor.
- Entangled SQL entity definitions may still support SQLite dialect generation for tests or historical clients.

## Assumptions

- Production Cortex and Entangled should be started only against `/opt/novaic/postgres/secrets/novaic_cortex_dsn` and `/opt/novaic/postgres/secrets/novaic_entangled_dsn`.
