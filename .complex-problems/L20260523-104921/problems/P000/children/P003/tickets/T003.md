# Remove Gateway and Device SQLite current-path utilities

## Problem Definition

Gateway still defaults to SQLite auth/config storage and ships scripts that directly mutate `gateway.db`. Device still has an executable local SQLite DB module and comments that describe SQLite as the current storage owner. These contradict the Postgres/Entangled server-state target and make deleted DB files look supported.

## Proposed Solution

Make Gateway runtime Postgres-only by removing SQLite backend selection and db-file configuration from the current entry point/access layer. Delete executable SQLite migration/admin scripts in Gateway/current root scripts. Remove the Device local SQLite DB module if unused, and rewrite misleading Device comments to EntityStore/Entangled ownership.

## Acceptance Criteria

- Gateway current runtime no longer defaults to or accepts `--db-backend sqlite`.
- Gateway DB access initializes `PostgresGatewayDatabase` only and requires a Postgres DSN file.
- Current executable scripts that mutate or migrate `gateway.db` are deleted from root/Gateway paths.
- Device local SQLite DB module is deleted if unused, and Device comments no longer claim local SQLite storage ownership.
- Focused Gateway/Device imports/tests and residue scans pass.

## Verification Plan

- Run py-compile for changed Gateway/Device modules.
- Run focused Gateway Postgres tests and Device import checks.
- Run residue scans for `gateway.db`, `device.db`, `sqlite3`, and `--db-backend` over current Gateway/Device runtime paths.

## Risks

- Some tests may still use SQLite schema validators or client-side cache references; those should be categorized rather than silently kept as server runtime paths.

## Assumptions

- `/opt/novaic/postgres/secrets/novaic_gateway_dsn` is the production Gateway auth/config DSN file.
