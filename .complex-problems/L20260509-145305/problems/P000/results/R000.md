# File/display artifact context architecture result

## Summary

The deeper design should treat screenshots, uploaded files, shell artifacts, and large payloads as first-class resources in a filesystem/blob/Cortex artifact layer, not as ordinary chat message content. The LLM prompt should carry small manifests, previews, and access hints; the agent should intentionally dereference resources through tools such as `display`, `payload_read`, and shell/file reads. This matches the agent-subject model: the environment offers resources, but the agent chooses what to attend to.

The earlier current-round gating fix is necessary as a guardrail, but not sufficient as architecture. It prevents old display payloads from exploding the prompt, but it still leaves large artifacts conceptually inside the message/history pipeline. The better model is reference-oriented context plus explicit perception tools.

- Compared the design against the observed backend slow-path evidence:
  - screenshots were persisted/expanded as large display/base64-bearing tool results;
  - prompt tokens grew from about 15k to about 99k and then about 189k;
  - tool execution itself was comparatively fast.
- Produced a layered resource/display architecture:
  1. Storage plane: Blob/Cortex/filesystem stores bytes and durable payloads.
  2. Artifact manifest plane: Cortex stores compact metadata and refs.
  3. LLM context plane: prompt assembly includes bounded previews and handles.
  4. Perception/tool plane: `display`, `payload_read`, `payload_search`, shell reads dereference handles on demand.
  5. Monitor/UI plane: user-visible rendering can show thumbnails or images without injecting them into model context.

## Verification

- Current user attachments already follow the right instinct in `user_content.py`: attachments are described and the agent is told to call `display(file_url=...)` when needed, instead of injecting base64 into user messages.
- Current `display` tool fetches Blob refs and converts bytes into model-visible content under a budget, which is the right perception-tool shape.
- Current `step_result_projection.py` and `expand_messages_for_llm()` still allow stored tool results to become model-visible image/base64 content during context assembly, which is the flawed boundary.
- Current Cortex step tree only carries `content` and `step_ref` for tool results; it lacks a rich artifact manifest and round/visibility policy at the merge boundary.

## Known Gaps

- This result is a design analysis only; no code was changed.
- A full implementation should create concrete tickets for artifact manifest schema, tool output migration, context assembler changes, monitor rendering, and compatibility cleanup.

## Proposed Architecture

### 1. Core Principle

Large artifacts are resources, not messages.

Message history should say:

```text
Tool produced screenshot artifact A123.
mime=image/png size=138KB thumbnail=... full=blob://...
Use display(file_url="blob://...") to inspect it.
```

It should not store or repeatedly replay:

```text
{"type":"image","data":"<base64...>"}
```

### 2. Planes

#### Storage Plane

Owns bytes and retention:

- Blob Service for user uploads, screenshots, binary tool results, thumbnails.
- Cortex payload store for structured tool payloads and compact JSON metadata.
- Workspace filesystem for shell-created files and temporary working sets.

The storage plane can hold large objects. It is not the LLM prompt.

#### Artifact Manifest Plane

Each tool result that creates or references a file should publish an artifact manifest:

```json
{
  "artifact_id": "artifact:step-ref:round2:hd_screenshot:0:screen",
  "uri": "blob://device-screenshot/...",
  "thumbnail_uri": "blob://device-screenshot-thumb/...",
  "modality": "image",
  "mime_type": "image/png",
  "size_bytes": 138320,
  "sha256": "...",
  "created_by_step_ref": "step-ref:...",
  "round_id": "round2",
  "summary": "Host desktop screenshot",
  "access_hint": "display(file_url=\"blob://device-screenshot/...\")"
}
```

Cortex history should preserve this manifest. It should not need to preserve raw image bytes in JSON.

#### LLM Context Plane

The prompt should receive a bounded textual projection:

```text
[Artifact: screenshot.png, image/png, 138KB, created by hd_screenshot]
To inspect: display(file_url="blob://device-screenshot/...")
```

Only two cases may include actual visual content:

- The current tool result is being returned to the model immediately and is within budget.
- The agent explicitly called `display(file_url=...)`, meaning the agent chose to perceive that file now.

Historical artifacts should never silently re-expand into images.

#### Perception/Tool Plane

`display` becomes the canonical "look at this visual/file resource" action.

This is important philosophically and operationally:

- The user/environment can provide resources.
- The filesystem/blob/Cortex layer can retain resources.
- The agent decides when to inspect them.
- The prompt budget is spent on chosen perception, not ambient history.

For text/log payloads, the parallel tools are:

- `payload_read`
- `payload_search`
- `payload_summarize`
- shell file reads with `head/tail/rg`

For images/video/PDF-like resources, `display` is the perception gate.

#### Monitor/UI Plane

Agent monitor should be allowed to render thumbnails or screenshots for humans without changing what enters LLM context. UI display and model display are separate consumers of the same artifact manifest.

### 3. Tool Result Contract

Move from:

```json
{
  "_mcp_content": [
    {"type": "image", "data": "...", "mimeType": "image/png"}
  ]
}
```

Toward:

```json
{
  "text": "Captured host desktop screenshot.",
  "artifacts": [
    {
      "uri": "blob://device-screenshot/...",
      "thumbnail_uri": "blob://device-screenshot-thumb/...",
      "modality": "image",
      "mime_type": "image/png",
      "size_bytes": 138320,
      "display_hint": "display(file_url=\"blob://device-screenshot/...\")"
    }
  ],
  "llm_current_turn": {
    "allow_display": true,
    "budget_bytes": 200000
  }
}
```

`display_files` can remain as a compatibility projection for a while, but the source of truth should become `artifacts`.

### 4. Screenshot-Specific Behavior

`hd_screenshot` should:

1. Capture bytes from Device.
2. Store full screenshot in Blob.
3. Optionally store a thumbnail in Blob.
4. Return a compact artifact manifest.
5. Allow current-turn visual expansion only through an explicit current-turn display grant.
6. Store only the manifest in Cortex history.

This preserves the nice UX: if the agent asks for a screenshot, it can see it immediately. But after that round, the image becomes a resource reference, not an endlessly replayed prompt payload.

### 5. Shell/Filesystem Behavior

For shell output:

- Small stdout can be inline.
- Large stdout should be written to a workspace file or Cortex payload, with a preview plus path/ref.
- Created files should be returned as `files_created`/`artifacts`.
- The agent can inspect later with shell tools, `payload_read`, or `display`.

This mirrors how a good human operator works: keep logs/files on disk; inspect slices on demand.

## Required Invariants

- `context.jsonl` must not accumulate raw base64 images.
- `steps/*.json` must not use raw image base64 as the durable representation for screenshots.
- Historical tool results render as manifest/preview by default.
- Current-turn visual expansion is bounded, explicit, and non-persistent.
- `display(file_url=...)` is the canonical operation for intentional visual dereference.
- Monitor/UI rendering is independent from model-visible context.
- Context assembly has hard budgets per message, per round, and per artifact modality.
- Artifact access is tenant/user scoped and auditable.

## Why This Is More Elegant Than Only `current_round_id`

`current_round_id` solves the immediate leak: old images stop reappearing. But it still frames images as something that context assembly may or may not include.

The file/display model reframes the whole system:

- artifacts live in storage;
- history records handles;
- context carries affordances;
- tools perform perception;
- UI can render independently;
- LLM budget becomes intentional attention, not accidental persistence cost.

That is the cleaner abstraction.

## Migration Plan

### Phase 0: Policy And Audit

- Add an artifact/context policy document.
- Add tests proving no historical image/base64 enters LLM context.
- Audit all tool result producers that return `_mcp_content` image data.

### Phase 1: Artifact Manifest Substrate

- Define `ArtifactManifest` as a shared contract.
- Add manifest parsing/projection in Cortex.
- Add stable fields: `uri`, `thumbnail_uri`, `modality`, `mime_type`, `size_bytes`, `sha256`, `created_by_step_ref`, `round_id`, `access_hint`.

### Phase 2: Tool Output Migration

- Migrate `hd_screenshot` to store screenshots as Blob refs.
- Migrate shell large outputs to file/payload refs.
- Normalize `display` outputs so current-turn visual content is explicitly marked ephemeral.

### Phase 3: Context Projection

- Change Cortex/Runtime context assembly so historical artifacts become bounded text manifests.
- Allow current-turn display only when the tool result has a current-turn grant.
- Make `display(file_url=...)` the normal explicit visual dereference path.

### Phase 4: Monitor/UI Projection

- Let Agent Monitor render artifact thumbnails/images from artifact refs.
- Keep UI rendering separate from LLM prompt projection.

### Phase 5: Cleanup And Guardrails

- Remove old raw `display_files`/base64 compatibility from durable history.
- Add prompt budget assertions in tests.
- Add production counters:
  - context bytes by modality;
  - number of historical artifacts rendered;
  - number of current-turn display expansions;
  - base64 bytes entering LLM context.

## Final Recommendation

Yes, we should deliberately leverage the filesystem/Blob/Cortex resource layer and the `display` tool. The correct target is not "never show images to the model"; it is "never let stored history accidentally become perception." The agent should see resources when it intentionally looks, and history should keep stable handles rather than replaying raw sensory data.
