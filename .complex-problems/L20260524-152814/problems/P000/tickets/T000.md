# Clean release-controller dry-run contract and stale residue

## Problem Definition

The release controller contains a misleading global `dry_run_default` branch. That branch contradicts the desired clean contract: requests execute unless they explicitly ask for `dry_run=true`. The workspace also has stale path residue from previous construction ledgers and temporary work packages that can mislead future maintenance.

## Proposed Solution

Split the cleanup into bounded child problems. First remove the active `dry_run_default` code/config/docs/test path and pin the explicit dry-run contract with tests. Then deploy and verify the cleaned release-controller on the API host. Finally inventory stale local work-package/path residue and delete only items that are safe, unrelated to active product behavior, and supported by evidence.

## Acceptance Criteria

- Active release-controller source, config samples, docs, tests, deploy scripts, and CI guards contain no `dry_run_default` references.
- Omitted `dry_run` resolves to real execution, while explicit `dry_run=true` remains simulation-only.
- Runtime API-host config has no global dry-run default key.
- A new cleaned release-controller image is deployed and healthy.
- Runtime smoke tests prove default real staging release and explicit dry-run behavior.
- Stale residue cleanup is performed only where safe, with an inventory of what was removed or intentionally left.

## Verification Plan

- Run targeted release-controller tests and release-controller CI guard tests.
- Run `rg dry_run_default` across active release-controller/docs/deploy/CI paths.
- Validate deploy script syntax and render/validate the problem ledger.
- Inspect remote runtime config, controller health, staging health, and release-controller trigger responses.
- Use `git status --short` to verify staged changes are scoped and unrelated user changes are preserved.

## Risks

- Automated branch polling is live, so source pushes may deploy staging while work is still in progress.
- The workspace already has many unrelated dirty/untracked files; cleanup must not delete user work or unrelated historical material without evidence.
- Removing the global default changes any caller that relied on omitted `dry_run` as simulation.

## Assumptions

- `dry_run=true` is still a valid explicit operator simulation mode and is not considered transitional residue.
- Production remains promotion/rollback guarded by existing branch rules and should not be branch-auto-deployed.
- The user's approval covers deployment actions on the API host, but not blind deletion of unrelated unverified work.
