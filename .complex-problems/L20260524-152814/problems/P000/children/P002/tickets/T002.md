# Deploy cleaned release-controller runtime

## Problem Definition

The source contract is cleaned locally, but the API host may still carry the removed global key in runtime config and may still run an image built from the older code path.

## Proposed Solution

Commit and push the cleaned source so the release-controller image can be built from immutable repository state. Remove the stale key from `/opt/novaic/release-controller/config.json`, build and push a new release-controller image on the API host from the cleaned commit, deploy it through `./deploy release-controller-image`, then verify controller health, staging health, real omitted-dry-run trigger behavior, and explicit dry-run behavior.

## Acceptance Criteria

- Local cleaned source is committed and available to the API host build path.
- API-host runtime config contains no removed global dry-run key.
- New release-controller image digest is built, pushed, and deployed.
- Controller `/health` passes and `/v1/status` reports polling healthy.
- Manual trigger without `dry_run` records `execution.dry_run=false` and succeeds.
- Manual trigger with `dry_run=true` records `execution.dry_run=true` and does not move the current release pointer.
- Staging gateway and factory health checks pass after deployment.

## Verification Plan

- Inspect remote config JSON with Python and verify the key is absent.
- Inspect running release-controller image digest and health endpoint.
- Use release-controller API trigger calls and compare release pointer before/after explicit dry-run.
- Curl local and public staging health endpoints.
- Run `git status --short` to verify only intended files are staged/committed.

## Risks

- Autonomous polling is live and may start a staging release after the commit is pushed.
- Building images on the API host depends on remote Docker and registry availability.
- Triggering the current commit twice may be skipped by operator expectations, but manual trigger should still produce a run.

## Assumptions

- The API host has SSH/Docker/registry access as in the prior deployment work.
- A real staging redeploy of the cleaned commit is acceptable.
