# Inventory and clean stale workspace residue

## Problem Definition

The working tree contains stale paths from previous problem-package runs and other pre-existing dirty files. Some paths are likely generated residue, while others may be user work or historical documentation changes. The cleanup needs to reduce misleading clutter without destroying unrelated user changes.

## Proposed Solution

Inventory dirty/untracked path groups, identify safe residue by ownership and current usefulness, delete only obsolete untracked problem-package files from older ledgers, preserve the current ledger, and leave unrelated modified/tracked files untouched with an explicit explanation.

## Acceptance Criteria

- Current ledger `.complex-problems/L20260524-152814` remains intact.
- Older untracked `.complex-problems` directories and loose old problem-package Markdown inputs are removed from the working tree.
- Unrelated modified tracked files such as `docs-archive`, `scripts/start.sh`, and pre-existing dashboard deletion are not silently staged or reverted.
- Final status clearly distinguishes remaining unrelated dirt from this cleanup's tracked changes.

## Verification Plan

- Run `git status --short` and `find .complex-problems` before and after cleanup.
- Verify current ledger status with `ledger.py status` and `ledger.py validate` if available.
- Confirm `git status --short -- .complex-problems` only shows the current ledger.
- Inspect branch residue separately and delete only merged/safe branches if evidence supports it.

## Risks

- Old untracked ledger files might contain useful history, but they are not part of the current active ledger and have no committed owner.
- Remote branch cleanup can be destructive if applied to unmerged branches; only safe evidence-based branch cleanup should happen in this ticket.

## Assumptions

- The active problem package for this work is `.complex-problems/L20260524-152814`.
- Untracked old problem package bodies from prior dates are generated residue, not source-of-truth product code.
