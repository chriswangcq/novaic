# Delete dry_run_default from active release-controller contract

## Problem Definition

The source contract still models a global `dry_run_default` even though the desired product behavior is explicit per-request dry-run only. This creates a misleading branch in model parsing, planner behavior, sample config, tests, and docs.

## Proposed Solution

Remove the `dry_run_default` field and all parsing/serialization/docs references. Add or update planner and API tests so omitted `dry_run` resolves to `False` and explicit `dry_run=true` remains supported. Keep the `dry_run` request field and runner simulation behavior intact.

## Acceptance Criteria

- No active source/config/docs/tests references to `dry_run_default` remain.
- Omitted `dry_run` creates command plans with `dry_run=False`.
- Explicit `dry_run=true` still creates dry-run command plans and skips command execution.
- Release-controller tests and CI guard tests pass.

## Verification Plan

- Run targeted release-controller tests.
- Run release-controller CI guard tests.
- Run `rg dry_run_default` over active release-controller/docs/deploy/CI paths.
- Inspect `git diff` for scoped cleanup without unrelated edits.

## Risks

- Tests that used the old default as a convenience need explicit dry-run inputs instead of restoring a hidden default.

## Assumptions

- Removing the global field is the intended non-conservative state, not an intermediate migration.
