# Make Release Controller enforce CI gates

## Problem Definition

The release pipeline now owns backend and LLM Factory deployment, but it still treats CI-like verification as a small generic `deploy.verify_commands` preflight list. That makes the desired flow ambiguous: developers may run unit tests locally, but the authoritative staging-admission gate is not modeled as a first-class Release Controller phase with clear ordering, naming, documentation, and failure semantics.

## Proposed Solution

Add an explicit Release Controller CI quality-gate capability. Introduce typed config for quality gates, have branch release plans execute those gates after checkout/submodule update and before image build, keep deploy smoke as a separate post-deploy stage, and ensure a failed gate stops build/deploy while recording a failed release run. Preserve existing backend/factory manual deploy rejection. Update sample config, docs, and tests so the official flow is local unit tests for fast feedback, controller CI gate for staging admission, staging smoke/integration checks, and prod promotion from the passed immutable release.

## Acceptance Criteria

- Release Controller config has an explicit quality-gate list with stable gate names and argv commands.
- Branch release command plans contain named `quality-*` steps after checkout/submodule update and before image builds.
- Promotion and rollback plans do not rebuild or rerun branch CI gates; they deploy/promote already accepted immutable image refs and smoke the target namespace.
- Runner failure behavior blocks subsequent build/deploy steps when a quality gate exits nonzero.
- Config validation rejects malformed quality gate definitions and empty commands.
- Sample config includes meaningful default quality gates that are executable in the controller worktree.
- Unit tests cover config parsing, step ordering, dry-run/real failure behavior, and retention of manual deploy guards.
- Docs describe the authoritative CI/CD flow and make clear that local tests are fast feedback, not the final admission authority.

## Verification Plan

Run release-controller unit tests, release-controller CI guard tests, deploy syntax checks, relevant CI lint scripts, and focused manual negative checks for direct `services-image` / `factory-image` invocation. Inspect generated command plans in tests to prove gate ordering and prod promotion behavior.

## Risks

- Adding a heavy full-repo test gate immediately can block all releases because the current monorepo full matrix still has historical red suites.
- If gates are modeled as generic verify commands only, the architecture remains ambiguous and future agents may treat smoke/preflight/CI as interchangeable.
- If prod promotion reruns branch gates or rebuilds images, it breaks the immutable promote model.

## Assumptions

- This ticket should introduce the clean quality-gate abstraction now, while the initial default gate set can stay focused on deterministic checks that are currently green in the controller worktree.
- Full application integration tests should be added as future named gates once their suites are made green and executable in the controller/builder environment.
- Release Controller self-upgrade remains a separate control-plane bootstrap concern.
