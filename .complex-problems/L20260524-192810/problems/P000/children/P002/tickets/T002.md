# Wire default CI gates and document the canonical flow

## Problem Definition

The code can now express quality gates, but the repository still needs a default controller config and docs that make the intended CI/CD flow unambiguous. Without this, future changes can leave the gate list empty or treat local tests, release preflight, staging smoke, and prod promotion as interchangeable.

## Proposed Solution

Add meaningful deterministic quality gates to `novaic-release-controller/config.sample.json`, keep lightweight `deploy.verify_commands` as deploy preflight, update Release Controller architecture and deploy runbook docs to describe the canonical flow, and extend repository guard tests to require the sample config and docs to mention quality gates and the local-test/staging/prod responsibilities.

## Acceptance Criteria

- Sample config has top-level `quality_gates` with stable names and executable argv commands.
- Sample config keeps deploy `verify_commands` as lightweight preflight distinct from quality gates.
- Docs explain local unit tests as fast feedback and Release Controller quality gates as authoritative staging admission.
- Docs explain staging smoke/integration and prod promotion using the already accepted immutable release.
- Repository guard tests fail if docs/sample config stop mentioning quality gates.
- Release-controller tests and CI guard tests pass.

## Verification Plan

Run release-controller unit tests, `scripts/ci/test_release_controller_ci.py`, relevant docs/lint scripts, `python3 -m pytest -q`, and inspect sample config parsing through `load_config`.

## Risks

- Choosing too-heavy default gates can block every release because historical full matrix failures still exist.
- Choosing too-light gates can make the new capability cosmetic.
- Docs can drift unless CI guard tests pin the important contract text.

## Assumptions

- Initial default gates should be deterministic checks that are green today: release-controller tests and release-controller CI guard tests.
- Full application integration suites should become future quality gates only after the suites are made green in the controller/builder environment.
