# Run final quality-gated staging poll

## Problem Definition

The controller image now includes the dependencies required by the default quality gates. The final proof is a successful real branch-poll staging release for the latest main commit with quality gates executed before build/deploy.

## Proposed Solution

Call `/v1/polls/once` while autonomous polling remains disabled, fetch the resulting run, verify plan and execution ordering, verify staging health and prod pointer stability, re-enable polling in runtime config and restart controller, confirm status is clean and main head is skipped as unchanged, then run direct manual deploy guard checks.

- A real `/v1/polls/once` outcome for `main` succeeds for commit `491bd7e6ad55a70d96f41d0fbfea872be72e92e8`.
- Quality gates are present before build/deploy in the run plan and both have successful execution evidence.
- Staging health endpoint passes.
- Prod current release pointer remains on `e4fb5740cc8481617bcd3d9f2928d7f4ebf586df`.
- Polling is re-enabled and reports `last_error=null`.
- Direct manual backend/factory deploy commands still reject without controller identity.

## Verification Plan

Use `/v1/polls/once`, `/v1/runs/{run_id}`, `/v1/status`, public health curls, remote config edit/restart, and local direct deploy guard commands.

## Risks

- The poll run may take several minutes because it builds and deploys API/Factory images.
- If poll records the branch head but deploy health fails, rollback/pointer state must be inspected before re-enabling polling.
