# Pause polling and publish quality-gate commit

## Problem Definition

The quality-gate code must reach `main`, but the running API-host controller image does not yet understand the new gate model. Polling must be paused before the push, then only intentional files should be committed and pushed.

## Proposed Solution

Pause API-host polling by editing the runtime config and redeploying the current Release Controller image, run local validation, stage the intended Release Controller source/config/docs/test changes plus the active problem ledger index if appropriate, commit to `main`, push to `origin/main`, and record commit/push evidence. Avoid staging unrelated pre-existing dirty docs/archive/app/start changes.

## Acceptance Criteria

- Remote polling is disabled and status confirms no new autonomous release will be triggered by the push.
- Local validation relevant to the quality-gate change passes.
- Git staging contains only intended files for this work and no unrelated dirty files.
- Commit is created on `main` and pushed to `origin/main`.
- Commit id and staged file scope are recorded.

## Verification Plan

Use remote `/v1/status`, local test commands, `git diff --cached --name-status`, `git status --short`, `git commit`, `git push`, and `git rev-parse HEAD`.

## Risks

- Remote polling pause can be incomplete if the config changes but the controller is not restarted.
- Staging unrelated dirty files could accidentally publish archived docs or user edits.
- If local validation is incomplete, the pushed commit can fail remote quality gates later.

## Assumptions

- It is acceptable to use `deploy release-controller-image` for control-plane self-upgrade/pause operations.
- Parent repo is already on `main` and submodules have no intended changes for this quality-gate work.
