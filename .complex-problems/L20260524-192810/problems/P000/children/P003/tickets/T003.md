# Upgrade API-host Release Controller with quality gates

## Problem Definition

The code and docs now define Release Controller quality gates, but the running API-host controller still uses the previous image and remote config. Until the controller image and `/opt/novaic/release-controller/config.json` are upgraded, branch releases will not enforce the new gates.

## Proposed Solution

Use the same safe control-plane rollout pattern as the previous controller-only migration: pause polling, commit and push the quality-gate code intentionally, build/push a new Release Controller image from that commit, update remote config with the quality gate list, deploy the new controller image, trigger or observe a staging release, verify the command plan contains `quality-*` steps before build/deploy, verify staging health, ensure prod pointer remains unchanged unless promoted explicitly, and resume polling.

## Acceptance Criteria

- Polling is paused before the quality-gate commit reaches `main`.
- Intended files are committed and pushed, without staging unrelated dirty workspace changes.
- A Release Controller image built from the quality-gate commit is deployed on the API host.
- Remote runtime config contains `quality_gates` with `release-controller-ci` and `release-path-lints`.
- A staging run through Release Controller succeeds and its command plan shows quality gates before image build/deploy.
- Release Controller status has `last_error=null`, polling is re-enabled, staging health is clean, and prod current pointer remains on the prior promoted release unless a deliberate promotion is made.
- Manual backend/factory deploy guards still fail direct invocation.

## Verification Plan

Use local tests before commit, git status/diff to stage only intended files, remote `/v1/status`, remote config inspection, Docker image digest, Release Controller trigger response/run details, public staging health, prod pointer comparison, and manual direct deploy guard checks.

## Risks

- If polling is not paused before push, the old controller could process the new commit without the new gate support.
- The default quality gates may fail in the remote controller container if dependencies or PYTHONPATH assumptions are wrong.
- Remote config edits must preserve existing branch rules, registry paths, health URLs, and polling settings.

## Assumptions

- The controller image can still be self-upgraded through the explicit `release-controller-image` bootstrap path.
- Prod should not be promoted as part of this child unless staging verification shows a code release that the user wants promoted; the acceptance target is CI gate enforcement on staging admission.
