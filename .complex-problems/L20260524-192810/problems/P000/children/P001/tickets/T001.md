# Add typed quality gate model and branch-plan execution

## Problem Definition

Release Controller needs first-class CI gate semantics instead of relying on an overloaded `deploy.verify_commands` list. The branch plan should make quality gates visible, ordered, and failure-blocking before image build/deploy, while promotion and rollback stay image-pointer operations.

## Proposed Solution

Add a `QualityGate` model and `quality_gates` field to controller config, parse and validate named gates from config, preserve backward compatibility by allowing existing `deploy.verify_commands` to continue as preflight steps, and update branch planning so gates become `quality-<name>` command steps after source checkout/submodule update and before image build. Extend tests to verify model parsing, malformed config rejection, branch step ordering, no gates in promotion/rollback, and runner short-circuit behavior when a quality step fails.

## Acceptance Criteria

- `ControllerConfig` exposes typed `quality_gates` and serializes them back to mappings.
- `QualityGate.from_mapping` rejects missing/empty names and empty argv arrays.
- Branch release plans include all quality gates with stable `quality-*` step names before build steps.
- Promotion and rollback command plans contain only deploy/smoke steps and no quality/build steps.
- Runner tests show a failing quality gate stops subsequent steps.
- Existing planner tests for controller deploy env and manual deploy guard tests continue to pass.

## Verification Plan

Run `PYTHONPATH=novaic-release-controller python3 -m pytest novaic-release-controller/tests -q`, `python3 -m pytest -q scripts/ci/test_release_controller_ci.py`, `python3 -m pytest -q`, and a manual direct deploy guard check if touched behavior warrants it.

## Risks

- If existing config parsing becomes strict without a default, remote controller config may break on upgrade.
- If gate names are not normalized, command plans may become unstable or invalid.
- If verify commands and quality gates both exist, ordering must stay clear rather than becoming another ambiguous list.

## Assumptions

- Existing `deploy.verify_commands` should remain as release preflight for now so older config keeps working, but new CI semantics live under top-level quality gates.
- Initial gates will be deterministic repo checks; heavier app integration gates belong in the next child and future follow-ups.
