# Stream reasoning through Factory, Runtime, Entangled, and App

## Problem Definition

The current monitor shows provider reasoning only after a full LLM response is saved. Users need live reasoning progress, but the architecture should stay clear: Factory owns provider transport, Runtime owns saga semantics and projection, Business/Entangled owns the sync data plane, and App renders subscribed projection rows.

## Proposed Solution

Implement reasoning streaming as an end-to-end platform path. Add provider streaming support in LLM Factory for the OpenAI-compatible route, add a Runtime streaming client/handler that aggregates the final OpenAI-style response while coalescing reasoning deltas into the existing `agent-activity-records` projection, and adjust the App timeline to render same-row running updates smoothly. Keep non-streaming behavior intact and avoid introducing a separate frontend transport.

The ticket must be split before execution because it crosses service boundaries and has independent verification needs. The split should respect the user's rule: broad depth-1 and depth-2 work is split; depth-3 leaves may use one_go only when bounded and directly testable.

## Acceptance Criteria

- Factory can return normalized streaming chunks and still return complete non-streaming chat completions.
- Runtime can request streaming, aggregate a final chat completion response, and keep existing saga outputs compatible.
- Runtime emits coalesced running/final reasoning updates through existing Business/Entangled projection writes.
- App Agent Monitor follows and renders same-row reasoning text updates without a separate frontend stream.
- Tests cover each service boundary and the end-to-end compatibility contract.
- No long-term duplicate fallback path or misleading legacy code remains.

## Verification Plan

- Unit-test provider stream parsing/normalization and non-streaming regression in `novaic-llm-factory`.
- Unit-test Runtime streaming aggregation, projection throttling/coalescing, and final response shape in `novaic-agent-runtime`.
- Unit-test App timeline title/detail/autoscroll behavior for running same-row updates in `novaic-app`.
- Run focused test suites for touched repos, then run repository guard checks that cover active agent path and app build/lint where practical.
- Review diffs for fallback residue and accidental raw reasoning persistence into future LLM input.

## Risks

- Provider streaming formats differ; the first implementation should explicitly scope provider support and fail clearly for unsupported providers.
- Too-frequent projection updates could pressure Entangled/Postgres; Runtime must coalesce updates.
- Partial streamed reasoning must not contaminate durable LLM context before the final response is accepted.
- Existing tests may rely on synchronous fake clients; contracts must be adapted without weakening them.

## Assumptions

- The first production path can target OpenAI-compatible streaming because current Factory already routes unknown compatible providers through `OpenAIProvider`.
- Anthropic and Google streaming can remain unsupported until implemented explicitly, as long as the Runtime does not silently pretend they are streaming.
- Existing Entangled delta invalidation is sufficient for App cache refresh when the same activity row is updated.
