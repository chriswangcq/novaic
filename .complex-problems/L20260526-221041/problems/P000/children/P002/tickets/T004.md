# Runtime streaming aggregation and projection path

## Problem Definition

Runtime currently calls Factory synchronously and only projects reasoning after the final assistant message is appended. It must consume normalized Factory streaming, aggregate the final response for existing saga logic, and publish coalesced running/final reasoning updates through existing Business/Entangled projection writes.

## Proposed Solution

Split Runtime work into three third-layer leaves: a Factory client streaming parser/aggregator, activity projection support for stable running/final reasoning records, and LLM handler integration that enables streaming while preserving final response semantics.

## Acceptance Criteria

- Runtime Factory client can consume Factory normalized SSE and aggregate content, reasoning, tool-call deltas, finish reason, usage, and factory metadata.
- Activity projection can upsert a stable reasoning record as running and final without creating duplicate rows.
- LLM handler uses streaming when possible and still returns a complete OpenAI-style response to `react_think`.
- Tests cover parser, projection coalescing/final state, and handler compatibility.

## Verification Plan

- Add Runtime unit tests for SSE parsing and final response assembly.
- Add Runtime unit tests for stable activity record IDs and running/final status updates.
- Add handler tests with fake streaming client proving final `response` shape is unchanged for saga code.
- Run focused Runtime tests for touched modules.

## Risks

- Tool-call delta aggregation can be tricky; implementation should support the current function call delta shape without overgeneralizing.
- Partial reasoning must not be appended to durable context before final response handling.
- Projection write frequency must be bounded.

## Assumptions

- Factory emits normalized SSE chunks from P001.
- Business client entity upsert remains the correct existing data-plane write surface.
