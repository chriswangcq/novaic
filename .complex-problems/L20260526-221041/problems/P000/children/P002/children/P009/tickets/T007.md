# Use streaming Factory calls in Runtime LLM handler

## Problem Definition

Runtime has a stream-capable Factory client and a stable reasoning projector, but `handle_llm_call` still uses the blocking path. The handler must connect these pieces without changing saga-facing result shape.

## Proposed Solution

Update `LLMBusiness.call()` to delegate to `stream_chat()` when requested. Update `handle_llm_call` to use streaming, accumulate reasoning deltas through `ReasoningStreamProjector`, finalize the reasoning row after completion, and return the same success response contract as before.

## Acceptance Criteria

- Handler calls Factory streaming path for LLM calls.
- Reasoning deltas produce coalesced running projection updates.
- Final response still has `success`, `scope_id`, `round_id`, `response`, and `model`.
- Durable context append remains downstream and final-response based.
- Focused tests cover handler streaming success and stream failure.

## Verification Plan

- Add/update handler tests with fake `LLMBusiness`/Factory client behavior.
- Run focused handler/client/projection tests.

## Risks

- Handler tests may need to mock Business config lookup and context preparation dependencies explicitly.

## Assumptions

- Factory may fail clearly for unsupported streaming providers; Runtime should surface that failure rather than silently falling back.
