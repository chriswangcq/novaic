# Verify App streaming reasoning stays on Entangled activity records

## Problem Definition

The frontend should receive reasoning progress through the existing `agent-activity-records` Entangled entity. Adding a separate browser SSE/WebSocket stream for reasoning would create duplicate state, race conditions, and unclear ownership between App, Runtime, and Entangled.

## Proposed Solution

Audit the App entity contract, data store, hook, and ActivityTimeline usage for the streaming fields required by the runtime projection. Add or tighten focused guard coverage if any contract field is missing. Verify no App-side reasoning-specific network stream path was introduced.

## Acceptance Criteria

- The App `agent-activity-records` entity/type/normalizer includes `public_title`, `status`, `text`, and timestamp fields needed for streaming updates.
- Reasoning streaming UI continues to be sourced from `agentActivityRecordsStore.useStream` and `ActivityTimeline`, not a new reasoning WebSocket/SSE/EventSource path.
- Focused contract/guard tests pass.

## Verification Plan

Inspect the App data contract and search for new reasoning stream transport usage, then run the focused App activity contract/guard tests.

## Risks

- A generic transport helper may contain SSE/EventSource code for unrelated features; the audit should distinguish existing infra from a new reasoning-specific channel.
- Guard tests can become brittle if they assert too much raw source text.

## Assumptions

- The contract-level source of truth for App activity records is under `novaic-app/src/data/entities` and `novaic-app/src/components/hooks/useActivityTimeline.ts`.
- Existing generic Entangled stream infrastructure is allowed and is not considered a parallel reasoning channel.
