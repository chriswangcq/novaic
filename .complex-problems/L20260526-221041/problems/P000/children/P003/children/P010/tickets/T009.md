# Make ActivityTimeline react to same-record reasoning stream mutations

## Problem Definition

`ActivityTimeline` can already render activity rows, but its follow-bottom trigger is keyed mostly by list growth and a latest record identity string. Streaming reasoning updates intentionally keep the same record identity while mutating text/status/title fields, so the app can miss update-driven scroll/render expectations even though Entangled delivered a changed row.

## Proposed Solution

Add a small explicit latest-record update key for `ActivityTimeline` that fingerprints the fields expected to change during streaming (`status`, titles, text/detail, truncation, and update timestamp when present). Use that key in the follow-bottom effect instead of the older identity-only key. Preserve the existing record rendering path and add focused component/unit coverage for running reasoning rows and same-record update key behavior.

## Acceptance Criteria

- The latest-record follow-bottom dependency changes when a streaming reasoning row changes text/status/title without adding a new row.
- Running reasoning rows render the public title `正在思考` and their detail text.
- Tests cover same-record update key changes for streaming-style mutations.
- The implementation stays inside the existing activity timeline data path and does not introduce a second activity transport.

## Verification Plan

Run the focused app test for `ActivityTimeline` after the change, and inspect the affected files to ensure the update key uses only explicit record fields rather than hidden UI state.

## Risks

- Including the full reasoning text in a dependency key could be noisy for very long streams.
- A too-small fingerprint could miss equal-length text replacements.
- Existing tests may need small adjustments if they assume the old key is internal-only.

## Assumptions

- Streaming reasoning text is append-oriented, so length and tail fingerprinting are enough to detect meaningful same-row progress.
- `ActivityTimelineRecord` can safely carry an optional update timestamp if the hook already receives one from synced activity records.
