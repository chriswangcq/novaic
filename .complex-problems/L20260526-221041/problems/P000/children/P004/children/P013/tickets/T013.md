# Audit streaming contract cleanliness and remove residue

## Problem Definition

The implementation must represent the intended architecture cleanly. A working stream is not enough if the code leaves fallback branches, duplicated frontend channels, or partial reasoning state that can leak into future prompts.

## Proposed Solution

Review the touched diffs and relevant source paths in Factory, Runtime, and App. Search for fallback/residue patterns and inspect the LLM history/response boundary. Make scoped cleanup edits only if the review finds concrete residue, then rerun the affected focused tests.

## Acceptance Criteria

- New reasoning streaming path has no long-term fallback branch that silently switches to old non-stream behavior.
- Runtime streams reasoning deltas to activity projection while the final assistant response remains the only response returned to the caller.
- No raw partial reasoning chunks are appended to message history for future LLM calls.
- App monitor remains on Entangled activity records with no parallel reasoning stream or stale monitor path.
- Any cleanup edits are verified by focused tests.

## Verification Plan

Use `git diff`, `rg`, and focused source reads across touched repos, then rerun only tests affected by any cleanup.

## Risks

- The codebase has existing non-streaming methods for other compatibility paths; the review must distinguish existing public APIs from forbidden fallback inside the new reasoning path.
- Overzealous deletion could break existing non-stream callers outside this construction.

## Assumptions

- `FactoryLLMClient.chat()` may remain as an existing explicit non-stream API, but `handle_llm_call` should not silently fall back to it for the new reasoning path.
- App generic sync infra is allowed; forbidden residue is a new reasoning-specific frontend stream outside Entangled activity records.
