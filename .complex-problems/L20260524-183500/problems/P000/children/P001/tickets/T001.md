# Add controller invocation contract and deploy guards

## Problem Definition

The Release Controller planner currently calls `./deploy services-image` and `./deploy factory-image` without explicit identity, and the runner replaces the environment when step env is present. The `deploy` script still exposes manual backend/factory release paths and legacy backend fallbacks.

## Proposed Solution

Pass release-controller identity env vars from planner to image deploy steps, merge those vars with the inherited environment in the runner, and add deploy-script guards that require controller identity for backend/factory image deployment. Disable obsolete backend remote-build and legacy release targets from the normal CLI by failing fast with a clear Release Controller-only message. Add focused tests for planner env, runner env merge, and deploy manual rejection.

## Acceptance Criteria

- Deploy steps generated by branch releases, prod promotions, and rollbacks carry controller env metadata.
- Runner subprocesses retain inherited environment variables while adding step env.
- `./deploy services-image staging <sha image>` and `./deploy factory-image staging <sha image>` fail before remote calls unless controller env vars are present.
- Backend remote-build/legacy targets are no longer executable as release paths.
- Test suite proves the intended behavior.

## Verification Plan

Run release-controller unit tests, deploy guard CI tests, root pytest suite, and `bash -n deploy`.

## Risks

- Guarding too broadly could disable non-backend operational targets.
- Controller image/runtime must be rolled out before guarded deploy is used remotely.

## Assumptions

- `release-controller-image` remains a controlled bootstrap/self-upgrade executor until the controller has a separate self-upgrade API; it must still reject mutable image refs.
