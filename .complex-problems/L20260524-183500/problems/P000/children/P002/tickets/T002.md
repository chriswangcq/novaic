# Roll out guarded deploy through Release Controller

## Problem Definition

The repository now requires Release Controller identity for backend/factory deploy executor calls. The API host still runs the previous controller image until upgraded. If the guarded `deploy` script reaches the controller worktree while the old controller is still active, staging deployment can fail because the old planner does not pass the required env vars.

## Proposed Solution

Run a controlled migration: pause polling, commit/push the controller-only changes, build and deploy a new Release Controller image, trigger or observe staging through the upgraded controller, promote prod to the same resulting release if needed, verify manual deploy paths fail, and resume polling. Finish by validating the ledger and documenting final state.

## Acceptance Criteria

- Full local test matrix relevant to release-controller/deploy guards passes.
- API host Release Controller is upgraded to an image built from the controller-only commit.
- Staging deployment runs through Release Controller with the guarded deploy executor.
- Prod remains enrolled in current/previous immutable pointer promote/rollback state after the new commit.
- Direct manual backend/factory deploy commands fail locally before remote side effects.
- Polling is left enabled and controller/status/health are clean.
- Documentation no longer presents GitHub Actions or direct deploy scripts as backend/factory release paths.

## Verification Plan

Run `./scripts/run_all_tests.sh`; pause remote polling before push; build/push/deploy release-controller image; call Release Controller status/trigger/promote/rollback endpoints as needed; verify public/local health and manual guard failure.

## Risks

- Pausing polling and not resuming would silently stop branch-driven staging deploys.
- Pushing before controller upgrade may create a failed staging run, though it should not corrupt release pointers.
- Controller self-upgrade still uses `release-controller-image` as a bootstrap path because it cannot replace itself through an already-stopped request yet.

## Assumptions

- `release-controller-image` remains allowed as a bootstrap/self-upgrade path for the control plane itself.
- Backend/factory deployment mutation must go through Release Controller; non-backend targets such as frontend, iOS, desktop, relay, and host-infra are outside this ticket.
