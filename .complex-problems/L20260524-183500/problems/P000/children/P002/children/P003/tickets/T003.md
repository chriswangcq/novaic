# Validate, pause polling, commit and push

## Problem Definition

Controller-only release changes must be validated and published without accidentally staging unrelated dirty files. Because the currently running remote controller may not yet pass deploy identity, remote polling should be paused before pushing the guarded `deploy` script to `main`.

## Proposed Solution

Run the canonical test matrix, inspect git status/diff, pause API-host Release Controller polling, stage only relevant controller-only files, commit, push `main`, and capture the resulting commit SHA for remote rollout.

## Acceptance Criteria

- `./scripts/run_all_tests.sh` passes.
- API-host polling is paused before push.
- Only intentional files are staged and committed.
- Commit is pushed to `origin/main`.

## Verification Plan

Use local test output, `git diff --cached --name-only`, `git status --short`, `git rev-parse HEAD`, and remote Release Controller status/polling state.

## Risks

- The worktree contains unrelated dirty files and older uncommitted ledger files; staging must be path-specific.
- If polling is paused and not resumed later, staging automation stops until P004 completes.

## Assumptions

- It is acceptable that the newest operational ledger remains uncommitted until rollout is fully verified, to avoid creating a second release loop after the code commit.
