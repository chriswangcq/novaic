# Upgrade controller and run guarded staging/prod verification

## Problem Definition

The API host currently has polling paused and may still be running a Release Controller image that predates the deploy invocation guard. It must be upgraded to an image built from parent commit `e4fb5740cc8481617bcd3d9f2928d7f4ebf586df`, then the guarded deploy path must be exercised through the controller.

## Proposed Solution

Build and push an immutable release-controller image from the new commit on the API host, deploy it with `release-controller-image`, trigger staging through `/v1/triggers`, verify staging health and manual guard failure, promote the same release to prod if staging succeeds, verify prod health/pointers, then resume polling.

## Acceptance Criteria

- Running Release Controller image digest corresponds to an image built from commit `e4fb5740cc8481617bcd3d9f2928d7f4ebf586df`.
- A staging run through Release Controller succeeds using guarded `services-image` and `factory-image` executor steps.
- Prod current pointer is promoted to the new commit/images and prod health is clean.
- Manual backend/factory deploy commands fail without controller identity.
- `/v1/status` shows polling enabled, not running, and no last error after migration.

## Verification Plan

Use Docker image digest/build logs, `./deploy release-controller-image`, Release Controller `/v1/triggers`, `/v1/promotions/prod`, `/v1/status`, public/local health curls, direct manual guard commands, and optional rollback dry-run.

## Risks

- Building the controller image on the API host is still a bootstrap/self-upgrade path; backend/factory deployments remain controlled by Release Controller.
- Staging trigger can fail if submodule pointers or remote worktree fetch are wrong.

## Assumptions

- The API host has access to GitHub and the local registry at `127.0.0.1:5000`.
- `release-controller-image` remains allowed as the control-plane self-upgrade bridge.
