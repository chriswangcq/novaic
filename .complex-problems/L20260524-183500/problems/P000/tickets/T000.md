# Make backend releases controller-only

## Problem Definition

Backend and LLM Factory releases still have manual `deploy` entrypoints that operators can run directly. Release Controller should be the only release interface for namespace deployments, promotions, and rollbacks; `deploy` should be reduced to an internal executor that refuses direct release mutation unless called by Release Controller with explicit run metadata.

## Proposed Solution

Implement an explicit controller invocation contract. Release Controller planner will attach controller identity, run id, namespace, and commit env vars to backend/factory deploy steps. The runner will merge those step vars with the process environment. The `deploy` script will reject direct manual calls to backend/factory image deploys and obsolete backend remote-build/legacy deploy paths, while keeping non-backend operational targets outside the scope unchanged. CI tests and docs will be updated to enforce that backend/factory deployment happens through Release Controller APIs or polling only.

## Acceptance Criteria

- Release Controller deploy steps include `NOVAIC_DEPLOY_CALLER=release-controller`, run id, namespace, and commit metadata.
- Runner execution merges the existing environment with step env instead of replacing it.
- `./deploy services-image ...` and `./deploy factory-image ...` fail fast without Release Controller env metadata.
- Obsolete backend remote-build and legacy targets (`services`, `api-backend`, `services-legacy`, single backend services, and `factory`) fail as disabled manual release paths.
- Tests cover planner env injection, runner env merge, and direct manual path rejection.
- Docs present Release Controller as the sole backend/factory release interface and describe `deploy` as internal executor.
- Deployment rollout does not leave staging/prod blocked by the new guard.

## Verification Plan

Run release-controller unit tests, root pytest suite, deploy guard tests, shell syntax check for `deploy`, and remote Release Controller/staging/prod health checks after rollout.

## Risks

- If the remote `deploy` guard lands before the running controller image sends the new env vars, auto staging deployment can fail.
- Release Controller self-upgrade still needs a bootstrap path unless/until a separate controller self-upgrade API exists.
- Some old docs/CI guards may still call manual deploy strings and must be corrected or intentionally scoped.

## Assumptions

- Scope is server backend and LLM Factory release mutation; frontend, iOS, desktop, relay, and host-infra deployment surfaces are not being converted in this ticket.
- `deploy` remains physically present because Release Controller needs an executor boundary for Docker Compose operations on the API host.
