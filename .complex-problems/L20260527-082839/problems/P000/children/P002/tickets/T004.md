# Commit and push source for Release Controller deployment

## Problem Definition

The release-controller can only deploy pushed commits. The local implementation spans subrepos and root ledger/submodule state, so commits must be created in the right order and pushed cleanly.

## Proposed Solution

Split into final pre-push verification/status capture, subrepo commits/pushes, and root submodule/ledger commit/push. Exclude unrelated untracked files.

## Acceptance Criteria

- Final focused tests or already-current test evidence is available before commit.
- Server-relevant subrepos (`novaic-llm-factory`, `novaic-agent-runtime`) are committed and pushed.
- App source changes are committed and pushed as source hygiene, even though server release only consumes backend/factory/runtime submodules.
- Root repo records updated submodule pointers plus deployment/reasoning-streaming ledgers and pushes to `origin/main`.
- `CLAUDE.md` remains untracked and not included.

## Verification Plan

Use git status/log/push output and focused test commands. Verify root commit SHA after push.

## Risks

- Subrepo push ordering matters; root pointer must point at pushed subrepo commits.
- Root may have unrelated dirty files; stage explicitly.

## Assumptions

- Current branch `main` is the intended branch in root and touched subrepos.
