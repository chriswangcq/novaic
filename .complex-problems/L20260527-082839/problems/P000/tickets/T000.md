# Release Controller deployment execution plan

## Problem Definition

Local reasoning streaming changes exist across subrepos and must be shipped through the centralized Release Controller CI/CD flow. The deployment must not bypass the controller by directly running manual deploy scripts.

## Proposed Solution

Split deployment into sequential, verifiable phases: discover the exact Release Controller configuration/API; prepare and push commits/submodule pointers; trigger the controller release; observe build/deploy/smoke status; and close with live service health verification and cleanup of any deployment-specific residue.

## Acceptance Criteria

- The controller trigger mechanism, namespaces, branch policy, health URLs, and current runtime status are known before triggering release.
- Touched code is committed and pushed in subrepos, then root submodule pointers and ledger are committed/pushed.
- Release Controller receives an explicit trigger for the pushed commit/branch.
- Controller run reaches success for server-side services, with run record/plan execution evidence.
- Public or internal health/smoke endpoints confirm the deployed server is serving after release.
- Failures are retried or repaired through the same controller path, not via direct manual deployment scripts.

## Verification Plan

Use Release Controller docs/config/API plus git status, focused tests, controller run records, and HTTP health checks. Record exact run IDs, commit SHAs, URLs, and command outcomes.

## Risks

- Controller may reject prod automation by policy; if so, finish staging server publication and use promotion only if policy allows or the request explicitly authorizes it.
- Remote host credentials or controller URL may live in local config; discover from existing repo/deploy docs rather than guessing.
- Unrelated untracked files should not be swept into deployment commits.

## Assumptions

- `main` is the canonical branch for release-controller CI/CD unless the controller config says otherwise.
- Server-side means API backend/Runtime/Factory stack; App desktop release is not required for the minimum success condition.
