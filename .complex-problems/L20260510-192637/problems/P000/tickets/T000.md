# Split Cortex State Authority Remediation Into Per-Plane Designs

## Problem Definition

The root problem has multiple independent architecture concerns. Treating it as a single one-go plan would hide important state-authority distinctions and likely produce a vague answer. The plan must explicitly cover every remaining state plane and decide whether it belongs in LogicalFS/Workspace, SQLite, Redis, Blob, or process memory.

## Proposed Solution

Split the root problem into child problems, one for each state plane:

- Authoritative state taxonomy and storage rule.
- Active stack/status projection.
- Scope transition observability log.
- Blob payload byte ownership.
- LogicalFS live view semantics.
- Process-local caches/config/docs residue.

Each child produces a concrete remediation plan with authority classification, implementation direction, migration/cleanup, and verification. The parent result summarizes the complete plan.

## Acceptance Criteria

- Root ticket is split instead of one-go.
- Child problems cover all audited gaps.
- Every child plan names owner storage, allowed persistence, invariants, and tests.
- Parent summary composes child plans into one staged implementation strategy.

## Verification Plan

Run ledger validation and success check. The success check must map each original audit gap to a child result and verify no gap is skipped.

## Risks

- Over-fragmenting could make the answer too mechanical; mitigate by keeping each child problem focused on a real state plane.
- Under-fragmenting could hide hard edges; mitigate by requiring explicit storage authority and verification per child.

## Assumptions

- This pass is design-only.
- SQLite is allowed as a durable state store where it improves correctness or queryability.
- Redis is allowed for coordination/lease state, not as the only durable semantic authority unless explicitly justified.
- Blob is allowed for raw bytes/artifacts, not semantic truth unless paired with a semantic reference in the authoritative state model.
